# NextClaw 当前核心用户群体分析

## 背景

这份分析来自一次围绕 V2EX 再次宣传 NextClaw 的讨论。讨论最初从“这次应该宣传什么”出发，随后逐步收敛到更上游的问题：哪些人会被 NextClaw 吸引，以及他们在什么场景下有理由选择 NextClaw，而不是继续使用 Codex、Claude Code、WorkBuddy 或其他更专业、开箱即用的产品。

当前结论不是要把 NextClaw 描述成适合所有 AI 用户的通用产品，也不是为了立即确定下一篇 V2EX 帖子的唯一主题。它记录的是现阶段较可信的三类高适配用户，作为后续产品场景、宣传选题和真实演示任务的上游依据。

## 核心判断

NextClaw 当前不适合围绕“比专业 Agent 更专业”建立选择理由。

- 对重度编程用户，Codex 和 Claude Code 的原生开发体验、代码理解、终端、diff、权限和上下文能力更专业。NextClaw 可以接入这些能力，但“可以接入”不等于其专业用户应该改用 NextClaw。
- 对轻编程、报告、PPT、数据分析和常规本地办公任务，WorkBuddy 等成熟办公 Agent 往往更开箱即用。NextClaw 仅仅宣称“这些也能做”不足以形成选择理由。
- “统一入口”“多模型”“多 Agent”“多渠道”都是能力描述，不是独立用户场景。只有当它们消除某类用户的真实阻力时，才构成产品价值。

因此，当前更值得关注的是下面三类用户。它们存在交集，但不能简单合并成一类，也不应被理解为已经验证完成的市场规模结论。

## 用户群体一：希望把 AI 运行在自己环境中的用户

### 典型特征

- 有 Linux 主机、Docker 环境、可运行容器的 NAS、迷你主机或云服务器。
- 希望 AI 工作环境可以长期在线，而不是只在某个商业桌面客户端打开时存在。
- 希望自己选择模型、Provider、API、工具、技能、数据目录和运行方式。
- 重视可迁移、可恢复、可修改和长期可控，不希望完整工作环境被单一厂商产品绑定。
- 愿意承担适量配置成本，但不想从零拼接 Agent、Web UI、消息入口、自动化和运行管理。

### 真实场景

用户希望在自己的机器或服务器上长期运行一套 AI 工作环境，平时可以通过浏览器或消息渠道发起任务，处理本地或服务器文件，执行周期任务，并在之后继续访问同一批会话、工具和结果。

这里的需求不是“为了部署而部署”。如果部署完成后只进行偶尔问答，自托管本身很可能只是短期兴趣。只有当自托管承载长期任务、个人工具或持续自动化时，它才成为稳定选择理由。

### 可能的吸引力

- 运行环境由用户掌控。
- 支持 macOS、Windows、Linux、Docker 和云服务器等多种部署方式。
- 可以自行选择模型和扩展能力。
- 开源代码降低迁移、审计和长期维护的不确定性。
- 同一实例可以承载会话、文件、Agent、技能、自动化和消息入口。

### 表达边界

- 不应笼统宣称“数据绝不离开本机”。用户接入的模型、消息渠道和外部服务仍可能接收完成任务所需的数据。
- 不应把 Docker、NAS 或 Linux 支持本身当作最终用户价值；必须说明这套环境长期替用户承载什么。

## 用户群体二：开源作者、独立开发者和架构关注者

### 典型特征

- 会持续关注开源 Agent、个人工作台、AI 基础设施和开发工具。
- 对架构边界、可维护性、运行时、技能、MCP、插件化和长期演进有兴趣。
- 可能自己开发产品、开源项目、脚本、自动化或小型服务。
- 看重产品是否可理解、可修改、可扩展，而不只看一轮演示效果。
- 愿意阅读工程复盘、参与讨论、提交 Issue、贡献代码或二次开发。

### 真实动机

这类用户不一定因为某个单点功能进入 NextClaw。他们更可能因为 NextClaw 是一套仍可理解、可修改、可继续生长的开源产品而被吸引。他们关心的不只是“现在能做什么”，也关心“以后能不能按自己的方式继续扩展”。

### 可能的吸引力

- 产品与代码完全开源，可以追踪真实实现和演进过程。
- 支持自定义模型、Agent runtime、Skill、MCP、CLI 工具、Panel App 和 Service App。
- 工程治理、架构取舍和长期维护过程相对透明。
- 可以把 NextClaw 作为个人能力底座、实验环境或二次开发起点，而不只是作为一个固定功能的软件使用。

### 表达边界

“开源作者”“架构关注者”是人群身份，不是独立产品需求。有人可能只会阅读、讨论或 Star，并不会形成长期使用。工程关注度不能直接当作产品留存。

后续宣传和产品分析需要继续区分：

- 会看帖和讨论的人；
- 会安装试用的人；
- 会把 NextClaw 放进日常工作的人；
- 会进行扩展或贡献的人。

## 用户群体三：喜欢开发自用小应用，希望拥有一体化工作台的人

### 典型特征

- 经常产生“这个问题做个小工具就方便了”的想法。
- 可能具备编程能力，也可能主要依靠 Agent 完成实现。
- 开发目标不是商业产品，而是解决自己的重复工作、数据查看、文件处理、记录、查询、计算或自动化问题。
- 不希望每个小工具都单独经历建仓库、选框架、做入口、部署、维护账号和搭建后台的完整产品流程。
- 希望开发过程、应用界面、Agent、资料和后续修改都留在一个工作环境中。

### 核心场景

用户和 Agent 在 NextClaw 会话中讨论一个个人需求，由 Agent 生成脚本、HTML 或 Panel App，在会话右侧直接运行和调试。应用完成后，不只是作为一次任务的代码产物留在文件夹里，而是可以进入应用列表、在后续会话中重新引用，并固定到全局右侧 Dock，像个人常用工具一样长期打开和使用。

需要更强能力时，Panel App 还可以通过受控桥接调用 Agent 或 Service App；需求变化后，用户可以回到会话中继续修改同一个应用。

理想闭环是：

```text
产生一个个人需求
  -> 和 Agent 讨论并生成小应用
  -> 在会话右侧直接运行和调试
  -> 固定到全局右侧 Dock
  -> 以后像内置工具一样随时打开
  -> 需要时调用 Agent 或受控本地服务
  -> 需求变化后继续对话并修改
```

### 典型应用

- CSV 清洗或合并工具。
- 图片压缩、格式转换或批量重命名工具。
- 费用计算器、报价工具或个人表单。
- 行情、服务器状态或项目进展仪表盘。
- 本地资料搜索页、阅读器或知识整理工具。
- 个人习惯、资产、任务或内容发布看板。
- 能继续调用 Agent 做分析、生成或解释的交互工具。

### 可能的吸引力

- 从“想到一个工具”到“拥有一个能用的工具”的距离明显缩短。
- Agent 工作界面和应用界面同时存在，不需要频繁切换独立产品。
- 生成的小应用可以成为 NextClaw 的长期使用表面，而不是一次性产物。
- 应用可以逐步获得 Agent、Service、文件和自动化能力，不必第一天就建设完整后端。
- 用户积累的不只是聊天记录，而是一组越来越贴合自己的个人软件。

### 与普通 AI 编程或 WorkBuddy 的区别

“可以生成一个本地应用”不是 NextClaw 的独有能力。Codex、Claude Code、WorkBuddy 和其他 Agent 同样可以生成代码或预览网页。

NextClaw 更值得验证的差异，是小应用能否成为工作台的一等长期表面：

- 在会话旁运行；
- 被应用列表管理；
- 在后续会话中重新引用和修改；
- 固定到全局右侧 Dock；
- 通过受控桥接调用 Agent 或本地服务；
- 与用户自己的 NextClaw 实例和运行环境共同长期存在。

因此，这类用户需要的不是“又一个帮我写代码的 Agent”，而是一个可以边聊边为自己造软件、造完立即使用、以后继续生长的个人工作台。

## 三类用户之间的关系

三类用户分别从不同维度描述当前高适配人群：

| 维度 | 对应用户群体 | 回答的问题 |
| --- | --- | --- |
| 运行与所有权偏好 | 希望把 AI 运行在自己环境中的用户 | 为什么不直接使用封闭的云端或桌面 Agent 产品？ |
| 身份与参与动机 | 开源作者、独立开发者和架构关注者 | 谁更愿意理解、传播、扩展或贡献 NextClaw？ |
| 核心任务与产品体验 | 自用小应用开发者 | 用户具体用 NextClaw 完成什么，并为什么持续回来？ |

一个用户可能同时属于三类。例如，一位独立开发者把 NextClaw 部署在自己的 NAS 上，并不断开发固定在右侧 Dock 的个人小工具。但三类人群仍应分别保留，因为它们对应不同的吸引力、转化路径和产品证据。

当前不急于确定哪一类必须成为唯一主用户，也不把三类强行收缩成一句过度抽象的画像。后续需要通过真实内容、演示和用户反馈验证各自的吸引力。

## 当前应排除或谨慎对待的伪需求

### 专业编程 Agent 的替代需求

重度编程用户通常更适合继续直接使用 Codex 或 Claude Code。NextClaw 可以编排这些能力，但当前不应以“更专业的编程体验”争夺其原生主场。

### 泛办公和轻编程需求

如果用户只需要报告、PPT、表格分析、本地文件处理或偶尔生成代码，且不关心开源、自托管、长期个人工具和工作台融合，WorkBuddy 等产品可能是更自然的选择。

### 为部署而部署

Linux、Docker、NAS 和云服务器是运行方式，不是最终场景。没有持续任务、个人工具或自动化承接时，部署兴趣不一定形成留存。

### 只看架构、不使用产品

工程故事可以吸引讨论和传播，但不能自动证明产品需求。后续必须分别观察浏览、Star、安装、任务完成和持续使用。

### 空泛的 All in One

“All in One”只有在用户能说出具体减少了哪些切换、重复配置和维护成本时才有意义。不能把功能数量或入口数量本身当作价值。

## 当前推荐倾向

现阶段最值得继续验证的产品表达是：

> NextClaw 是一个可以边聊边为自己造软件的开源 AI 工作台。小应用做出来后，可以直接留在工作台里长期使用，并继续接入 Agent、文件、服务和自动化能力。

这不是最终对外文案，只是当前最能连接三类用户的内部判断：

- 自用小应用提供具体任务和可观察结果；
- 开源、可扩展吸引独立开发者和建设者；
- Linux、Docker、NAS 与云服务器提供长期运行和所有权基础。

后续 V2EX 宣传不必在一篇帖子里平均介绍三类用户。更合适的方式可能是选择一个最有展示力的具体任务作为主叙事，再用开源、自托管和工作台融合解释为什么这件事更适合发生在 NextClaw 中。

## 未决问题

- 三类用户中，哪一类对 NextClaw 当前真实产品完成度的匹配最高？
- 自用小应用场景中，哪一种应用最容易让 V2EX 用户立即代入？
- Panel App 的创建、调试、固定、重新引用和 Agent/Service 桥接是否已经足够顺滑，能够支撑完整公开演示？
- 用户是否愿意长期把一组个人小工具留在 NextClaw 中，而不是生成后独立部署？
- 自托管用户真正需要的是网页入口、消息入口、长期自动化，还是更强的运行管理？
- 开源和架构透明度带来的主要结果是传播、贡献，还是产品留存？
- 现有 [NextClaw 内容策略](../plans/2026-07-12-nextclaw-content-strategy.plan.md) 中的目标用户画像，哪些需要在本分析验证后更新？

## 升级条件

满足以下任一条件后，可以把相关内容升级到更稳定的 PRD、设计或执行计划：

- 通过真实用户访谈、V2EX 反馈、安装数据或使用记录验证其中一类人群的稳定需求。
- 为自用小应用场景选定一个可复现演示，并完成从创建、运行、固定到再次修改的真实验收。
- 明确某类用户需要补齐的产品能力、范围、非目标和验收标准，可以进入 `docs/prd`。
- 确定下一轮 V2EX 帖子的目标人群、核心场景、证据素材和发布结构，可以进入 `docs/plans`。
