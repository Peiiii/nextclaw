# Agent OS 不是 Everything App，而是 Everything Entry

## 背景

用户提出希望 NextClaw 成为 Everything App、All-in-one App 或 Agent OS：用这一个产品抵一万个产品。它不一定把所有功能都塞进内部，但在各种场景下都可以作为起点。

讨论中的例子是“我想看小说”：用户理论上可以让 AI 去网上找资源、下载小说，然后在 NextClaw 内部直接阅读。问题是这种能力应该怎么做才优雅。

## 核心判断

NextClaw 不应该被设计成传统意义上的 Everything App，而应该成为 Everything Entry：用户进入数字场景的默认起点。

它的价值不是内置一万个低质量小应用，而是把用户意图翻译成可执行流程，连接外部服务、工具、插件、本地文件和云资源，再把结果以统一、可继续、可沉淀的方式交付给用户。

这与产品愿景中的统一入口、能力编排、自感知、自进化和生态扩展一致。

## 方案空间

### 方案一：内置大量小 App

短期看起来最像 All-in-one，但长期风险最大。阅读器、播放器、网盘、日历、邮箱、下载器都内置后，产品会快速膨胀成低质量功能集合，破坏统一入口和能力编排的边界。

### 方案二：Agent Workflow 加轻量内部承载

NextClaw 负责理解意图、编排工具、获取或导入内容，并在必要时提供轻量内部承载体验。例如 Reader、Document Viewer、Media Viewer、Task Workspace。

这个方向保留 NextClaw 的入口地位，又避免把所有能力都硬塞进主产品。

### 方案三：只调外部 App

实现最轻，但用户资产和上下文难以沉淀。它适合作为补充路径，例如打开 Kindle、微信读书、浏览器或网盘，但不应成为唯一主路径。

## 推荐倾向

推荐方案二：Agent Workflow 加轻量内部承载。

以“看小说”为例，NextClaw 可以先理解用户目标，再搜索合法来源、在线书店、公开版权库、图书馆、本地文件或网盘资源。拿到 EPUB、PDF、TXT 或网页章节后，进入内部 Reader 或外部 App。内部 Reader 不必一开始复刻完整阅读平台，但应支持目录、阅读进度、书签、划线、笔记、AI 问答、摘要和继续阅读。

关键不是“NextClaw 自己变成小说站”，而是“从 NextClaw 开始，用户能完成找书、打开、阅读、理解和沉淀的闭环”。

## 个人资产层

长期护城河来自用户资产沉淀：读过什么、看到哪里、喜欢什么类型、哪些来源可信、哪些笔记进入个人知识库、哪些内容需要追更。

这让 NextClaw 不只是一次性帮用户找内容，而是逐步形成用户自己的数字生活操作层。

## 未决问题

- `Personal Library` 是否应成为一等产品工作区。
- Reader 是核心内置能力，还是由 marketplace panel app / service app 承接。
- 内容获取需要哪些合法来源、版权边界和风险提示。
- 如何区分“打开外部服务”和“导入个人资产”两种路径。
- 阅读记忆应归属哪个长期数据 owner。

## 升级条件

当讨论进入具体模块边界、数据模型、插件协议、Reader 交互或实现路径时，升级为 `docs/designs`。

当已经确定要执行某个阶段，例如实现 Personal Library MVP、Reader MVP 或内容导入工作流时，升级为 `docs/plans`。
