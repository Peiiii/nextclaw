# 官网产品站完整优化方案

## 背景

上一轮官网已经把首页从早期截图堆叠，调整为围绕主工作台、Agent、消息渠道、技能市场、面板应用和文件预览来展示。但当前站点仍然偏“一页式介绍”：用户能看到主界面，也能找到下载和安装方式，却缺少独立的使用场景、集成能力和版本更新入口。

这会带来三个问题：

- 新用户只能在首页浏览，用途启发不够充分，无法快速按自己的任务类型对号入座。
- 模型、微信、飞书、QQ、自定义模型、技能、MCP、CLI、定时任务等能力散落在首页中段，不利于搜索引擎和用户直接发现。
- 版本更新说明没有形成官网可拉取、可索引、可跳转的公开入口，后续产品内“检查更新”也缺少稳定落点。

## 目标

本轮把官网从“单页展示”升级为更完整的产品站结构，并保持首页清晰、不臃肿。

成功标准：

- 首页优先展示最直观的产品界面和代表性能力，不把安装细节、内部方法论或配置说明堆在首屏。
- 顶部导航提供稳定入口：下载、使用场景、集成、安装方式、文档；更新页保留为低权重入口，供产品内更新提示、footer 和搜索索引使用。
- 新增 `/use-cases/`、`/integrations/`、`/releases/` 三组中英文页面，并纳入 Vite 构建、sitemap、语言切换和 SEO 元信息。
- 使用场景页表达具体任务，不把“统一入口”“安装技能”“接入模型”这类能力描述伪装成用例。
- 集成页集中展示模型、消息渠道、技能与自动化能力，让用户能发现微信、飞书、自定义模型、Docker/npm 等入口背后的能力面。
- 更新页提供可公开访问的版本说明入口，为后续桌面端更新提示拉取说明预留稳定 URL。

## 核心判断

官网的信息层级应该按用户决策路径组织：

1. 先确认这是什么产品，以及实际界面长什么样。
2. 再判断它能不能解决自己的问题。
3. 然后看它能接入哪些模型、渠道、工具和部署方式。
4. 最后才进入下载、安装、文档和版本说明。

所以首页继续承担“第一印象”和“能力概览”；独立页面承担“深一层的信息发现”。这样普通用户不会被安装细节打断，而有明确需求的用户可以从导航直接进入对应页面。

## 页面结构

### 首页

首页保留当前结构，但顶部导航从锚点导向升级为站点路由：

- 下载：进入 `/download/`
- 使用场景：进入 `/use-cases/`
- 集成：进入 `/integrations/`
- 安装方式：进入 `/install/`
- 文档：进入 docs 站

更新页不进入顶部导航。它属于版本说明与产品内更新提示的支撑页面，不属于新用户首次了解产品的主路径。

首页内容顺序保持以图片和直观体验为先：

1. 首屏主工作台截图。
2. 主工作台、Agent、消息渠道、技能市场。
3. 面板应用、文件预览、图片生成、Doc Browser、应用列表。
4. 典型用例卡片。
5. 协作能力与生态接入。
6. FAQ、CTA、社群。

### 使用场景页

路径：

- `/en/use-cases/`
- `/zh/use-cases/`

页面职责：

- 把“它能做什么”表达成具体任务。
- 每个卡片都必须像用户真实会提出的需求，而不是能力名。
- 重点覆盖数据分析与可视化、资料调研、写作、个人小工具、文件处理、群聊请求、定时简报和反馈整理。

### 集成页

路径：

- `/en/integrations/`
- `/zh/integrations/`

页面职责：

- 集中展示 NextClaw 可以连接的模型、消息渠道、技能与自动化。
- 明确体现微信、飞书、QQ、钉钉、自定义模型、OpenAI-compatible endpoint、MCP、CLI、Cron、本地文件等能力。
- 不把集成页写成技术配置手册，只展示用户可以发现的能力入口；具体配置继续交给文档。

### 更新页

路径：

- `/en/releases/`
- `/zh/releases/`

页面职责：

- 提供稳定的版本更新说明入口。
- 首版先展示本轮官网与桌面/NPM 发布闭环相关的用户可见更新分类：新增、增强、修复/稳定性、安装与发布。
- 链接到 GitHub Releases，用于查看完整资产和历史版本。
- 后续可以由发布流程把 release note 同步到可拉取 JSON 或静态内容；本轮先建立可索引页面和导航入口。

## Owner 与数据流

- 路由 owner：`apps/landing/src/main.ts` 的 `PageRoute`、`ROUTES`、`resolvePageRoute` 和语言切换逻辑。
- 页面内容 owner：`LandingCopy` 与 `COPY` 中英文文案。
- 页面渲染 owner：`LandingPage` 内部 render helper。
- 静态入口 owner：`apps/landing/{en,zh}/.../index.html`。
- SEO owner：各静态 HTML、`apps/landing/public/sitemap.xml`、`apps/landing/public/llm.txt` 和 `llms.txt`。
- 构建 owner：`apps/landing/vite.config.ts` 的 Rollup input。

本轮不引入服务端数据流，不新增运行时 API；所有新增页面均由现有 landing app 静态渲染。

## 内容边界

用户可见页面只写结果，不写内部讨论、设计原则或“我们为什么这样组织”的过程说明。

允许出现在设计文档里的内容：

- 为什么要新增页面。
- 为什么导航这样排序。
- 为什么首页不承载安装细节。

不允许出现在官网页面里的内容：

- “主界面才是第一现场”这类内部方法论表达。
- “我们调研后决定”这类生产过程。
- “为了 SEO”这类站点运营解释。

## SEO 与可发现性

本轮补齐：

- 新增页面的 canonical、hreflang、OpenGraph、Twitter card。
- 新增页面的 JSON-LD，使用 `WebPage` / `CollectionPage` / `SoftwareApplication` 中合适类型。
- sitemap 新增 6 个页面，并更新 lastmod。
- `llm.txt` 更新过期说明，同时新增标准命名 `llms.txt`，方便 AI/搜索类读取站点入口。

## 非目标

- 不重做视觉品牌系统。
- 不删除现有截图资产。
- 不改变桌面端 release 拉取逻辑。
- 不实现动态 release note API；本轮只建立公开页面和后续可接入的位置。
- 不提交、不推送，除非用户后续明确要求。

## 验收标准

- `/zh/`、`/zh/use-cases/`、`/zh/integrations/`、`/zh/releases/`、`/zh/download/`、`/zh/install/` 均可打开。
- 英文对应路径也可打开。
- 顶部导航与移动菜单能跳转下载、使用场景、集成、安装方式和文档；更新页可通过 footer、sitemap、`llms.txt` 和产品内更新提示访问。
- 语言切换在新增页面保持同一路由语义。
- sitemap 包含新增页面。
- 构建产物包含新增入口。
- `pnpm --filter @nextclaw/landing tsc` 通过。
- `pnpm --filter @nextclaw/landing lint` 至少不新增阻塞；若仍有既有文件长度警告，需要记录。
- `pnpm --filter @nextclaw/landing build` 通过。
- 使用浏览器检查中文首页和新增页面的首屏、导航、截图比例与移动端可读性。
