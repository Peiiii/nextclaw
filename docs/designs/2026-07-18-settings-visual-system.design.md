# 设置界面视觉系统整体优化方案

## 1. 背景与目标

NextClaw 当前设置界面并非缺少单个圆角、间距或配色，而是缺少一套稳定的设置页视觉语法：普通设置行被做成独立卡片，页面、分区、设置项和控件同时使用边框；提供商、渠道、搜索又各自重复实现列表项和分栏容器；复杂页面继续在卡片内嵌套卡片。结果是层级依赖框线而不是信息结构，页面松散、扫读效率低，也很难通过复用保证后续一致性。

本次优化的目标不是复刻某个产品的像素，而是对齐 ChatGPT、Cursor、VS Code 等成熟产品共同采用的设置范式：

- 页面按“标题 -> 分区 -> 分组 -> 设置行”组织；
- 一行表达一个设置意图，左侧标题与说明，右侧控件或状态；
- 主要依靠字号、字重、留白和浅色分组面建立层级；
- 控件保留必要边界，容器不重复描边；
- 相同语义必须经过共享纯展示/交互组件，不允许页面各写一套卡片；
- 在常规桌面、窄桌面、侧边文档打开和移动端下，都保持可扫读、可操作。

这项改动服务 NextClaw 的“统一体验优先”：设置是用户理解和掌控个人操作层的统一控制面，视觉与交互不能随能力模块增长而继续碎片化。

## 2. 现状证据

### 2.1 普通设置页

- `SettingRow` 自带圆角、边框和背景，外层页面又使用 `Card`，形成“页面卡片 -> 设置项卡片 -> 控件边框”的三层边界。
- 外观页在页面标题后再次用卡片标题重复“外观”，信息层级冗余。
- 模型页为两个字段分别创建大卡片，字段数量少但占用面积大。
- 登录管理、运行时、密钥、远程访问和更新页各自使用不同的卡片标题、内嵌卡片、硬编码灰色与间距。

### 2.2 资源管理页

- 提供商、渠道、搜索共用 `ConfigSplitPage`，但分栏组件把列表面板和详情面板分别做成独立描边卡片，中间还保留大间距。
- 列表项再次使用圆角边框，选中、悬停和默认状态通过增减边框表达，边缘数量过多。
- 详情头、正文和底部操作继续使用分割线；局部表单区又嵌套小卡片或胶囊。
- 分栏只在视口 `xl` 断点生效，打开侧边文档后内容容器变窄，仍可能出现低效布局。

### 2.3 根因

直接原因是页面以 `Card` 作为默认布局原语；结构性原因是缺少设置专用的分区、分组和设置行合同；长期原因是共享层只抽出了通用卡片和表单控件，没有抽出“设置页”这一稳定产品语义。

正确 owner 分为两类：

- 普通设置页视觉合同由 shared settings primitives 负责；
- 列表—详情型设置页由现有 `ConfigSplitPage` 组件族负责。

业务页面继续负责业务数据、状态和动作，不把 store、query、manager 或 i18n key 放入共享展示组件。

## 3. 设计原则

### 3.1 信息层级优先于容器层级

先使用标题、说明、分区标签、间距和浅色背景；只有确实需要区分可交互控件、危险提示或独立内容资产时才增加边界。

### 3.2 边框预算

一个正常设置区域遵守以下预算：

- 页面画布：0 条外框；
- 一个设置分组：0 条外框，以浅色背景和圆角形成整体；
- 分组内部：相邻行最多 1 条低对比度分隔线；
- 输入、选择器、按钮等真实控件：保留自身边界；
- 列表—详情资源管理器：整体最多 1 个外边界，列表与详情之间最多 1 条分隔线；
- 选中项优先使用填充色，不额外增加描边；
- 警告、错误、代码块和独立文档内容可按语义例外。

任意区域的视觉容器嵌套深度默认不超过两层。页面、分区、设置行、选项卡和控件同时描边属于违规。

### 3.3 紧凑密度

- 页面标题与正文间距：20px 左右；
- 分区之间：24px 至 32px；
- 普通设置行最小高度：56px 至 64px；
- 设置行横向内边距：16px 至 20px；纵向内边距：12px 至 14px；
- 标题使用 14px 中等字重；说明使用 12px 至 13px、紧凑行高；
- 控件宽度由语义决定，不默认铺满整行；长文本输入和复杂编辑器除外。

### 3.4 颜色与状态

- 所有基础设置组件只使用设计 token：`background`、`foreground`、`muted`、`border`、`primary`、`destructive` 等；
- 禁止在共享设置组件中使用 `gray-*`、`white` 等固定色；
- 默认、悬停、聚焦、选中、禁用必须可区分；
- 焦点状态使用统一 focus ring，不依赖颜色变化；
- 危险、警告、成功颜色只表达真实状态，不用于普通层级装饰。

## 4. 组件合同

### 4.1 普通设置组件

在现有 shared settings owner 中收敛以下原语：

- `SettingsPage`：统一所有设置路由的内容起止线、标题/说明/操作区和标题到正文的节奏；只保留普通页与列表—详情页两种布局变体；
- `SettingsSection`：分区标题、说明、可选操作与内容；不读取业务状态；
- `SettingsGroup`：同一语义分区下的一组设置行，统一背景、圆角和行分隔；
- `SettingRow`：设置标题、说明、右侧控件或下方扩展内容；支持常规和纵向布局；

`SettingsPage` 内部复用通用 `PageLayout` 与 `PageHeader`，业务页面不得再自行拼接 `max-w-*`、`mx-auto`、根级 `space-y-*` 或分栏高度类。普通页和资源管理页共用同一页面画布与左右边距，差异只发生在正文内部：普通页按设置分区纵向排列，资源管理页把剩余高度交给 `ConfigSplitPage`。

共享组件只接收 React 展示属性和事件，不导入业务 store、query、manager、路由或业务 i18n key。业务页面负责把业务事实转换成 props。

### 4.2 资源管理组件

继续使用并改造现有 `ConfigSplitPage` 组件族，不新增平行资源管理器：

- `ConfigSplitPage`：一个整体 surface，内部为列表区和详情区；
- `ConfigSplitSidebar` / `ConfigSplitDetailPane`：不再各自拥有外框；
- `ConfigSelectionCard`：改为无边框列表项，选中用填充和文字色；
- `ConfigSplitPaneHeader` / `Footer`：只保留必要的单条结构分隔；
- 桌面采用流动列宽，侧边文档打开后仍维持可用的双栏；移动端继续使用列表—详情钻取。

## 5. 页面迁移范围

### 5.1 必须迁移

- 模型：两张字段卡片收敛成一个或两个语义分区的设置行；
- 外观：删除重复标题卡片，主题、语言、消息布局、快捷栏进入统一分组；
- 提供商、渠道、搜索：统一整体分栏 surface、列表选择状态、头部和底部密度；
- 登录管理：账号状态、启用开关、密码设置使用分区与设置组；
- 运行时：运行控制、桌面行为、默认设置、runtime/agent/binding 编辑区使用同一分区语法；
- 更新：概览、发布渠道和操作使用设置分组；发布说明作为独立内容资产保留；
- 远程访问：状态与设备信息进入紧凑分组，操作与异常提示保留语义状态；
- 密钥：启用、默认 provider、provider/ref 编辑列表统一分区与低边框条目。

### 5.2 有意保留的专用表面

- MCP 市场的商品卡片、筛选和详情是发现/安装任务，不是属性设置行；保留其专用视觉结构，但页面标题与设置壳层间距继续遵守全局页面节奏。
- 更新日志预览、警告、错误、代码和空状态是独立内容或反馈，不强行伪装成设置行。
- 提供商模板选择器属于一次性选择面板，可保留网格卡片，但默认列表和详情区必须遵守资源管理合同。

## 6. 响应式与可访问性

- 资源管理页桌面双栏使用流动列表宽度，避免固定 340px 加详情在窄容器下失衡；
- 移动端继续保留列表和详情两步钻取及明确返回入口；
- 普通设置行在窄宽度下自动改成上下布局，控件不挤压标题；
- 单选选项保持 `radiogroup` / `radio` 语义；
- 图标按钮必须保留可访问名称或 tooltip；
- 所有可交互状态具备 hover、focus-visible、active、disabled 反馈；
- 文本和控件不通过截断隐藏关键配置值，必要时允许换行或纵向布局。

## 7. 实施策略

1. 先改造现有 `SettingRow` owner，增加分区与分组能力，并删除每行卡片化默认样式。
2. 改造现有 `ConfigSplitPage` owner，把双卡片收敛为单 surface，并统一列表项状态。
3. 先迁移外观、模型、提供商、渠道、搜索作为两类模板。
4. 再迁移登录管理、运行时、更新、远程访问和密钥，删除重复 `Card` 包装与硬编码灰色。
5. 将本规范的强制部分写入前端样式治理 skill，确保以后新增设置页会自动触发检查。

本次不新增新的业务状态、manager、store、兼容路径或二级视觉系统。共享原语是稳定设置语义的 owner，不是为复用而增加的空包装。

## 8. 验收标准

### 8.1 静态验收

- 普通设置页不再默认使用“每个设置项一张卡”的结构；
- 外观页不再重复显示“外观”标题；
- 提供商、渠道、搜索不再显示两个彼此独立的外框面板；
- 默认列表项无边框，选中态通过填充表达；
- shared settings primitives 不包含业务依赖和固定灰色；
- 设置相关页面新增样式优先消费 token；
- 中文语言下的设置页标题、分区标题和设置项标题不得保留英文占位；品牌名、协议名和正式工具名除外。
- 模型、提供商、渠道、外观、登录管理、搜索、更新、远程访问、运行时、密钥和 MCP 页的标题与正文必须共用同一左右起止线，标题到首个正文表面的间距统一为 24px；
- 设置业务页不得直接渲染后端 Schema / uiHint 中的英文标签作为本地化界面标题；Schema 可继续提供占位符与帮助信息，但用户可见标签由前端 i18n owner 提供。

### 8.2 功能验收

- 主题、语言、消息布局、侧边快捷栏仍可修改；
- 模型和工作区仍可保存；
- 提供商、渠道、搜索的选择、启停、编辑和保存行为不变；
- 登录管理、运行时、更新、远程访问和密钥的原有动作与状态仍可使用；
- 移动端资源列表—详情返回路径不回归；
- 中文语言下逐页显示中文标题，登录相关页面使用“登录管理”，`Secret Providers`、`Secret Refs` 等英文占位不再出现。

### 8.3 真实界面验收

至少在以下场景逐页抽样并保存截图：

- 1440px 常规桌面；
- 1180px 窄桌面；
- 1180px 且右侧文档面板打开；
- 移动端宽度。

重点对照外观、模型、提供商、渠道、搜索、运行时和登录管理页面，检查：

- 是否一眼能识别页面、分区、设置项和控件四级结构；
- 是否不存在连续三层以上边框或卡片；
- 相同设置行的高度、标题、说明和控件对齐是否一致；
- 列表—详情是否在可用高度内并列呈现，滚动归属是否清楚；
- 与参考产品是否属于同一类“紧凑、清晰、简约”的设置风格，而非仅仅换了颜色。

### 8.4 工程验收

- 设置页定向测试通过；
- `nextclaw-ui` TypeScript 检查通过；
- 触达文件 ESLint、governance 与 maintainability guard 通过；
- 真实浏览器冒烟通过；
- 非测试语义代码净增不大于零，新增共享合同由删除重复卡片结构抵消。
