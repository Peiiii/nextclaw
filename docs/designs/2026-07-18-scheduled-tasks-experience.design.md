# 定时任务体验优化设计

## 背景

NextClaw 已经具备一次性任务、固定间隔任务、Cron 表达式任务、启停、立即执行、删除、Agent 归属和会话关联能力，但当前定时任务页主要承担“查看运行信息”的管理职责。用户第一次进入时，只看到空状态提示；已有任务时，则需要从长消息、运行时间和技术字段中自行理解任务。

本设计参考两类成熟交互：

- ChatGPT 将“安排任务”作为带定时任务语境的 AI 会话入口。用户回车后进入会话，由 AI 理解意图并创建任务；已有任务采用列表—详情结构，模板位于真实任务之后。
- WorkBuddy 在零任务时突出单一行动和大面积模板网格，帮助用户迅速理解自动化可以解决哪些具体问题。

NextClaw 不复制任一产品的表面布局，而是组合两者优势，并保持产品愿景中的统一入口、意图优先、自感知和自治：复杂创建与调整从会话开始，任务页负责发现、概览、日常控制和真实运行反馈。

## 现状依据

### 当前页面

`packages/nextclaw-ui/src/features/cron/components/cron-config.tsx` 当前负责：

- 查询全部任务；
- 按名称、消息、ID 和会话搜索；
- 按启用状态筛选；
- 展示任务消息、计划、下次/上次运行和错误；
- 启停、立即执行和删除任务。

页面没有创建入口、模板目录、任务详情侧栏或返回关联会话的主操作。空状态只提示用户在对话中告诉 Agent。

### 已有可复用链路

- `ChatSessionListManager.createSession` 是新建草稿会话的现有 owner，负责清空选中会话、同步草稿线程状态并导航到聊天草稿页。
- `ChatWelcomeCapabilityGrid` 已经证明“模板只填写会话请求”的交互可以复用，不需要模板直接执行领域写入。
- Agent 侧 `cron` tool 是内置定时任务调度入口，支持创建、列出、启停和删除任务。
- Cron payload 已支持 `agentId` 和 `sessionId`，任务页可以把 Agent、来源会话和执行会话作为用户可理解的关联信息。
- UI Server 已有创建任务接口，但当前没有更新名称、提示词和计划的完整 mutation 合同。

## 核心判断

定时任务页不应继续被定义成 Cron 管理后台，也不应在页面内复制一套复杂任务创建向导。它应成为“自动运行的 Agent 工作中心”，并保持两类职责清晰：

1. AI 会话负责理解自然语言、澄清条件、选择 Agent/工具并创建复杂任务。
2. 定时任务页负责发现用法、查看任务、执行轻量控制、检查结果和回到关联会话。

模板不是第二套任务创建机制。模板只提供一条可编辑的自然语言请求，最终仍通过新会话和同一个 Cron owner 创建任务。

## 目标与成功标准

### 用户价值

- 零任务用户可以立即看懂定时任务能做什么，并在一次点击后开始表达需求。
- 已有任务用户优先看到任务、下次执行和健康状态，不被模板或技术字段淹没。
- 用户可以从任务回到关联会话，继续补充目标或纠偏。
- 失败状态能说明发生了什么，并提供查看会话或重新运行的明确入口。

### 可观察成功标准

- 零任务、已有任务、选中任务三种状态均有完整页面结构。
- 输入请求并回车后进入新的 AI 会话草稿，请求被预填并等待用户确认，而不是直接写 Cron。
- 点击模板先把请求带入输入框并聚焦，用户确认后走同一会话路径。
- 选中任务后可以查看提示词、计划、Agent、会话和最近运行结果。
- 页面在桌面、窄桌面和移动宽度下保持主任务与主要操作可达。

## 推荐方案：自适应任务工作台

原型评审已经收敛到一个方向，不再继续比较三套平行页面。推荐方案根据任务数量调整内容权重，但始终保持同一个页面骨架：

- 零任务：使用居中引导和完整模板网格，帮助用户完成第一次创建；
- 已有任务：AI 输入保留在顶部，当前任务概况紧随其后，任务列表成为页面唯一持续增长的主内容；
- 展开任务：在当前行下方显示只读快捷预览，同一时间只展开一项；
- 编辑任务：右侧抽屉覆盖打开完整详情，不把列表永久压进分栏或卡片；
- 模板：已有任务时退到列表之后，只承担低频启发，不和真实任务争夺首屏。

这套方案的核心不是把 ChatGPT 和 WorkBuddy 的视觉拼在一起，而是区分两种生命周期：模板只服务冷启动，任务列表服务长期使用。页面必须优先为后者提供可扩展空间。

### 关键取舍

- 当前任务概况放在任务列表之前，用一条紧凑概况带回答“有多少任务正在运行、暂停或需要处理”，不虚构历史统计，也不在页面底部重复总结。
- 任务列表直接生长在页面内容流中，不使用固定高度、内部滚动、网格列或外层卡片约束。
- 搜索、状态筛选和结果计数贴近列表标题，任务增多后仍能快速定位。
- 任务很多时使用服务端分页，默认每页 10 项；搜索和状态筛选与分页使用同一份服务端查询合同，禁止先拉全量再在浏览器切页。
- 详情是按需出现的临时层，不改变任务列表的长期布局模型。
- 长列表会自然把模板推到更靠后的位置，这是成熟用户使用重心变化的正确结果，不需要把模板固定在首屏。

## 推荐信息架构

```text
定时任务
├── 页面标题
├── 安排任务输入框
├── 当前任务概况带
│   ├── 全部任务
│   ├── 运行中
│   ├── 已暂停
│   └── 需要关注
├── 真实任务主列表
│   ├── 搜索、状态筛选与数量
│   ├── 任务名称与健康状态
│   ├── 计划与下次执行
│   ├── 单项行内快捷预览
│   ├── 启用/暂停与更多操作
│   └── 多页时显示范围与前后翻页
├── 模板/灵感区
└── 按需打开的任务详情抽屉
    ├── 任务内容
    ├── 当前执行计划
    ├── Agent 与关联会话
    ├── 最近一次执行快照
    └── 暂停、立即执行、打开会话、删除
```

零任务时隐藏运行概况、搜索和状态筛选，中央引导承担主操作；存在任务后不再显示大面积空状态，任务列表拥有页面纵向空间，模板降低视觉权重。

## 交互设计

### 自然语言入口

- 占位文案使用“描述你想定期完成的事”。
- Enter 创建并进入新会话草稿，请求预填到会话输入框；用户可继续修改后发送。
- 输入容器沿用 shared Input 的低装饰合同：已有任务状态使用标准 36px 高度、`rounded-xl`、低对比度边框和卡片背景，不再叠加外层输入壳或悬浮阴影；聚焦只接收键盘输入，不改变外框颜色、粗细、阴影或尺寸。
- 定时任务页不解析时间、不生成 Cron 表达式，也不直接写 Cron；会话中的 Agent 负责澄清并创建任务。
- 导航失败或运行时未就绪时保留输入内容并给出可恢复错误，不静默丢失。

### 模板

- 模板标题表达具体结果，例如“每周五汇总项目进展”，而不是“工作效率”。
- 点击模板后将完整请求带入输入框并聚焦，不直接启用任务。
- 首版保留 6～9 个跨工作、信息、学习和生活场景的模板；已有任务时只展示 3～4 个，并提供“查看全部”。
- 模板内容属于前端定时任务 feature 的用户可见目录，文案进入 i18n owner；它不拥有 Cron 数据或执行逻辑。

### 任务列表

- 一行优先展示名称、自然语言计划、下次执行、健康状态以及“独立会话 / 绑定会话”范围。
- 长提示词只显示短预览，完整内容进入详情。
- 列表直接位于页面内容流，不设置固定高度或内部滚动；页面滚动是唯一纵向滚动 owner。
- 列表不使用外层卡片或 Dashboard 网格约束，行之间只保留轻量分隔与 hover 反馈；hover 不改变行的位置、圆角或尺寸。
- 搜索、状态筛选和总数与列表标题形成同一工具区；零任务时整组隐藏。
- 列表默认每页 10 项，只有结果超过一页时才显示分页；分页展示当前范围、总结果数、当前页与总页数，并提供上一页/下一页。
- 搜索、状态变化和手动刷新后回到第一页；删除末页最后一项时，回到前一页。
- 概况统计始终针对全部任务，不随搜索、状态或当前页变化；列表标题中的数量才表示当前查询的结果总数。
- 点击任务行切换只读快捷预览，预览展示完整任务内容、Agent、执行会话、最近运行和最近错误；同一时间只允许展开一项，避免列表变成多个详情面板的堆叠。
- 行内预览不复制启停、立即执行、删除等完整管理操作；这些操作继续归行级入口与详情抽屉，保持读与管的层级清晰。
- 展开态使用低对比度填充，hover 与展开态分别表达；整行点击语义不再额外放置常驻展开图标，避免和编辑、更多操作竞争注意力。搜索、状态筛选、翻页和手动刷新会清理展开态，避免上下文变化后保留过期展开项。
- 启停开关始终可见，因为它同时表达当前状态；编辑和更多操作在桌面端仅于行 hover 或键盘聚焦时出现，在无 hover 的触屏设备上始终可见。
- 编辑图标直接打开任务详情抽屉；更多操作使用纵向省略图标；启停、编辑和更多操作都不触发行展开。

### 任务详情

- 桌面端使用基于 Radix Dialog 语义的 Shadcn Sheet 结构从右侧覆盖打开，移动端使用全宽详情层；遮罩淡入、面板滑入，支持 Esc、遮罩和关闭按钮退出，关闭后回到原列表位置。
- 详情层不改变列表宽度，也不成为列表的常驻第二列。
- 抽屉头部和底部操作区固定，只有中间详情内容设置高度约束并独立滚动，短视口下关闭与核心操作仍保持可达。
- 日常控制提供暂停/启用、立即执行和删除。
- “打开会话”优先使用任务的 `sessionId`；任务自有会话使用 `cron:<jobId>`。
- 当前 Cron 持久化只拥有最近一次运行的时间、状态和错误，因此首版只展示这份真实快照，不从会话消息反推或伪造多次运行历史。
- 绑定任务使用显式 `sessionId` 打开会话；独立任务使用 `cron:<jobId>`，但首次运行前不提供无效的会话入口。
- 名称、提示词和计划的结构化编辑需要新增统一 update mutation；AI tool 和 UI controller 都调用同一个 Cron owner，禁止各自维护更新规则。

## Owner 与数据流

### 创建主链路

```text
定时任务输入/模板
  -> ChatSessionListManager 创建草稿会话，并通过草稿路由状态交接初始请求
  -> Session conversation input state owner 在首次渲染前用初始请求建立 composer snapshot
  -> 用户确认后由 Chat conversation owner 发送
  -> Agent 理解并调用 cron tool
  -> Cron owner 持久化任务
  -> cron query 失效并刷新任务页
```

模板不得直接调用创建任务 API。

### 管理主链路

```text
Cron owner
  -> UI Server list(query, status, offset, limit) 负责筛选、分页与全局概况
  -> Client SDK 使用具名查询合同序列化参数
  -> useCronJobs query/mutations
  -> 定时任务 presenter/view model
  -> 列表与详情组件
```

任务事实只归 Cron owner。页面只维护展开任务 ID、抽屉任务 ID、查询词、状态和页码等视图状态，不复制任务实体。服务端先在同一份任务集合上计算全局概况，再应用搜索与状态筛选并分页，返回 `jobs + filtered total + global summary`。

## 目录组织

原型阶段只新增：

- `docs/designs/2026-07-18-scheduled-tasks-experience.design.md`
- `docs/designs/2026-07-18-scheduled-tasks-experience.prototype.html`

正式实现继续在现有 `packages/nextclaw-ui/src/features/cron` feature root 内演进，不新建平行 automation feature。角色如下：

- `cron-config.tsx`：查询、筛选、选择和 mutation 编排 owner；
- `cron-task-discovery.tsx`：AI 输入与模板目录；
- `cron-job-row.tsx`：可自然增长的任务行与行级操作；
- `cron-job-detail-dialog.tsx`：组合 shared Sheet 的任务详情内容；
- `shared/components/ui/overlays/sheet.tsx`：基于 Radix Dialog 的通用 Sheet 交互、动画与可访问性 owner；
- `hooks/`：继续承接 query/mutation 连接；
- `utils/`：仅承接无状态的 view model 映射；
- manager/presenter：只有出现跨页面会话交接或稳定视图编排责任时才新增，不能为拆文件制造空 owner。

## 可维护性与删减策略

- 正式实现时重构或替换现有 `CronJobCard`，不保留旧卡片和新列表两套长期实现。
- 复用现有聊天输入视觉和草稿会话 owner，不复制完整 composer 或另建 schedule composer 状态机。
- 模板复用“模板 -> 会话请求”的通用合同，不调用 Cron API。
- 搜索、筛选和计数只在已有任务状态出现，零任务不渲染无意义控制。
- 分页、搜索和状态筛选收敛到同一个 list API；删除旧的前端全量 `filter` 路径，避免数据量增长后出现传输、渲染和统计口径分叉。
- structured update 必须先进入 Cron owner，再由 UI 和 Agent tool 共同消费，避免两套更新语义。

## 兼容与迁移

- 现有任务数据无需迁移；新页面继续读取当前 CronJobView。
- 现有 `/cron`、`/chat/cron` 导航入口和启停、立即执行、删除行为保留。
- 当前没有统一 update mutation，详情中的名称、任务内容和计划保持只读；编辑图标用于打开详情，进一步调整通过执行会话完成。不得用删除后重建伪装编辑。
- 不自动把旧任务绑定到创建会话。没有 `sessionId` 时使用任务自有 Cron 会话。

## 验收标准

### 原型阶段

- 原型只保留推荐方向，避免评审继续被已淘汰结构分散。
- 支持零任务、已有任务和选中详情三种状态。
- 已有任务默认打开后，概况位于列表之前，任务列表不受卡片、固定高度或内部滚动限制。
- 点击任务行只展开当前项的快捷预览，再点击收起；切换到另一项时上一项自动收起。
- 选中详情以覆盖层打开，列表的页面结构和宽度保持不变。
- 详情抽屉的头尾固定，中间内容在受限高度内滚动；Esc、遮罩和关闭按钮均可退出。
- 输入回车和模板选择可以演示进入会话草稿的完整意图。
- 任务行 hover 不发生位移或几何变化，只使用轻量背景和行级操作透明度；输入框 focus 不改变容器边框或阴影。
- 1440px、1024px、760px 和 390px 宽度下无横向溢出，主要操作仍可达。

### 正式实现阶段

- Cron feature 定向组件测试覆盖单项展开、切换收起、状态切换、模板预填、会话导航和独立操作隔离。
- Server/SDK/UI 跨层测试覆盖分页参数、服务端搜索与状态筛选、全局概况、翻页和筛选后页码归零。
- 任务超过 10 项时只渲染当前页；翻页期间保留上一页内容并显示加载反馈，少于两页时不显示分页控件。
- TypeScript 与触达 package ESLint 通过。
- `pnpm lint:new-code:governance` 与治理 ratchet 通过。
- 真实页面从定时任务入口输入请求，成功进入新会话草稿；用户发送后由 Agent 创建任务。
- 模板请求与手工输入走同一条会话创建链路。
- 运行、启停、删除和打开会话完成浏览器冒烟。

## 非目标

- 本轮不修改 Cron 持久化合同，不新增多次运行历史。
- 本轮不实现模板 marketplace、用户自定义模板或远程模板下发。
- 本轮不新增名称、提示词和计划的结构化 update mutation。
- 本轮不重做 Cron 调度器、Agent tool 或会话运行时。
- 本轮不直接发布或部署；用户可见变化通过独立 changeset 进入后续统一发布流程。

## 后续实现顺序

1. HTML 原型与真实页面保持同一信息层级和会话范围语义。
2. 在现有 Cron feature 内替换旧卡片路径，保留唯一任务列表实现。
3. 完成定向测试、类型检查、治理检查和真实浏览器验收。
4. 后续若需要可审计的多次运行历史，先在 Cron owner 中新增 `CronRunRecord`，再扩展 UI。
5. 后续若需要页面内结构化编辑，先补齐统一 update mutation，再由 UI 和 Agent tool 共用。
