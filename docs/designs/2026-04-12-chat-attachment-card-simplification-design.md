# 聊天附件卡片简化设计方案

日期：2026-04-12
适用范围：聊天消息中的文件附件卡片
状态：建议采用

## 1. 背景

当前附件卡片已经能表达“这是一个文件，可点击打开”，但还没有达到 NextClaw 应有的产品质感。主要问题不是能力缺失，而是展示层还不够克制：

- 信息过多，主次不清
- 中英文混杂，像未收尾状态
- 技术性字段直接暴露给用户
- 文件类型识别弱，导致用户仍需读文字确认

NextClaw 的长期目标是成为 AI 时代的个人操作层，因此界面不应让用户处理底层技术细节，而应把常见判断预先消化掉，让信息表达更自然、更统一、更可快速扫描。

## 2. 设计目标

本次方案只解决一个问题：把附件卡片收敛成更简约、更统一、更成熟的展示。

目标：

- 让用户一眼看懂“这是什么文件”
- 让最重要的信息只剩文件名、类型和大小
- 清除不必要的技术字段和重复表达
- 统一语言，避免中英文混杂
- 为高频文件类型建立稳定视觉锚点

非目标：

- 不扩展附件编辑能力
- 不引入新的复杂操作区
- 不为低频格式设计过重的特殊视觉
- 不把卡片做成文件管理器式的高信息密度组件

## 3. 当前问题诊断

### 3.1 信息噪音过多

当前卡片同时展示了文件名、`Archive attachment`、`Archive`、`76 MB`、`application/octet-stream`。其中真正有价值的只有文件名和大小，类型信息也只需要保留一次。

`application/octet-stream` 属于 MIME 技术细节，不应作为主卡片内容暴露给普通用户。`Archive attachment` 与 `Archive` 也存在重复表达。

### 3.2 信息层级被打散

文件名本应是绝对主角，但当前多枚标签和英文描述把注意力拉散了。用户需要在多处文本中拼凑“这是什么”，而不是扫一眼直接理解。

### 3.3 语言系统不统一

文件名是中文，说明文案和按钮是英文，这会显著拉低完成度。对中文语境用户来说，这种混杂会让界面像开发中间态，而不是成熟产品。

### 3.4 类型识别能力不足

对于 ZIP、PDF、Word、Excel、图片、视频等高频类型，用户天然期待更明确的专属识别。如果仍使用高度泛化的占位图形，用户仍需要读文字确认，界面也很难形成稳定记忆点。

## 4. 核心设计判断

### 4.1 采用“减法优先”而不是“补充解释”

附件卡片的目标不是解释文件系统，而是帮助用户快速识别和行动。因此推荐采用单一路径：

- 保留：文件名
- 保留：一句合并后的辅助信息
- 保留：一个主要操作
- 删除：MIME 类型
- 删除：重复类型标签
- 删除：不必要的英文说明

### 4.2 高概率常见类型必须有专属图标

如果产品希望附件展示更自然、更成熟，那么常见文件类型应有专属图标或专属视觉标识。这不是装饰优化，而是信息识别优化。

推荐覆盖首批高频类型：

- PDF
- Word
- Excel
- PowerPoint
- ZIP / 压缩包
- 图片
- 视频
- 音频
- 文本 / 代码
- 其它未知类型使用通用文件图标回退

### 4.3 预览优先于类型图标

当文件本身可预览时，应优先展示内容缩略信息，而不是继续强调抽象类型图标：

- 图片：优先缩略图
- 视频：优先封面
- PDF：可优先首页缩略图，其次再回退 PDF 图标
- 其它不可预览文件：使用类型专属图标

这会让附件从“文件说明块”进化为“内容入口块”，更符合统一入口产品的方向。

## 5. 推荐信息结构

推荐把附件卡片收敛为两层信息：

### 第一层：主信息

- 文件名

要求：

- 最醒目
- 单行优先，必要时截断
- 保留原始文件名与扩展名

### 第二层：辅助信息

将类型和大小合并成一行：

`ZIP 压缩包 · 76 MB`

要求：

- 一行表达完成
- 不拆成多个 badge
- 不再额外展示 MIME
- 语言与卡片主语言保持一致

### 操作区

右侧仅保留一个安静的次级按钮：

- `打开`

如果需要区分行为，可按场景替换为：

- `查看`
- `下载`

但同一页面内应保持单一术语，不建议同时出现 `Open file`、`查看`、`下载` 等混杂命名。

## 6. 文案规范

中文环境下，附件卡片默认使用中文 UI 文案。英文只保留在以下内容中：

- 用户原始文件名中的英文部分
- 文件扩展名
- 行业通用缩写，如 `PDF`、`ZIP`

推荐文案：

- `ZIP 压缩包`
- `PDF 文档`
- `Word 文档`
- `Excel 表格`
- `PowerPoint 演示文稿`
- `图片`
- `视频`
- `音频`
- `文本文件`
- `代码文件`
- 按钮：`打开` / `查看` / `下载`

禁止继续使用：

- `Archive attachment`
- `Open file`
- 直接展示 `application/octet-stream`

## 7. 图标策略

### 7.1 图标体系原则

图标应承担“快速识别”职责，而不是承担装饰职责。设计上应统一使用同一骨架与同一视觉语言，只在类型标识和少量颜色上区分。

原则：

- 风格统一
- 轮廓统一
- 装饰克制
- 颜色适度区分
- 保证在小尺寸下也能快速识别

### 7.2 首批类型映射建议

- `pdf`：红色系 PDF 图标
- `doc/docx`：蓝色系 Word 图标
- `xls/xlsx/csv`：绿色系表格图标
- `ppt/pptx`：橙色系演示图标
- `zip/rar/7z/tar/gz`：琥珀色压缩包图标
- 图片：优先缩略图，回退多彩图像图标
- 视频：优先封面，回退视频图标
- 音频：音频图标
- `txt/md/json/xml/yaml/js/ts/py/java/go/rs/sh`：文本/代码图标
- 未知类型：中性通用文件图标

### 7.3 回退规则

推荐按以下顺序决策：

1. 能否直接预览内容
2. 不能预览时，是否属于高频已识别类型
3. 若不是，回退到通用文件图标

这个顺序比“所有文件先给一个 generic icon”更符合用户直觉。

## 8. 视觉结构建议

推荐卡片结构：

```text
+--------------------------------------------------------------+
| [类型图标/缩略图]  文件名                          [打开]     |
|                  ZIP 压缩包 · 76 MB                         |
+--------------------------------------------------------------+
```

视觉原则：

- 左侧图标尺寸中等，不抢主标题
- 文件名与辅助信息左对齐
- 按钮弱化为次级操作，不与文件名竞争
- 卡片整体减少无意义描边和标签感
- 用留白和对齐建立秩序，而不是依赖多个 badge

不推荐：

- 同时堆多个圆角标签
- 让类型、大小、格式分散在不同区域
- 把技术字段放在辅助信息区
- 通过大面积高饱和色块抢占视觉中心

## 9. 推荐状态设计

建议至少覆盖以下状态：

- 默认态：展示图标、文件名、辅助信息、主操作
- 上传中：按钮禁用，辅助信息显示进度或 `上传中`
- 处理中：用于服务端识别类型或生成预览时，显示 `处理中`
- 失败态：显示简短错误与重试入口，但仍保留文件名
- 不可预览态：回退类型图标，不额外暴露技术错误

状态表达应简洁，不把底层失败细节直接抛给用户。

## 10. 优先级建议

### P0：必须立即收敛

- 去掉 `application/octet-stream`
- 去掉重复类型 badge
- 去掉英文说明 `Archive attachment`
- 统一按钮和辅助文案为中文
- 把辅助信息收敛为 `类型 + 大小` 一行

### P1：高价值增强

- 为高频文件类型补专属图标
- 建立未知类型回退图标
- 统一图标骨架和颜色策略

### P2：更高级的体验

- 图片、视频、PDF 引入缩略图优先策略
- 补齐上传中、处理中、失败态的统一视觉

## 11. 验收标准

满足以下条件即可认为本轮设计收敛达标：

- 用户扫一眼即可知道附件类型，不依赖阅读多段辅助文字
- 卡片主信息只剩文件名、类型、大小和一个操作
- 不再展示 MIME 等技术实现细节
- 中文语境下不再出现无必要的中英文混杂
- ZIP、PDF、Office、图片、视频、音频、代码/文本等高频类型都有稳定识别方式
- 卡片整体视觉更安静，信息更少，但判断更快

## 12. 我的明确推荐

如果只允许做一轮最小但有效的优化，我推荐直接采用下面这条单一路径：

1. 把附件卡片改成“双层信息 + 单按钮”结构
2. 辅助信息固定为 `类型 · 大小`
3. 删除 MIME 和重复类型标签
4. 全面统一中文文案
5. 为高频文件类型补专属图标

这是当前投入产出比最高、同时最符合 NextClaw 产品气质的方案。它不是单纯“做得更好看”，而是在减少用户判断成本，让附件展示更像成熟入口产品的一部分，而不是一个还在解释自己的技术组件。
