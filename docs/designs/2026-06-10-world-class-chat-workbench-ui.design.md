# Chat UI 通用视觉质感改造设计

## 背景

这份文档只讨论通用前端视觉改造：如何让当前 Chat UI 看起来更简约、美观、大气、上档次。它不讨论 NextClaw 的产品定位、个人操作层愿景、执行链路重构或复杂业务架构。

目标是把现有界面从“功能齐全、能用、局部还不错”提升到“高级、克制、统一、耐看”的视觉水准。这里说的高级感主要来自基础设计能力：布局比例、留白、字体层级、颜色克制、边框阴影、组件一致性、交互状态和细节完成度。

## 目标气质

推荐目标不是炫酷、科技感、厚重视觉或强品牌包装，而是：

- 简约：减少视觉噪音，界面只强调真正重要的信息。
- 美观：比例、间距、字体、颜色和组件形态协调。
- 大气：布局有呼吸感，重点区域有足够空间，避免拥挤。
- 上档次：细节稳定，状态完整，控件精致但不花哨。
- 耐看：长时间使用不累，不靠强色块、重阴影、渐变背景制造第一眼刺激。

可以参考的通用审美方向：

- Linear 的克制层级和细腻边框。
- Arc / Raycast 的清晰操作密度。
- Notion / Craft 的留白和排版节奏。
- Apple 系统界面的轻量、稳定和统一。

不建议追求：

- 大面积渐变背景。
- 花哨动效。
- 过多卡片包裹。
- 高饱和撞色。
- 每个模块都加阴影和圆角。
- 过度“AI 科技感”的深色蓝紫视觉。

## 当前视觉问题

### 1. 视觉层级偏散

当前很多区域都有自己的局部样式：侧边栏、消息、输入框、工具卡、右侧按钮、空态卡片都在各自表达视觉重点。结果是单看每个局部都不差，但整体不够统一。

改造方向：

- 明确一级、二级、三级信息。
- 只让一个区域成为当前视觉中心。
- 降低次要按钮、说明、meta 信息的存在感。
- 避免多个大卡片同时抢注意力。

### 2. 圆角偏多偏大

当前界面里 `rounded-xl`、`rounded-2xl`、胶囊按钮和大圆角卡片使用较多。圆角太多会让界面显得软、散、玩具感强。

改造方向：

- 常规容器使用 8px 到 12px。
- 输入框可稍大，但不要无限接近“胖卡片”。
- 小 badge / chip 可以用圆角胶囊。
- 同一层级的组件使用一致圆角。

### 3. 留白节奏不够统一

有些地方很宽松，有些地方又很密；输入器和空态卡片较重，侧边栏列表又较挤。高级感来自稳定节奏，而不是单纯加大间距。

改造方向：

- 页面外框留白统一。
- 列表项内部留白统一。
- 消息正文和工具卡间距区分清楚。
- 输入区与消息区之间留足稳定缓冲。

### 4. 字体层级需要更克制

当前标题、按钮、列表、meta、正文有不少局部字号。通用高级 UI 通常字号不多，但权重、颜色和间距用得准。

改造方向：

- 控制字号层级数量。
- 标题不用过大，用字重和留白体现重点。
- meta 信息降低颜色，而不是无限缩小字号。
- 中文界面避免过紧字距和过多 uppercase 风格。

### 5. 色彩偏单一，但局部状态色又偏散

主基调是暖灰和橄榄绿，整体温和。但如果全部区域都用接近的暖灰，会显得糊；而状态色如果各处临时使用，又会显得散。

改造方向：

- 背景、容器、边框、选中态建立清晰灰阶。
- 主色只用于主操作和当前选中。
- 成功、警告、错误、运行中使用固定语义色。
- 减少单个组件临时发明颜色。

### 6. 边框和阴影需要降噪

高级 UI 很少靠重阴影表现层级。更多是轻边框、轻背景、非常克制的阴影。

改造方向：

- 普通容器用边框和背景区分。
- 浮层、输入器、popover 才使用轻阴影。
- 列表选中态不要靠厚阴影。
- 同屏阴影数量要少。

## 通用设计原则

### 1. 少即是多

先删视觉噪音，再加视觉效果。能用边框解决的，不用阴影；能用间距解决的，不用分割线；能用颜色深浅解决的，不加图标。

### 2. 一个屏幕一个视觉中心

空态时，中心欢迎区是主角。会话中，消息内容是主角。输入时，composer 是主角。不要让侧边栏大按钮、空态卡、输入框、右侧 dock 同时很重。

### 3. 统一比惊艳更重要

每个组件都好看，不等于整个界面高级。按钮、卡片、列表项、chip、tooltip、popover 的尺寸、圆角、边框、hover 都要统一。

### 4. 状态必须完整

高级产品不只是静态好看，还要 hover、focus、active、disabled、loading、empty、error 都稳定。任何只设计默认态的组件都会显得粗糙。

### 5. 细节克制

图标大小、线宽、间距、点击区域、文本截断、滚动条、分割线透明度，这些小东西比大面积视觉改造更影响档次。

## 视觉改造清单

### 1. 页面布局

建议：

- 左侧栏保持稳定宽度，但降低顶部大按钮的视觉重量。
- 主内容区背景更干净，减少渐变存在感。
- 右侧 dock 如果常驻，应更轻，不要像第三个强区域。
- 中间内容最大宽度保持舒适，不要让消息横向铺满。

验收：

- 一眼能分清侧栏、主内容、输入区。
- 主内容永远是视觉中心。
- 宽屏不空散，窄屏不拥挤。

### 2. 侧边栏

建议：

- 顶部品牌区更紧凑，版本和连接状态弱化。
- 新任务按钮缩小高度，减少厚重阴影。
- 搜索框高度、圆角和列表项保持同一体系。
- 时间/项目切换做成更标准的 segmented control。
- 会话列表项减少 hover-only 的关键操作，避免默认态信息不足。

视觉标准：

- 当前选中项清楚但不厚重。
- 未读、运行、子会话状态能扫读。
- 列表项之间的间距保持稳定。

### 3. 顶部标题栏

建议：

- 标题栏高度收敛，文字和 badge 垂直居中。
- title、session type、更多操作之间保持稳定间距。
- badge 用浅背景和细边框，不要太像按钮。
- 更多操作图标弱化，hover 再增强。

视觉标准：

- 标题是清楚的，但不抢正文。
- 长标题截断优雅。
- 移动端不拥挤、不换行乱跳。

### 4. 空态

建议：

- 空态减少大图标和大卡片数量。
- 三个能力卡可以更轻，或改为更细的 action rows。
- 文案短一点，视觉重心集中。
- 空态不要像 landing page，要像产品内部的起始状态。

视觉标准：

- 有呼吸感。
- 不营销化。
- 用户知道可以开始输入或选择能力。

### 5. 消息区

建议：

- 用户消息和助手消息的视觉差异保持，但不要过度气泡化。
- 助手消息正文优先阅读体验，减少厚边框和重阴影。
- 工具卡、代码块、引用块使用低一层的视觉层级。
- meta 信息更轻，复制按钮 hover 时出现或降低默认权重。

视觉标准：

- 长文本阅读舒服。
- 工具卡不会打断正文。
- 用户能快速扫出每轮对话边界。

### 6. 输入器

建议：

- 输入器是全页面最值得精修的组件。
- 外壳保持轻微浮起，但不要像巨大卡片。
- placeholder 颜色更克制。
- toolbar 分组明确：技能 / 模型 / 附件 / 发送。
- 发送按钮大小和输入器比例协调，不要显得过圆或过重。

视觉标准：

- 输入器精致、稳定、有焦点。
- 各控件像一套系统，不像拼在一起。
- 移动端工具栏压缩后仍能看懂。

### 7. 颜色

建议建立基础色阶：

- 页面背景：很浅、低存在感。
- 容器背景：白或接近白。
- 悬浮背景：略高一层。
- 选中背景：浅主色或浅灰，不用深色块。
- 主按钮：使用品牌色。
- 次级按钮：透明或浅灰。
- 危险操作：只在必要处使用红色。

避免：

- 大面积主色。
- 大面积暖黄/米色导致发糊。
- 所有状态都用 amber。
- 同一个区域里出现太多语义色。

### 8. 字体

建议字号层级：

- 页面标题：18 到 20。
- 区块标题：14 到 16。
- 正文：14 到 15。
- 列表项：13 到 14。
- meta：11 到 12。

字重建议：

- 标题 600。
- 重点按钮 600。
- 正文 400 / 500。
- meta 400 / 500。

避免：

- 到处加粗。
- 过多 `text-[11px]`。
- 过多 uppercase。
- 中文界面负字距。

### 9. 圆角

建议：

- 列表项：8px 到 10px。
- 输入器：16px 左右。
- 卡片：10px 到 12px。
- 小 chip：999px。
- 弹窗和 popover：12px 到 16px。

避免：

- 所有东西都是 20px 以上大圆角。
- 大按钮和大卡片同时很圆。
- 同一层级圆角不一致。

### 10. 阴影和边框

建议：

- 默认层级主要靠背景和边框。
- 输入器、popover、dialog 用轻阴影。
- hover 可以微调边框色，不一定加阴影。
- 滚动区域边界用很浅分割线。

避免：

- 每个卡片都有 shadow。
- hover 阴影跳动太明显。
- 边框颜色太深。

### 11. 图标

建议：

- 图标线宽统一。
- 常规图标 16px 到 18px。
- 小按钮图标 14px 到 16px。
- 只让主操作图标更醒目。
- icon-only 按钮必须有 tooltip 和 aria-label。

避免：

- 同一行图标大小不一。
- 图标颜色比文字更抢眼。
- 只靠图标表达复杂含义。

### 12. 动效

建议：

- 动效只用于反馈，不用于炫技。
- hover / active 150ms 左右。
- popover / menu 轻微 fade + scale。
- loading 用稳定 skeleton，不要大范围闪烁。

避免：

- 大幅位移动画。
- 过强 spring。
- 长时间循环动效。

## 第一阶段改造建议

第一阶段只做通用视觉提升，不做产品结构重构：

1. 收敛颜色、圆角、阴影、字体层级。
2. 降低侧边栏顶部和新任务按钮重量。
3. 优化会话列表项选中、hover、未读、运行状态。
4. 优化标题栏间距、badge 和更多操作。
5. 精修输入器比例、toolbar、发送按钮和 placeholder。
6. 统一消息、工具卡、代码块的边框和阴影层级。
7. 检查移动端标题栏和输入器压缩效果。

不做：

- 不重新定义产品定位。
- 不重构执行链路。
- 不引入复杂新组件体系。
- 不全站重做。
- 不为了高级感加装饰背景。

## 验收标准

### 视觉验收

至少看三张截图：

- 桌面宽屏。
- 窄桌面。
- 移动端。

每张截图检查：

- 是否有明确视觉中心。
- 是否有不必要的重阴影、大圆角、强色块。
- 主内容是否比侧栏和 dock 更突出。
- 输入器是否精致但不过重。
- 文字是否清晰、舒服、不过度加粗。

### 交互验收

检查状态：

- 默认态。
- hover。
- focus-visible。
- active。
- disabled。
- loading。
- error。
- empty。

尤其检查 icon-only 按钮是否有 tooltip 和 aria-label。

### 响应式验收

检查：

- 侧栏在桌面宽度下不挤。
- 主内容在窄桌面下仍可读。
- 移动端标题、badge、更多操作不重叠。
- 移动端输入器工具栏不丢核心信息。

## 实现注意

- 优先改组件自身样式，不用全局 CSS 反选组件内部结构。
- 如果一个样式是基础组件固有表现，应沉到基础组件或 shared primitive。
- 如果一个样式只属于 chat 页面布局，留在 chat feature。
- 不要为了一个视觉点新增复杂抽象。
- 不要把业务逻辑塞进纯 UI primitive。
- 改完必须截图验证，不能只靠肉眼想象。

## 推荐路线

推荐先做一轮轻量视觉 polish：

1. 先截图保存当前基线。
2. 统一 token 和基础视觉比例。
3. 改 sidebar。
4. 改 header。
5. 改 composer。
6. 改 message/tool card。
7. 做桌面、窄桌面、移动截图对比。

这一轮的成功标准很简单：不追求功能变化，只追求界面更简约、更协调、更克制、更像成熟产品。
