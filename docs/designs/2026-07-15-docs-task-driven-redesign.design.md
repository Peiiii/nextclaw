# NextClaw Agent 产品文档站重设计

## 1. 目标

这次不是给旧文档换主题、换首页或重新分组，而是按当前 NextClaw 的产品事实，从零建立一套 Agent 产品文档。

新版文档要让第一次接触 NextClaw 的读者依次看懂：

1. NextClaw 是什么，以及它和普通聊天助手有什么不同。
2. 一个任务如何从目标、材料和工作目录开始，经过执行与干预，最终交付回复、文件、网页或应用。
3. 结果在哪里查看，如何继续修改，如何把一次任务变成可复用能力。
4. NextClaw 有哪些真实工作表面，以及它们分别解决什么问题。
5. 如何安装、配置、接入渠道、控制权限并排查问题。

旧文档不再作为结构基线，只作为待核实的信息线索。页面是否保留、重写或退出导航，由当前源码、真实界面、产品愿景和用户任务共同决定。

## 2. 事实源顺序

所有用户文档按以下优先级取材：

1. `docs/VISION.md` 定义的产品方向。
2. 当前源码、路由、i18n 文案、真实运行界面与真实产品截图。
3. 已确认仍有效的 README、官网和发布说明。
4. WorkBuddy、QoderWork 等同类 Agent 产品的内容组织方法。
5. 旧文档中经重新验证仍然正确的事实。

旧页面的标题、篇幅、分组和表述不具备默认继承权。外部产品只提供类型认知和组织参照，不能替代 NextClaw 自己的产品事实。

## 3. 调研结论

### 3.1 WorkBuddy

WorkBuddy 的文档不是从配置和命令开始，而是先建立完整使用链路：简介、快速开始、创建任务、任务管理、任务对话、结果查看、首个任务。之后再拆解技能、自动化、右侧栏、模型、权限和记忆，并用数据分析、文件处理、本地应用、定时简报等案例说明结果。

可借鉴的不是具体品牌话术，而是三层结构：

- 产品认知：Agent 接收目标，规划并执行，交付可检查结果。
- 任务生命周期：创建任务、补充上下文、观察过程、干预执行、检查结果、继续迭代。
- 能力与实践：功能页解释工作表面，案例页给材料、指令、过程和产物。

### 3.2 QoderWork

QoderWork 进一步证明，Agent 案例不能只是场景名称。它的案例通常包含“能完成什么、可以怎样说、执行时发生什么、最后得到什么”，高级案例再把文件、浏览器、定时任务和消息渠道组合起来。

可借鉴的内容合同：

- 明确的用户待办；
- 可直接修改的示例指令；
- 输入材料与工作目录；
- Agent 的关键执行步骤；
- 可检查的文件、图表、页面或消息；
- 继续调整和复用的方法。

### 3.3 典型 Agent 产品的共同骨架

ChatGPT agent、Manus 等产品也围绕同一条主线组织认知：一个 Agent 在同一任务中使用浏览器、终端、文件和连接器，用户可以观察、暂停、接管或继续，最终检查它交付的结果。

因此，NextClaw 文档的第一层不应是“指南 / 教程 / 手册 / 参考”这些出版类型，而应是用户完成任务时实际经历的阶段。

## 4. NextClaw 的产品模型

文档中的统一产品模型是：

```text
目标 + 材料 + 工作目录
        ↓
Agent 理解任务并调用模型、技能、MCP、浏览器、终端和本机文件
        ↓
用户观察过程，补充要求，停止或继续
        ↓
回复 + 文件 + 源码 + Markdown + HTML + Panel App + 渠道消息
        ↓
继续修改 / 交给专用 Agent / 保存为技能 / 设为定时任务 / 从消息渠道再次调用
```

这条链路是首页、入门文档、功能文档和案例的共同骨架。不同页面只展开其中一段，不重新发明另一套产品解释。

## 5. 当前真实能力覆盖

从当前前端路由、i18n 文案、功能目录和真实截图确认，文档需要覆盖以下工作表面：

| 工作表面 | 用户问题 | 当前事实 |
| --- | --- | --- |
| 新任务与会话 | 如何发起、延续和拆分工作 | 支持工作目录、会话类型、历史会话和长上下文管理 |
| 会话工作区 | 任务材料和结果放在哪里 | 支持项目文件、打开文件、子会话、会话定时任务与右侧预览 |
| 结果与预览 | 如何检查 Agent 的交付物 | 支持源码、Markdown、HTML、文档、表格、演示文稿和 diff 等预览 |
| Agent | 如何为不同工作保留独立角色 | Agent 可拥有身份、记忆、技能、运行时和主目录 |
| 工具、技能与 MCP | Agent 如何获得执行能力 | 支持内置工具、技能市场、MCP 市场、工具检查与安装管理 |
| Panel Apps | 如何把结果变成持续可用的小应用 | 支持在会话旁打开本地面板应用，也支持应用列表与引用 |
| Doc Browser | 如何边工作边查看网页和资料 | 全局右侧栏支持文档、浏览器标签、停靠与浮动 |
| 自动化 | 如何让任务按时间继续运行 | 支持创建、启停、立即运行和查看下次/上次执行 |
| 消息渠道 | 如何从常用聊天工具发起任务 | 当前界面覆盖微信、飞书、QQ、钉钉、企微、Telegram、Discord、Slack、邮箱等渠道 |
| 模型与运行时 | 如何选择云端、本地或自定义能力 | 支持多家模型提供方、自定义 OpenAI 兼容接口及多种会话运行时 |
| 远程与本地部署 | 如何在自己的设备或服务器运行 | 提供桌面版、npm、Docker、后台运行和远程访问路径 |
| 权限与密钥 | Agent 可以访问什么 | 已有安全、密钥、渠道授权和服务应用授权配置表面 |

文档不照搬 WorkBuddy 的自动记忆、垂直工作台或企业知识库承诺；只有 NextClaw 当前真实存在并能从界面或源码验证的能力才进入正式说明。

## 6. 新信息架构

### 6.1 一级导航

- **开始**：认识产品、安装、创建第一个任务、查看结果。
- **工作方式**：任务、工作区、结果、Agent、工具、Panel Apps、自动化和渠道。
- **任务案例**：按真实待办组织的完整案例库。
- **安装与配置**：安装路径、模型、渠道、运行、远程和安全。
- **参考**：配置、命令、故障排查和版本更新。
- **更多**：项目、愿景、路线图、博客、社区和官网。

“更新”不继续占一个独立顶栏标签，进入“参考”下拉和项目局部导航。读者仍能快速找到，但不会打断首次使用路径。

### 6.2 开始：首次成功链路

1. NextClaw 是什么。
2. 选择安装方式。
3. 创建第一个任务。
4. 在任务中补充材料和要求。
5. 查看结果与文件。
6. 继续修改或开始下一步。

快速开始的成功标准不是“服务返回一条消息”，而是用户完成一个带工作目录或材料的真实任务，并在工作区里看见可检查结果。

### 6.3 工作方式：按产品表面讲清能力

- 任务与会话
- 会话工作区
- 结果、文件与预览
- Agents 与子任务
- 工具、Skills 与 MCP
- Panel Apps 与 Service Apps
- Doc Browser 与网页操作
- 定时任务
- 消息渠道
- 模型与运行时
- 权限、密钥与本地数据

这里回答“这个表面是什么、在哪里、什么时候用、会产生什么结果、有什么边界”。不混入长篇案例和安装手册。

### 6.4 任务案例：按待办而不是能力命名

首批案例至少覆盖：

1. 抓取网页数据并生成图表报告。
2. 整理、重命名并汇总一批本地文件。
3. 调研多个来源并写成带出处的文档。
4. 根据资料写文章、周报或方案初稿。
5. 汇总用户反馈并归纳问题优先级。
6. 生成配图并保留本地文件。
7. 开发一个可在右侧使用的本地小应用。
8. 检查和修改一个代码项目。
9. 定时生成简报并发送到指定渠道。
10. 从微信或飞书发起任务并接收结果。

每篇案例固定包含：最终交付、准备材料、示例指令、执行过程、检查结果、继续修改、复用为 Agent/Skill/定时任务/渠道工作流。

### 6.5 安装、配置与参考

安装和配置是任务成立的支撑层，不占首页主体。Desktop、npm、Docker 的信息必须完整保留；模型、渠道、后台运行、远程、安全、密钥和故障排查分别进入清晰页面。

## 7. 首页内容合同

首页不是旧导航的目录，也不是第二个营销官网。它承担五件事：

1. 用一句话说明 NextClaw 是本地优先的 AI Agent 工作台。
2. 用真实截图证明目标、执行和结果在一个工作表面里发生。
3. 展示“创建任务 -> Agent 执行 -> 检查结果 -> 继续或复用”的生命周期。
4. 让读者从核心工作方式或具体任务案例中直接进入。
5. 为已有明确问题的用户提供安装、配置、命令和故障排查入口。

首页避免抽象词堆叠，不解释改版原则，不出现竞品名称。截图只使用当前真实实例、默认或雾蓝主题，并与对应文字直接匹配。

| 旧内容 | 处置 | 原因 |
| --- | --- | --- |
| introduction / getting-started / after-setup | 完整重写 | 必须共同组成首次任务生命周期 |
| chat / sessions | 合并重写后保留兼容页 | 当前内容过薄，概念重复 |
| tools | 完整重写 | 不能只当旧链接占位，应讲清浏览器、终端、文件、技能和 MCP 的关系 |
| multi-agent | 完整重写 | 旧页只讲 runtime 路由，没有呈现 Agent 身份、主目录、记忆、技能和子任务 |
| cron / channels / model-selection | 完整重写 | 从配置提示升级为工作方式、用户价值、入口和边界 |
| tutorials/* | 保留有效配置事实，降为安装与配置层 | 不再承担产品骨架 |
| commands / configuration / troubleshooting | 保留并纳入参考 | 适合已知问题查阅 |
| project / notes / blog | 保留局部入口 | 不干扰首次使用路径 |
| 无事实、重复或只含跳转的页面 | 退出主导航，必要时保留兼容链接 | 避免旧链接失效，同时不让垃圾内容继续定义体验 |

## 9. 实现 owner

- `apps/docs/.vitepress/navigation/docs-navigation.config.ts`：唯一导航与侧栏 owner。
- `apps/docs/zh` 与 `apps/docs/en`：中英文用户内容，URL 和页面结构一一对应。
- `apps/docs/.vitepress/theme`：只负责文档视觉和响应式合同，不保存产品文案。
- `images/screenshots`：真实产品截图唯一 owner；文档站通过 `public/product-screenshots` 软链接引用，不复制第二份素材。
- `docs/designs`：内部设计依据；不得把本文件中的过程和判断写进用户页面。

## 11. 验收标准

1. 一个不了解 NextClaw 的读者只看首页和“开始”章节，就能解释 Agent 如何从目标走到可检查结果。
2. 工作方式章节完整覆盖当前真实产品表面，不能再出现大量只有几行的占位页。
3. 案例标题都是具体待办；每篇都有材料、指令、执行、结果和复用方式。
4. 安装方式、模型、渠道、命令和故障排查没有因重构丢失。
5. 中英文页面结构一致，所有导航链接和图片可访问。
6. 桌面端与移动端无横向溢出；真实截图清晰、完整、主题一致。
7. 用户页面不出现设计过程、竞品引用、内部取舍或未经验证的能力承诺。
8. VitePress build、TypeScript、ESLint、i18n、治理检查与真实浏览器冒烟全部通过。

## 12. 非目标

- 不承诺当前产品没有的垂直专家工作台、企业知识库或自动长期记忆机制。
- 不复制 WorkBuddy 或 QoderWork 的品牌文案、页面句子和专有隐喻。
- 不在本轮提交、推送或部署；先提供本地完整效果供确认。
