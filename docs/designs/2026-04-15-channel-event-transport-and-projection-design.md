# 渠道事件传输与语义投影设计

日期：2026-04-15

相关实施计划：

- [2026-04-15-channel-reply-runtime-simplification-plan.md](../plans/2026-04-15-channel-reply-runtime-simplification-plan.md)

## 1. 背景

这份文档沉淀本次关于“渠道通信内核应该如何分层”的讨论结果。

讨论起点不是抽象地争论 `channel`、`event bus` 或 `endpoint` 哪个名字更好，而是一个非常具体的真实问题：

- 当前微信渠道只能在 Agent 全部完成后一次性发送最终消息。
- 但更好的体验应该是边说边做，至少在一个稳定文本块完成时就及时发送。
- 之前尝试为微信补这类能力时，明显感受到当前架构实现成本高、路径不清晰、特判很多。

这说明我们需要先回答一个更基础的问题：

**NextClaw 的渠道通信底座，到底应该把“传输”与“语义解释”分在哪一层。**

## 2. 上位目标对齐

结合 [VISION.md](../VISION.md)，这次设计判断要服务以下目标：

- 让 NextClaw 更像统一入口，而不是每接一个渠道就重新发明一次通信语义。
- 让不同渠道最终收敛到同一套内核事件，而不是长期维持渠道特判拼接。
- 让未来接更多输入源、输出目标、订阅源、自动化流时，新增成本主要落在 adapter，而不是重新改内核。

这次不是为了“把协议讲得更漂亮”，而是为了给后续微信、飞书、Webhook、订阅型输入、自动化投递等场景提供一个更稳定的落点。

## 3. 本次结论

### 3.1 最底层不应理解业务语义

最底层纯通信层只负责：

- 发布事件
- 订阅事件
- 保证顺序、路由、投递、取消、重连等基础能力

最底层**不应**关心事件是：

- `message.text-delta`
- `message.tool-call-start`
- `message.completed`
- 还是任何未来新增事件

换句话说：

**transport / event stream 只传事件，不解释事件。**

### 3.2 统一事件语言应尽量复用 NCP

当前仓库里已经有一套比旧 channel bus 更完整的事件定义，主要在 NCP 中：

- `message.text-*`
- `message.reasoning-*`
- `message.tool-call-*`
- `message.completed`
- `run.*`

这套事件语言已经足够表达“边说边做”“工具生命周期”“最终完成态”等大部分渠道场景。

因此本次结论是：

**未来渠道通信的统一事件层，应尽量直接复用 NCP 事件定义，而不是再平行发明第二套事件协议。**

如果后续确有缺口，应优先补充 NCP 事件，而不是在渠道层私自长出新语义。

### 3.3 语义解释不应分散到每个渠道

虽然底层不应理解业务语义，但也不代表每个渠道 adapter 都要各自直接生啃全部原始 NCP 事件。

否则会出现：

- Telegram 一套 flush 规则
- 微信一套 flush 规则
- 飞书一套 flush 规则
- 最终同一语义在不同渠道长期漂移

因此需要一层共享的语义投影：

**先由共享 projector / planner 把 NCP 事件流解释成较稳定的投递语义，再由各渠道 adapter 决定如何呈现。**

## 4. 推荐分层

本次推荐的分层如下：

### L0：Event Transport

职责：

- 事件发布
- 事件订阅
- 顺序、路由、取消、背压、连接管理

禁止：

- 猜测业务完成条件
- 理解特定事件名的产品语义
- 为了某个渠道增加特定分支

### L1：NCP Event Protocol

职责：

- 定义标准事件种类
- 统一事件 payload 结构
- 提供跨运行时、跨渠道的公共事件语言

这层表达的是“系统里发生了什么”，例如：

- 文本开始/增量/结束
- reasoning 开始/增量/结束
- tool call 开始/参数增量/结束/结果
- 完成、失败、中止

### L2：Shared Projector / Delivery Planner

职责：

- 消费原始 NCP 事件流
- 推导更稳定的上层投递语义
- 决定何时 flush、何时结束 typing、何时进入 final

这层不直接绑定某个渠道 API，但允许有明确的策略，例如：

- 文本块何时被视为“足够稳定，可以对外发送”
- tool call 开始时是否应先 flush 前面的文本
- completed 到达时是否应补一次最终收尾

### L3：Channel Adapter

职责：

- 把共享投影结果映射成具体渠道动作

例如：

- 微信：一个稳定块完成就发一条消息
- Telegram：先发送草稿，再 edit 更新
- 邮件：只在 final 时输出
- 飞书：可按卡片、分块消息、线程回复等不同方式呈现

## 5. 对当前代码的判断

### 5.1 现状不是“纯事件底座 + 共享解释层”

当前渠道链路大致是：

1. 旧 channel bus 接收入站消息
2. gateway 把消息路由到 NCP 执行
3. NCP 产生事件流
4. gateway 把其中一小部分事件压扁后重新塞回旧 channel 输出

这里的核心问题是：

- 统一事件层存在
- 但渠道层没有直接站在统一事件层上
- 中间被旧 `OutboundMessage` 终态消息模型截断了

### 5.2 旧 bus 的主模型仍然偏“最终消息”

当前旧 bus 的核心输出是 `OutboundMessage`：

- `channel`
- `chatId`
- `content`
- `media`
- `metadata`

这个模型适合“最终发一条消息”，但不适合天然表达：

- 一段文本的阶段性完成
- tool 前后边界
- 多块输出
- 草稿更新
- 渠道级增量投递策略

### 5.3 当前流式能力是特例旁路，不是一等能力

目前已经存在一条旁路：

- gateway 把 `message.text-delta` 压成私有 control message
- Telegram 特判消费这些 control message
- 微信不消费，因此仍只能等最终消息

这说明当前“流式渠道输出”并不是正式共享能力，而是：

**NCP 事件 -> 私有 control message -> 个别渠道特判**

这种结构会导致：

- 只有少数渠道能做流式体验
- 每加一个渠道都要重新补一套旁路逻辑
- 无法自然支持 reasoning / tool-call 等更丰富边界

## 6. 微信案例分析

### 6.1 目标体验

微信场景下，更理想的表现不是：

- 等整轮完成后一次性发送全部文本

而是：

- 当一段稳定文本已经形成，就尽快发送
- 如果后面进入 tool call，可以先把前面已经稳定的文本发出去
- 最终完成时再做必要收尾

这里强调的是：

**不是一定要逐 token 流式输出，而是要支持“按稳定块增量投递”。**

### 6.2 当前为什么难做

当前微信难做，不是因为微信 API 必然做不到，而是因为当前架构里缺了一层正式的共享投影能力。

现在如果想做微信增量体验，通常只能选下面两条坏路：

1. 直接在微信 adapter 里理解大量 NCP 原始事件
2. 继续在 gateway 和 control message 层堆微信特判

这两条都不理想：

- 前者会让渠道 adapter 过重
- 后者会让旁路越来越多

### 6.3 正确落点

正确做法不是让微信直接依赖 `final message`，
也不是让 transport 层知道“什么时候该发微信消息”，
而是：

**让微信订阅统一事件流，再通过共享 projector 得到较稳定的 flush 信号和文本块，然后由微信 adapter 决定如何发送。**

## 7. 设计原则

后续做这条链路时，默认遵循以下原则：

### 7.1 传输层不解释业务

- 不在 transport 里写“看到某事件就算完成”
- 不在 transport 里写“看到某事件就触发微信发送”
- 不在 transport 里做渠道白名单

### 7.2 统一事件层只保留一份

- 尽量直接复用 NCP 事件
- 如果不够，补 NCP
- 不再平行发明第二套渠道事件定义

### 7.3 共享语义解释优先

- flush 规则
- block 边界
- typing 结束时机
- final 收尾规则

这些优先放共享 projector，而不是散落在每个渠道里。

### 7.4 渠道 adapter 只负责“如何呈现”

渠道 adapter 关注：

- 发一条
- edit 一条
- reply 一条
- 发卡片
- 发线程消息

但不应拥有整套跨渠道语义解释权。

## 8. 对后续实现的直接影响

如果后续继续推进微信“按块发送”，最小正确方向不是继续补旧 control message，而是：

1. 保留底层事件传输的语义中立
2. 让渠道主链尽量直接消费 NCP 事件
3. 在 NCP 事件与渠道 adapter 之间新增共享 projector / planner
4. 让旧 `OutboundMessage` 直接退出 reply 主链，而不是继续作为长期兼容层

这不要求一次性重写全部入站链路，但 reply 主链本身应尽量单切，不应长期保留“双主链”。

## 9. 暂不在本次文档内决定的内容

以下内容在本次讨论中故意不定死：

- reasoning 是否对外可见，或默认只用于内部策略判断
- tool call 结果是否直接对终端用户可见
- 微信是否采用“逐块发新消息”还是“草稿消息 + 更新”

以下内容已经在实施计划中定死，不再反复讨论抽象：

- reply 唯一上游真相源就是 NCP event stream
- reply 共享解释层就是一个 projector，而不是每个渠道自己解释
- reply 唯一出站 owner 应收敛为共享的 `OutboundInteractionOrchestrator`
- 旧 `assistant_stream` / outbound / final-string bridge 不作为长期主链保留

这些都属于下一轮更具体的设计与实现问题。

本次文档只固定最关键的架构判断：

**纯传输层保持语义中立；**
**统一事件语言尽量复用 NCP；**
**共享语义解释层负责把原始事件流投影成渠道可消费的投递语义；**
**渠道 adapter 只负责最终呈现。**

## 10. 结论

本次讨论最终收敛出的不是一个“大而全的新协议”，而是一条更清晰的边界：

- 底层：传事件，不解释
- 中层：统一事件语言
- 上层：共享语义投影
- 末端：渠道呈现

如果这条边界成立，那么未来不管接：

- 微信
- 飞书
- 邮件
- Webhook
- 订阅型数据源
- 任意新的输入/输出系统

新增工作的主要成本都应落在：

- adapter
- projector 策略补充
- 少量事件扩展

而不应再反复撕开通信内核本身。
