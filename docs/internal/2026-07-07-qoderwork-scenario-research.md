# QoderWork 场景表达调研

## 1. 背景

本调研服务于 NextClaw 官网和产品介绍的场景表达升级。QoderWork 与 NextClaw 有相似处：都是桌面端、Agent 执行、本地文件、浏览器/自动化、技能扩展和日常工作场景。因此它的表达方式尤其值得参考。

本文件对齐 [NextClaw 产品愿景](../VISION.md)：NextClaw 不只是聊天工具，而是用户使用软件、互联网、系统、服务与云计算的统一入口。调研重点是 QoderWork 如何把“统一入口和能力编排”翻译成用户能代入的具体任务。

## 3. 官方定位摘要

QoderWork 的核心表达是“桌面级通用智能体助手”。它把 Qoder 的 Agent 能力从代码领域扩展到日常工作，强调用户用自然语言描述需求，系统自动规划、执行并交付结果。

它的官方文档不是只列功能，而是明确拆成：

- 核心能力：文件处理、浏览器自动化、桌面控制、定时任务、IM 频道、技能与扩展、语音输入、跨会话记忆。
- 典型场景：文件整理、照片管理、数据分析、文档创作、研究整合。
- 工作台：通用、设计、幻灯片、写作。
- 用户案例：年度总结、联网调研、客户答辩 PPT、内部平台配置发布自动化等。

## 5. QoderWork 的场景组织方式

### 5.1 第一层：日常工作入口

QoderWork 首屏先讲“文件整理、数据处理、文档生成”，这些都是普通用户熟悉的工作对象。它没有先讲模型、协议、插件或 Agent 架构。

这对 NextClaw 的启发是：用户第一眼应该看到自己已经在做的工作，而不是系统能力名词。

### 5.2 第二层：任务故事

QoderWork 产品页用多个任务故事表达价值：

- Reddit 用户反馈调研：抓取社区讨论，分析痛点和购买因素，输出结构化洞察。
- 教学算法动画：把抽象算法做成可视化演示。
- 销售数据分析：读取 Excel，做多维分析、可视化图表和分析报告。
- 博客配图：理解文章主题，生成匹配配图并导出。

每个故事基本包含：

- 任务对象。
- 传统痛点。
- QoderWork 处理方式。
- 时间缩短或产物结果。

这种结构很强，因为用户看到的不是“它支持浏览器自动化”，而是“它能帮我做一次竞品反馈调研”。

### 5.3 第三层：典型案例库

QoderWork docs 的“典型案例”写得更像可复制 prompt 库。它把每个场景拆成：

- 场景说明。
- 提示词示例。
- 执行效果。

覆盖的基础场景包括：

- 文件整理：识别类型、创建目录、移动文件、生成整理日志。
- 照片管理：读取 EXIF、按时间建目录、重命名、生成清单。
- 数据分析：读取数据、统计指标、找 Top 项、按月份统计趋势、生成 HTML 报告。
- 文档创作：生成 Word 文档、目录、页眉页脚、流程图和架构说明。
- 研究整合：搜索资料、整理对比、总结趋势、生成 Markdown 报告。

这比“用例卡片”更进一步：它给用户提供了几乎可以直接复制的任务模板。

### 5.4 第四层：真实用户案例

QoderWork 的用户案例文档把任务写成故事，而不是短功能点。例如：

- 50 多份文档整合为年度工作总结。
- 自动联网搜索并生成可汇报 PPT。
- 基于客户资料生成答辩 PPT 和演讲稿。
- 把复杂发布流程封装为可复用 Skill。

这里最重要的是“材料规模”和“交付物”。用户能马上判断：这不是一句短回答，而是一个完整工作链路。

### 5.5 第五层：工作台分化

QoderWork 用工作台解决“一个聊天框装不下所有任务”的问题：

- 通用：文件整理、数据处理、信息汇总、自动化。
- 设计：以代码方式生成可运行、可编辑、可交付的设计产物。
- 幻灯片：生成 PPT。
- 写作：长文档和结构化写作。

这说明它把“场景”从文案进一步落到了产品结构。不同场景不只是换一句 prompt，而是对应不同视图、工具组合和产物标准。

## 6. 它最清楚的表达技巧

### 6.1 场景标题都包含对象和结果

好的例子：

- 文件整理。
- 销售数据分析。
- 联网调研 + 分析 PPT。
- 企业年度工作总结自动生成。
- 客户答辩 PPT 生成。

这些标题不是“支持文件能力”“支持网页能力”“支持技能扩展”，而是用户手头可能真的有的工作。

### 6.2 把复杂能力拆成用户步骤

QoderWork docs 常用“选择工作目录、描述需求、回答澄清、确认大纲、生成产物”这类步骤。这样用户知道任务从哪里开始，也知道中途可以干预。

NextClaw 官网如果只说“自动完成多步任务”，会比较虚。可以改成：

- 选一个本地目录。
- 发起任务。
- 它读取资料、调用技能、生成中间结果。
- 用户确认关键节点。
- 结果留在会话里，下次继续改。

### 6.3 给高级能力一个现实落点

QoderWork 的专家套件不是只说“可以安装技能”，而是说团队可以把岗位流程、判断标准和输出标准封装成一套方案并分发。

对 NextClaw 来说，技能市场、MCP、Agent 管理也要找现实落点：

- 不是“安装技能”，而是“以后每周都按同一套流程写周报”。
- 不是“接入 MCP”，而是“把 CRM、文档库、浏览器和本地文件放进同一个任务”。
- 不是“创建 Agent”，而是“让不同 Agent 分别处理研究、数据、写作和审校”。

## 7. 对 NextClaw 的可借鉴场景

### 7.1 适合首页展示的场景

| 场景 | 用户原话示例 | NextClaw 可展示结果 |
| --- | --- | --- |
| 文件整理 | 帮我整理这个下载目录，把文件按用途归类 | 目录结构、移动日志、整理摘要 |
| 数据分析 | 读取这份 CSV，做趋势图和结论 | HTML 报告、图表、结论段落 |
| 研究整合 | 查一下这个产品领域，给我竞品对比 | 来源列表、对比表、报告草稿 |
| 写作 | 用这些材料写一份发布说明 | 草稿、修改记录、最终文档 |
| PPT/汇报 | 把调研结果整理成汇报大纲 | 大纲、讲稿、素材清单 |
| 图片/设计辅助 | 给这篇文章配一张图，风格干净一点 | 图片候选、使用说明 |
| 自动化 | 每天 9 点汇总昨天项目状态 | 定时任务、执行记录、推送结果 |
| IM 协作 | 群里有人发文件时，帮我先总结重点 | 群聊入口、主工作台上下文、结果回传 |

### 7.2 适合做“用例库”的场景

NextClaw 可以考虑在官网或文档中单独做一组“可以直接试的任务”：

- 从网页抓取表格，生成 CSV 和图表。
- 把本地一组 Markdown 整理成知识库目录。
- 把会议纪要转成行动项，并推送到飞书。
- 把项目 README、代码结构和 issue 整理成开发计划。
- 把客户反馈汇总成优先级列表。
- 生成一个本地小工具，比如 CSV 合并器、图片压缩器、发票整理器。
- 设置每日简报或定时检查链接变化。

这组内容应当是“任务模板”，不是普通功能介绍。

## 8. 对 NextClaw 官网的启发

### 8.1 场景区和能力区要分开

场景区回答：

- 我能拿它做什么？
- 我手头有什么材料？
- 它交付什么结果？

能力区回答：

- 它为什么能做到？
- 它能接哪些模型、渠道、工具和技能？
- 它如何持续执行或复用？

如果把“自定义模型”“安装技能”“统一入口”放进场景区，用户不会产生任务代入。它们应该放在能力区。

### 8.2 可以引入“任务模板”而不只是“用例卡片”

QoderWork 的文档优势在于可复制。NextClaw 后续可以新增一个轻量页面：

- 标题：可以直接交给 NextClaw 的任务。
- 每个任务包含：适合材料、示例指令、预期结果、用到的能力。
- 示例指令必须具体，避免泛泛说“帮我分析一下”。

### 8.3 官网截图要服务场景

QoderWork 的产品页经常把案例和产物绑定。NextClaw 目前已有主工作台、Agent 管理、定时任务、技能市场截图，但还可以进一步让截图和场景绑定：

- 主工作台截图：展示一个正在完成的数据分析或文档整理任务。
- Agent 管理截图：展示不同 Agent 分工，不只是卡片列表。
- 定时任务截图：展示每日简报、状态巡检、链接监控等任务。
- 技能市场截图：展示某个任务需要的技能如何被安装和使用。

## 9. 应该避免照搬的地方

- QoderWork 有明确的“设计 / 幻灯片 / 写作”工作台。NextClaw 当前如果没有同等产品化视图，不应假装已有。
- QoderWork 的效率数字很强，但 NextClaw 不应随意写类似时间缩短，除非有可复现实测。
- 专家套件的企业分发语义可以参考，但 NextClaw 目前更适合先讲个人和小团队场景，不宜过早拔到企业级治理。

## 10. 未决问题

- NextClaw 是否要做“任务模板库”页面，还是先在首页用例区小规模呈现。
- 当前截图是否能展示真实产物，而不仅是界面入口。
- 是否需要为数据分析、写作、文件处理、小应用开发分别准备一组可复现 demo 数据。
- 技能市场、Agent 管理和定时任务怎样才能被用户理解为“完成任务的过程”，而不是单独功能页。
