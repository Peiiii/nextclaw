# WorkBuddy 场景表达调研

## 1. 背景

本调研服务于 NextClaw 官网和产品介绍的场景表达升级。目标不是复刻 WorkBuddy 的说法，而是看清它如何把“AI Agent 办公工具”讲成用户能理解的任务场景。

本文件对齐 [NextClaw 产品愿景](../VISION.md)：NextClaw 追求统一入口、能力编排、自感知、自治、自进化和生态扩展；因此对竞品的学习重点，是它如何让用户快速感知“我可以把什么事交给它做”，而不是照搬能力名词。

## 3. 官方定位摘要

WorkBuddy 的核心表达是“桌面端 AI Agent 办公工具 / AI 工作台 / AI 专家团”。它不是先讲模型、插件或协议，而是先讲用户用一句话发起任务，由系统规划、执行并交付结果。

官网 meta 的核心意思是：

- 它来自腾讯云代码助手团队。
- 它面向 AI Agent 办公。
- 它强调自主规划和多模态复杂任务交付。
- 它支持多 Agent 并行。

腾讯云 TokenHub 文档把它描述为适配 QQ / 企微生态、支持本地电脑操作和多模型切换的桌面智能体，并把商业化模型配置作为文档重点。

腾讯新闻稿里，WorkBuddy 面向非技术背景职场人群，强调自然语言描述任务、本地电脑执行、批量处理文件、生成文档/表格/PPT、多模态创作、数据分析、行业调研和多任务 Agent 并行。

## 5. WorkBuddy 的场景组织方式

### 5.1 第一层：结果承诺

WorkBuddy 的第一层不是列功能，而是给出一个强结果承诺：用户只需给任务，系统规划并交付结果。

这层表达解决的是用户最早的问题：

- 我需要学很多配置吗？
- 它只是聊天，还是能做事？
- 它能不能处理多步骤任务？
- 它最后会不会交付一个可用产物？

### 5.2 第二层：专家团隐喻

WorkBuddy 用“专家团”来包装 Agent、多角色、技能和任务分解。这个隐喻的好处是普通用户不需要理解 agent runtime，也能明白：

- 不同工作可以交给不同角色。
- 一个复杂任务可以被拆开。
- 任务不是一个聊天模型单线程回答，而是多个角色协作。

它公开展示的角色方向覆盖运营、设计、数据、开发、财务、法务等。这里的重点不是角色名单本身，而是它把抽象能力翻译成岗位心智。

### 5.3 第三层：任务场景

WorkBuddy 官网和腾讯系介绍反复出现的场景包括：

- 软件开发：需求、架构、任务拆分、代码实现、质量验证。
- 内容创作：视频、图文、剪辑、跨语言素材改编。
- 外部调研：行业/竞品市场分析、调研报告、演示文稿。
- PPT/报告：根据主题或模板收集资料并生成汇报材料。
- 业务数据洞察：分析数据表格或日志，提炼问题、规律和建议。
- 用户反馈分析：做情感分类、问题提炼和周期性洞察。
- 销售业绩分析：分析销售管道、成交/丢单原因，输出跟进建议。
- 文件与本地电脑操作：读取授权文件夹、批量处理文件。

这些是“场景”，因为它们都有对象、动作和结果。相反，“多模型切换”“MCP Server”“Skills 技能包”只是能力支撑，不是用户场景。

### 5.4 第四层：生态和接入

WorkBuddy 也讲 QQ、企微等生态接入，多模型、MCP、Skills 技能包等能力。但这些内容通常作为“为什么能做”的支撑层出现，而不是作为首页唯一主叙事。

对 NextClaw 有启发的一点是：微信、飞书、钉钉、QQ 等入口应该出现在“任务从哪里来、结果回到哪里去”的场景里，而不是只以 logo 墙或能力清单出现。

## 6. 它最清楚的表达技巧

### 6.1 把 Agent 解释成岗位协作

“Agent 管理”“多 Agent 并行”对新用户是技术词；“运营、设计、数据、开发一起处理一个任务”更容易懂。

NextClaw 如果展示 Agent 管理界面，不应该只说“管理不同分工的协作者”，可以补具体任务：

- 研究 Agent 搜集资料。
- 写作 Agent 起草报告。
- 数据 Agent 清洗并画图。
- 审校 Agent 检查格式和结论。

### 6.2 把工具能力解释成可交付产物

WorkBuddy 不停把任务落到报告、PPT、图表、方案、洞察、代码、视频这类产物上。用户看到产物，才知道它能替自己完成哪一段工作。

NextClaw 官网也应该优先讲：

- 生成什么文件。
- 保存在哪里。
- 能不能继续修改。
- 能不能发回微信/飞书。
- 能不能下次复用。

### 6.3 用“传统痛点 -> AI 方案 -> 时间收益”增强代入

WorkBuddy 和 QoderWork 都在用类似结构：原来要手动搜集、整理、分析、排版，现在交给 Agent 做。

NextClaw 可以借鉴这个结构，但不要写成空泛效率口号。更适合写成具体例子：

- “把 20 份客户反馈整理成问题清单。”
- “从 CSV 生成一份带图表的 HTML 报告。”
- “每天早上把昨日日志异常发到飞书群。”

## 7. 对 NextClaw 的可借鉴场景

### 7.1 可直接吸收的场景类型

| 场景 | 用户原话示例 | NextClaw 可展示结果 |
| --- | --- | --- |
| 资料调研 | 帮我查这个行业最近三个月的重要变化 | 摘要、来源列表、对比表、报告草稿 |
| 数据分析 | 分析这份销售表，找出下滑原因 | 图表、统计表、结论、后续动作 |
| 文件整理 | 把这个项目目录里的资料按用途整理好 | 新目录结构、移动日志、摘要 |
| 写作产出 | 把这些材料整理成一篇发布说明 | 文档草稿、改写版本、引用材料 |
| 周期简报 | 每天早上汇总昨天的项目状态 | 定时任务、推送记录、会话上下文 |
| 跨渠道协作 | 群里有人提问时先收集资料再回复 | 微信/飞书入口、主工作台过程、结果回传 |
| 多 Agent 分工 | 让一个 Agent 调研，一个 Agent 写稿 | Agent 列表、任务状态、最终汇总 |

### 7.2 应该避免照搬的地方

WorkBuddy 的“专家团”和“一个人顶一支团队”很有传播感，但也容易变成夸张承诺。NextClaw 更适合保持可信：

- 不说“替代一个团队”。
- 不说“全行业专家”。
- 不承诺所有复杂任务都能自动完成。
- 重点讲“可接入、可编排、可留痕、可继续做”。

## 8. 对 NextClaw 官网的启发

### 8.1 首页用例区应该写具体任务

建议首页用例卡片优先呈现这些任务：

- 把一批文件整理成项目资料夹。
- 抓取网页和表格数据，生成图表报告。
- 把会议记录、聊天记录和文档整理成周报。
- 做一次竞品/行业调研，输出对比表和结论。
- 写一个本地小工具，后续继续改。
- 定时生成简报并推送到飞书或微信。

### 8.2 能力区再讲支撑能力

能力区可以讲：

- 自定义模型。
- 微信、飞书、钉钉、QQ 等渠道。
- Skill / MCP / CLI 工具。
- 定时任务。
- 本机文件与浏览器操作。
- 多 Agent 管理。

这些能力必须服务场景，不要冒充场景。

## 9. 未决问题

- NextClaw 是否要采用“专家团”类心智，还是保持“个人操作层 / 工作台 / 协作者”心智。
- 多 Agent 分工在当前产品里是否足够稳定，能不能作为首页重点表达。
- 微信/飞书接入在官网里应该展示为“入口能力”，还是展示为真实跨渠道任务故事。
- 是否要为官网准备一组真实演示任务，保证每个场景都有截图或可复现样例。
