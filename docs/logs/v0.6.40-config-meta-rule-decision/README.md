# v0.6.40-config-meta-rule-decision

## 迭代完成说明（改了什么）

- 更新 `/config-meta` 指令定义：要求执行时先自行判断是“修正已有规则”还是“删减/新增规则条目”。
- 更新 `AGENTS.md` 的 `/config-meta` 索引说明，明确该判断机制。
- 更新 Rulebook 条目 `config-meta-no-new-modules`：
  - 保留“未明确授权不得新增一级模块”的约束；
  - 增加“必须先做判断再执行修正/删减/新增”的执行要求。
- 将“迭代说明文档必须包含用户/产品视角验收步骤”从该 Rule 下移出，统一上移到 `AGENTS.md` 的“迭代制度（docs/logs）”模块作为全局文档要求。
- 同步更新 `commands/commands.md` 中 `/config-meta` 描述，保持索引一致。

## 测试 / 验证 / 验收方式

- 本次仅涉及文档与规则机制变更，不涉及运行时代码、构建产物或接口行为。
- 规则一致性验收：
  - 检查 `AGENTS.md` 与 `commands/commands.md` 对 `/config-meta` 的定义是否一致；
  - 检查 Rulebook 条目 `config-meta-no-new-modules` 是否仅保留规则决策与模块边界约束；
  - 检查“迭代制度（docs/logs）”是否已包含“用户/产品视角的验收步骤”作为必填项。

### 用户/产品视角验收步骤

1. 以产品负责人身份提出一个模糊元需求，例如：“请用 `/config-meta` 调整规则机制，避免冲突并补齐缺失规则”。
2. 检查执行结果是否先给出“判断结论”：本次是修正已有规则，还是删减/新增规则条目（而不是直接机械改文案）。
3. 检查迭代说明文档是否明确给出用户可读的验收路径（例如：我该看哪些规则、如何判断改动已生效、何时算通过）。
4. 检查结果是否符合产品目标：
   - 规则可理解（不是只改措辞）；
   - 规则可执行（有明确触发条件和行为）；
   - 规则可验收（有可复核步骤）。

- 因改动范围纯文档，`build/lint/tsc` 对行为无增量验证价值，本次标记为不适用。

## 发布 / 部署方式

- 合并文档变更后立即生效。
