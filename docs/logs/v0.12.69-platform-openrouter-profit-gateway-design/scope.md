# Scope（OpenRouter 型中转）

## 1. 产品定位

NextClaw 在该阶段的定位是：**平台托管上游的 OpenRouter 对齐中转站**。  
用户只对接 NextClaw，不直接接入上游厂商，通过统一模型目录和统一 API 使用模型能力。

## 2. Goals（要做什么）

- 提供统一 OpenAI 兼容接口（`/v1/*`），作为唯一对外模型访问入口。
- 对外模型命名采用 OpenRouter 风格 `provider/model`（例如 `openai/gpt-4o`）。
- 平台侧统一管理上游连接、路由、定价、限流、熔断。
- 建立经营闭环：每次请求可核算收入、上游成本、毛利。
- 默认优先平台侧 OAuth 接入上游账号，降低凭证管理风险。

## 3. Non-Goals（明确不做）

- 不让用户自带上游 API Key/OAuth 凭证直接接入平台。
- 不允许用户绕过平台直接访问上游厂商接口。
- 不向用户暴露上游凭证与上游侧账单明细。
- 不做“仅转发不算利润”的工具型网关。

## 4. Not-Now（暂时不做）

- 上游 API Key 接入路径后置（先完成平台 OAuth 主路径）。
- 企业级复杂结算能力后置（合同账期、多租户财务体系）。
- 跨境税务/发票系统后置。
- 复杂 BI/数仓体系后置（先满足运营必需指标）。

## 5. 命名与对齐策略

- 平台公开模型标识：`provider/model`（OpenRouter 对齐）。
- 该命名会显式体现上游供应商语义，属于产品设计目标，不是泄露内部实现。

## 6. 边界变更机制

- 任何“做/不做/暂不做”调整，必须先更新本文件，再同步更新接入文档与迭代文档引用。
