# 可移植微应用模型设计

## 这份文档回答什么

这份文档用于沉淀本轮关于“新型微应用形态”的讨论结论，重点回答下面几个问题：

1. 我们到底想定义一种什么新的应用类型。
2. 它和传统 Web 应用、本地应用、Docker 应用、小程序分别有什么不同。
3. 为什么“安全”只是必要基础，不应该成为这类应用的唯一主叙事。
4. 如果它既要轻量、可移植、可安装，又允许存在受控逻辑层，推荐的技术与产品模型应该是什么。
5. 这套模型未来如何独立成立，以及之后如何再考虑接入 NextClaw 一类宿主产品。

这不是一份立刻进入某个单一技术实现的任务单，而是一份上层产品与架构收敛文档。目标是先把模型定义清楚，再决定是否直接进入原型实现。

## 长期目标对齐

虽然本轮收敛刻意不把方案绑定到 `NextClaw` 当前产品边界，但它和 `docs/VISION.md` 的长期方向高度一致。

原因不在于“又多了一个能力”，而在于它可能补上一个长期缺口：

- 如果 AI 时代真的要让普通用户大量使用、安装、组合、分享别人做出来的能力单元，那么只靠“运行一个仓库”“拉一个 Docker 镜像”“装一个脚本”是不够的。
- 真正缺失的是一种比传统全栈项目更轻、比纯网页更强、比容器更低门槛、又比裸跑本地服务更可信的应用形态。
- 这种形态一旦成立，就有机会成为未来 agent 生态、个人操作层生态、能力市场生态的统一载体之一。

换句话说，这份设计不是在讨论“做一个更安全的工具”，而是在讨论“是否能定义一种更适合 AI 时代分发和使用的软件形态”。

## 问题重新定义

当前问题很容易被误写成：

- “怎样做一个更安全的应用运行方式”
- “怎样把后端塞进 Wasm”
- “怎样做一个轻量版 Docker”

这些表述都不够准确。

更准确的问题应该是：

**我们能不能定义一种新的微应用形态，让用户拿到的是“可安装、可运行、可理解、可授权”的应用，而不是一套部署说明；同时它又不退化成只能做静态前端页面的弱能力载体。**

这里的关键不是把“安全”做成宣传词，而是让下面几件事同时成立：

1. 应用应该像应用，而不是像源码仓库。
2. 用户安装和使用时，面对的是能力与权限，不是环境配置和依赖地狱。
3. 应用可以有自己的界面，也可以有自己的受控逻辑层，而不必退化成纯网页。
4. 宿主能够限制其访问的文件、网络、系统能力和用户数据边界。
5. 应用能跨宿主、跨设备、跨环境保持较高一致性，而不是高度依赖某台机器的本地状态。

## 结论先行

本文档的核心结论是：

**值得尝试定义一种“可移植微应用”模型。**

这类应用的本质不是“安全应用”，而是：

- 一种新的应用包装与运行形态
- 一种比传统全栈项目更轻的分发方式
- 一种默认带有权限边界与宿主契约的逻辑承载方式
- 一种能同时承载 UI 与受控逻辑层的微应用模型

“安全、隔离、默认受限”是它成立的基础条件，但不是它的唯一价值表达。

它真正对用户有吸引力的卖点应该是：

- 更轻
- 更省心
- 更容易安装
- 更容易分发
- 更容易理解自己装了什么
- 更容易在不同环境里稳定运行

## 这种应用不是什么

为了避免后续讨论混乱，需要先写清楚边界。

### 它不是传统 Web 网站

传统网站通常是：

- 前端和后端由站点拥有者托管
- 用户主要通过浏览器访问
- 用户不直接“安装”逻辑层
- 本地权限控制不是主要产品命题

而本文讨论的微应用，是一种用户可以主动安装、运行、授权、卸载的单元。

### 它不是传统本地全栈应用

传统本地全栈应用通常意味着：

- 要么分发一个完整桌面包
- 要么要求用户自己安装 Node / Python / 数据库 / 依赖
- 要么直接在本地跑一个可信度未知的后端

这类路径对开发者灵活，但对普通用户门槛和风险都很高。

### 它不是 Docker 的简化皮肤

Docker 解决的是：

- 环境一致性
- 依赖打包
- 进程隔离

但它的问题是：

- 心智负担重
- 权限模型不够产品化
- 普通用户不把“镜像、容器、端口、挂载卷”理解成安装应用

所以这类微应用不该被定位成“轻量 Docker UI”。

### 它也不是平台专属小程序的简单复刻

小程序的优势是：

- 分发轻
- 宿主统一
- 权限由平台管理

但它的问题是：

- 强依赖单一超级平台
- 平台定义能力边界和商业秩序
- 应用很难独立存在

我们想定义的模型更接近“小程序的宿主化体验”，但不应该被锁死在某个超级平台里。

## 推荐的本质定义

推荐把它定义为：

**可移植微应用 = 一个可安装的应用单元，它包含界面、资源、配置与受控逻辑层，通过统一宿主契约运行，并以显式权限而不是隐式系统访问来获得能力。**

这句话里有五个关键词：

### 1. 可安装

用户面对的应该是一个“应用包”或“应用单元”，而不是一个源码仓库或部署教程。

### 2. 界面

它不能只是一段后台逻辑，至少要支持明确的用户交互入口。

### 3. 受控逻辑层

它可以包含业务逻辑、数据处理、任务执行、联网访问等能力，但这些能力应当经过宿主授权和约束。

### 4. 统一宿主契约

应用不直接假设自己拥有整台机器，而是通过宿主提供的能力接口获取文件、网络、身份、通知、自动化等能力。

### 5. 显式权限

应用能做什么，不靠“默认拿到了什么”，而靠“被授予了什么”。

## 核心产品原则

这类微应用模型建议从第一天就坚持以下原则。

### 1. 应用优先，不是仓库优先

用户安装的是应用，不是项目源码。

### 2. 使用优先，不是部署优先

用户关注的是“它能做什么”和“它会访问什么”，而不是“如何手动跑起来”。

### 3. 权限可读，不是能力隐式

应用申请文件访问、网络访问、联系人访问、自动化执行等能力时，用户应能理解。

### 4. 默认受限，不是默认全能

不应该让应用天然拥有宿主全部能力，再通过文档提醒用户“小心点”。

### 5. 宿主中立，不先绑定某个产品

第一阶段先定义清楚应用模型本身，而不是先把它做成某个现有产品的插件格式。

### 6. 独立成立，后续可嵌入

应用应该能先作为独立运行形态成立；之后再考虑被 `NextClaw`、桌面壳、浏览器壳、移动壳等不同宿主吸纳。

## 推荐的整体模型

推荐采用“三层模型”来理解这种微应用。

### 第一层：应用包

应用包描述“这是什么应用”。

它至少应包含：

- 应用元信息
- 图标与资源
- UI 入口
- 逻辑模块
- 权限声明
- 数据目录声明
- 宿主兼容信息

### 第二层：受控运行时

运行时解决“它怎么被安全、稳定、可移植地执行”。

这一层建议采用：

- UI 继续走标准 Web 技术栈
- 逻辑层优先采用 `Wasm/WASI-first` 运行方式
- 宿主通过 capability-based API 暴露有限能力
- 文件、网络、持久化、任务调度、通知等都经过宿主桥接

需要特别强调的是：

这里推荐 `Wasm/WASI-first`，不是因为用户会在意 Wasm，而是因为它有助于实现可移植、受限执行和宿主能力收口。

### 第三层：宿主

宿主解决“它运行在哪里、被谁管理、如何授权、如何更新、如何被调用”。

宿主可以是：

- 独立桌面壳
- 浏览器壳
- 本地 agent 宿主
- 未来的 `NextClaw` 一类统一入口

第一阶段不要求宿主唯一，但要求宿主契约稳定。

## UI 与逻辑层的关系

这类微应用的关键之一，是它既不能退化成纯前端页面，也不能重新滑回传统本地全栈应用。

推荐的关系是：

- UI 负责交互、展示、状态表达、安装入口与权限可视化。
- 逻辑层负责业务计算、数据处理、任务执行、受控联网、文件处理和有限后台能力。
- UI 不直接拥有宿主系统权限。
- 逻辑层也不直接拥有机器级默认权限。
- 两者都通过宿主提供的统一桥接访问能力。

这样做的好处是：

- UI 仍然保留 Web 的开发效率和跨平台渲染能力。
- 逻辑层获得比纯前端更完整的执行能力。
- 权限边界收敛在宿主层，而不是散落在应用内部。

## 为什么推荐 Wasm/WASI-first，而不是别的

如果只看技术实现，候选路线很多：

- 直接允许 Node 后端
- 直接允许 Python 后端
- 走容器
- 走 Wasm
- 走远端托管逻辑

本文的推荐不是“永远只有 Wasm”，而是：

**第一代参考实现优先采用 Wasm/WASI-first。**

原因主要有四个：

### 1. 可移植性更强

同一逻辑模块更容易在不同宿主、不同系统、不同分发渠道中维持一致行为。

### 2. 权限模型更容易产品化

相比直接给 Node/Python 一个本地进程，Wasm/WASI 更适合作为“默认不具备系统能力，只能通过授权获取能力”的执行环境。

### 3. 宿主边界更清晰

宿主提供什么能力、能力如何命名、哪些资源可以访问，更容易被收口成标准接口。

### 4. 更适合生态分发

如果未来真的要形成应用生态，那么“宿主 contract 稳定 + 逻辑模块可移植”比“语言运行环境任意自带”更容易长期治理。

但也必须明确写出限制：

- 不是所有现有后端都适合直接迁移到 Wasm。
- 重系统调用、强 native 依赖、复杂语言运行时生态的应用，首代迁移成本会高。
- 如果桥接 API 设计太宽松，Wasm 本身也不能自动保证产品级安全。

所以产品层不应对外宣传成“因为用了 Wasm 所以安全”，而应理解为“通过受控运行时与宿主权限体系实现更可信的微应用模型”。

## 权限模型建议

权限模型是这类应用成立的核心，但它不应该喧宾夺主。

推荐把权限设计成“用户能看懂、宿主能执行、开发者能声明”的三方契约。

### 权限应按资源类型表达

建议从第一版就按下面几类表达：

- 文件访问
- 网络访问
- 本地持久化
- 通知
- 剪贴板
- 定时任务
- 身份与账号
- 外部服务连接
- 宿主自动化能力

### 权限应按范围表达，而不是只按开关表达

例如：

- 文件不是“允许文件访问 / 不允许文件访问”
- 而是“允许访问哪些目录，以只读还是读写方式访问”

网络也不是：

- “允许联网 / 不允许联网”

而是：

- “允许访问哪些域名、哪些协议、是否允许外发数据”

### 权限应支持三种时态

- 安装时声明
- 首次使用时请求
- 运行后可回收和重授予

### 权限应有清晰的人话说明

宿主展示给用户的，不该只是技术字段，而应是类似：

- 读取你选择的 `Downloads/Invoices` 目录中的文件
- 连接 `api.notion.com`
- 在每天上午 9 点运行自动整理任务

## 数据与文件系统模型

这类微应用如果要避免重新退化成“随便写本机文件”，就必须从第一版开始定义清楚数据边界。

推荐把数据分成三层：

### 1. 应用私有数据

每个应用默认拥有自己的私有持久化空间，用于缓存、配置、状态和内部数据。

这部分空间：

- 默认只有该应用可见
- 卸载时可清理
- 不应与宿主其它应用默认共享

### 2. 用户授权挂载目录

用户可以显式授权某些目录给应用访问。

例如：

- `~/Documents/Invoices`
- `~/Projects/demo-site/content`

授权时至少应表达：

- 哪个目录
- 只读还是读写
- 授权是单次、会话期还是长期

### 3. 宿主共享资源

未来若存在多应用协同，可以考虑引入宿主级共享资源，但不建议首代就开放过宽。

首代默认策略应偏保守：

- 先有私有空间
- 再有用户显式挂载
- 共享资源后置

## 联网模型建议

联网能力是最容易让“微应用”重新变成“拿到整台机器外连能力”的地方，因此需要尽早收敛。

推荐默认策略为：

- 默认禁网
- 应用通过权限声明申请出站网络访问
- 网络权限按域名或域组授权
- 宿主保留请求可观测性

建议第一阶段只支持：

- HTTPS 出站请求
- 域名白名单
- 基础请求方法

先不要支持过于宽泛的原始 socket 或任意进程代理。

## 应用包建议结构

第一代可以先定义一个足够简单的应用包结构，例如：

```text
my-app/
  app.json
  ui/
  logic/
  assets/
```

其中：

- `app.json` 负责元信息、入口、权限、兼容信息
- `ui/` 负责前端界面资源
- `logic/` 负责受控逻辑模块
- `assets/` 负责图标、静态资源、示例文件等

`app.json` 至少应包含：

- `id`
- `name`
- `version`
- `description`
- `ui.entry`
- `logic.entry`
- `permissions`
- `dataScopes`
- `hostCompatibility`

这样做的意义是先把“应用 contract”立起来，而不是先陷入某门语言或某个 bundler 的细节。

## 宿主职责建议

宿主不是“帮应用跑起来的容器壳”，而是这套模型成立的关键一层。

宿主至少应负责：

- 安装与卸载
- 权限展示与授权
- 应用生命周期管理
- 逻辑运行时启动与销毁
- 应用私有数据管理
- 应用更新
- 应用崩溃与异常隔离
- 用户可理解的日志与诊断视图

如果没有宿主，应用就会退化成“一个被压缩包装过的本地项目”；如果宿主太弱，用户依然无法理解自己安装了什么。

## 用户体验要求

这类微应用成立与否，最终不是只看架构，而是看用户拿到的体验是否真的像“装应用”。

推荐第一代目标体验是：

### 安装前

- 用户看到应用介绍、来源、版本、作者和权限请求
- 用户能明确理解该应用要做什么

### 安装时

- 用户看到一份可理解的权限清单
- 用户可选择是否授权某些目录或网络能力

### 运行时

- 用户可打开应用
- 用户可在设置中复查和回收权限
- 用户能看到应用占用的数据空间和后台任务

### 卸载时

- 用户可选择保留或删除私有数据
- 卸载应尽可能彻底，不残留一堆不可见后台状态

## 与 NextClaw 的关系

本轮不把模型绑定到 `NextClaw`，但仍应提前记录潜在关系，避免后续重复讨论。

长期看，`NextClaw` 可以处于三种可能角色之一：

### 1. 宿主

`NextClaw` 直接成为微应用宿主，负责安装、权限、调用、自动化和统一入口。

### 2. 编排层

微应用先独立成立，`NextClaw` 只负责发现、调用、组合和跨应用工作流编排。

### 3. 生态入口

`NextClaw` 既不一定是唯一宿主，也不一定负责全部运行时，而是提供市场、分发、调度、记忆和用户上下文。

这三种角色都成立，但都应该放在应用模型清楚之后再决定。

## 推荐的第一阶段产品策略

如果我们决定继续推进，这件事不应该一开始就做成“大而全平台”。

推荐的第一阶段目标是：

**先做一个能跑通安装、授权、打开、受控文件访问、受控联网、卸载的最小微应用模型。**

也就是说，第一阶段只需要证明六件事：

1. 一个应用包可以被安装。
2. 它有自己的 UI。
3. 它有自己的受控逻辑模块。
4. 它只能访问被授权的文件和网络资源。
5. 它可以独立打开和运行。
6. 它可以被宿主干净地卸载和回收。

只要这六件事成立，这个模型就已经比“跑一个本地全栈仓库”更接近真正的产品形态了。

## 不建议第一阶段做的事

为了避免失控，第一阶段明确不建议做下面这些事情：

1. 不追求兼容任意现有 Node/Python 项目一键导入。
2. 不追求做完整的容器替代品。
3. 不追求一开始就支持过宽的系统调用桥接。
4. 不追求先做 marketplace、社交分享、商业化体系。
5. 不追求先把它深度绑定进 `NextClaw` 当前架构。

这些都可能有价值，但都不应该抢在模型定义与最小闭环之前。

## 推荐的实现顺序

如果后续要进入实现，推荐顺序如下：

### 阶段 1：定义 contract

明确：

- 应用包格式
- 宿主 contract
- 权限模型
- 数据目录模型
- UI 与逻辑层通信方式

### 阶段 2：做参考宿主

实现一个最小宿主，支持：

- 安装
- 打开
- 授权
- 卸载
- 运行逻辑模块

### 阶段 3：做参考应用

做 2 到 3 个参考微应用，例如：

- 目录整理器
- Markdown 知识卡片应用
- 受限联网的数据采集器

这样可以快速暴露模型是否合理，而不是只在抽象层自我说服。

### 阶段 4：再讨论 NextClaw 集成

等模型独立成立后，再决定：

- `NextClaw` 是否直接成为宿主
- 是否只先做调用与编排接入
- 是否做成未来 marketplace 的能力单元标准

## 开放问题

虽然本文已经给出推荐方向，但仍有一些问题需要在原型前继续确认：

1. 第一代逻辑模块是否只支持 `Wasm/WASI`，还是设计 contract 时就保留多 runtime 余地。
2. UI 与逻辑层的通信协议是否要一开始就做成 component model 风格，还是先做宿主自定义 bridge。
3. 宿主第一代是桌面壳、浏览器壳，还是 CLI + 本地 UI 管理器。
4. 权限界面如何做到“普通用户可理解”，而不是把系统 API 术语直接暴露给用户。
5. 应用更新与签名验证机制第一代做到什么深度。

## 最终建议

我的最终建议是：

**把这件事继续定义为“可移植微应用模型”，而不是“安全应用模型”。**

原因很简单：

- “安全”只是它必须成立的底层条件。
- 真正决定它是否值得做的，是它能否成为 AI 时代一种更自然的软件载体。
- 它需要同时解决分发、安装、权限、逻辑承载、用户理解、宿主收口和未来生态的问题。

如果这个模型成立，它未来既可以独立发展，也可以自然接入 `NextClaw` 这样的个人操作层产品。

而如果一开始就把它定义成“一个更安全的应用方案”，它很容易在叙事上变窄，在产品上变成工具，在生态上失去想象力。
