# 官网场景表达优化方案

## 背景

当前官网已经开始从“功能清单”转向“工作台 + 真实截图 + 具体用例”，但还有两个问题需要系统处理：

1. 用例、能力和截图之间还没有形成稳定分层，部分内容容易再次把“能力入口”写成“用户场景”。
2. 页面已经展示主工作台、Agent、定时任务和技能市场，但还需要让用户更快想到“我能拿它做什么”。

本方案基于以下内部调研：

- [WorkBuddy 场景表达调研](../internal/2026-07-07-workbuddy-scenario-research.md)
- [QoderWork 场景表达调研](../internal/2026-07-07-qoderwork-scenario-research.md)

参考权重：

- WorkBuddy 作为主参考：重点参考它的首页心智、结果承诺、多角色协作和任务交付叙事。
- QoderWork 作为补充参考：重点参考它的任务模板、案例库、材料/指令/结果结构。

本方案是内部执行文档，可以记录判断、取舍和实现顺序；真正落到官网时，必须按 [user-facing-content-boundary](../../.agents/skills/user-facing-content-boundary/SKILL.md) 把内部判断转换成用户结果表达。

## 目标

让官网首页在 10 秒内回答新用户三个问题：

1. NextClaw 能帮我完成哪些具体事情？
2. 它不是只会聊天，具体怎么把任务做下去？
3. 模型、微信/飞书、Agent、技能、定时任务这些能力为什么和我的任务有关？

成功标准：

- 首屏和第二屏能直接看到具体任务，不需要用户先理解“个人操作层”。
- 用例区每一项都是具体任务，标题包含对象、动作或结果。
- 能力区只解释“为什么能做到”，不冒充用例。
- 截图不只是功能页陈列，而是能证明某类任务如何发生。
- 中文文案自然、短句、少抽象名词，不出现内部方法论话术。

## 现状依据

当前首页中文内容已经包含这些模块：

- 首屏：`NextClaw` + “你的 AI 工作台”。
- 展示区：主工作台、Agent 管理、定时任务、技能市场。
- 生态区：模型、聊天工具、技能和自动化。
- 用例区：群聊问题处理、数据抓取与图表、写作、定时简报、小工具、文件处理。
- 功能区：任务开始、资料跟会话走、技能调用、会话连续、本地运行。

这说明当前方向已经接近正确分层，但还缺少一条更强的用户路径：

```text
我有一个任务
  -> NextClaw 能接住
  -> 它会读取资料、调用工具、必要时分工或定时执行
  -> 结果留在工作台里，还能发回渠道或继续修改
```

## 调研结论

### WorkBuddy 可借鉴点

WorkBuddy 应作为本轮官网优化的主参考。它的表达重点不是“有什么配置”，而是“一句话发起任务，多角色协作并交付结果”。它把 Agent、多技能、多模型这类抽象能力包装成“专家团”和岗位分工，让用户能理解复杂任务可以被拆开执行。

对 NextClaw 的启发：

- Agent 管理不要只讲“创建不同 Agent”，要讲“研究、数据、写作、审校可以分工”。
- 微信、飞书、QQ、钉钉不要只做 logo 列表，要放进“任务从哪里来、结果回到哪里去”的故事里。
- 技能、MCP、CLI、定时任务要服务于报告、图表、简报、文件整理、小工具这些产物。
- 首页可以重点借鉴 WorkBuddy 的节奏：强结果承诺 -> 多角色/多能力协作 -> 典型任务 -> 下载安装。
- 可以借它的结构，不逐字借它的品牌话术；对外文案必须换成 NextClaw 自己的产品事实和语气。

### QoderWork 可借鉴点

QoderWork 作为补充参考。它的表达更偏任务样例库：文件整理、照片管理、数据分析、文档创作、研究整合，并给出提示词和执行效果。它的强点是让用户马上能复制一个任务去试。

对 NextClaw 的启发：

- 首页用例区应继续具体化，但还可以补“材料 -> 指令 -> 结果”的结构。
- 可以新增一个独立的“任务模板 / 用例库”页面，不把所有细节挤在首页。
- 截图要和任务绑定：不是“这是技能市场”，而是“这个任务需要浏览器技能，所以在这里安装并继续执行”。

## 核心判断

NextClaw 的官网可以重点参考 WorkBuddy 的叙事骨架，但不能 1:1 复刻它的“专家团”品牌话术。更适合 NextClaw 的表达是：

**从一个用户任务开始，把会话、资料、模型、渠道、Agent、技能、定时任务和本机操作放到同一个可继续推进的工作台里。**

这里的关键取舍：

- 可以讲“不同 Agent 分工协作”，但不说“替代团队”，避免夸张承诺。
- 可以借鉴“专家团”的易懂心智，但不说“全行业专家”，避免和真实能力不匹配。
- 不把“统一入口”当首屏主卖点，首屏先讲用户任务。
- 不把“安装技能”“接入模型”“管理 Agent”写成用例。
- 保留模型、渠道和技能表达，但放在能力支撑区。
- 保留主工作台截图，但截图必须服务任务，不只是界面角落。

## WorkBuddy 主参考转译规则

本轮可以直接参考 WorkBuddy 的页面组织，但要做“产品事实转译”，不能做“句子替换式抄写”。

| WorkBuddy 表达骨架 | NextClaw 转译方向 | 不能照搬的点 |
| --- | --- | --- |
| AI 专家团 / 全场景办公 | 多个 Agent、技能和工具在同一个任务里协作 | 不直接使用“专家团”“全场景办公”作为主 slogan |
| 一句话指令自主规划并交付完整结果 | 说出目标，NextClaw 在工作台里读取资料、调用工具并交付结果 | 不承诺所有任务都能完整自动交付 |
| 多专家并行协作，一个人顶一支团队 | 复杂任务可以交给不同 Agent 分工处理 | 不写“一个人顶一支团队” |
| 从策略到交付一站搞定 | 从任务发起、资料处理到结果沉淀都留在同一个工作台 | 不说“一站搞定”这种过满承诺 |
| 运营、设计、数据、开发等角色覆盖 | 研究、数据、写作、开发、自动化等任务可以分工推进 | 不假装拥有未产品化的垂直专家体系 |
| 外部调研、PPT、业务数据洞察、用户反馈分析 | 调研报告、数据图表、周报文档、客户反馈整理、小工具开发 | 用 NextClaw 真实能展示的任务替换 |

页面节奏建议更贴近 WorkBuddy：

```text
1. 强结果承诺：说出要做什么，任务在工作台里推进到结果。
2. 协作者心智：不同 Agent / 技能 / 工具可以参与同一件事。
3. 典型任务：调研、数据、写作、文件、小工具、定时简报、群聊处理。
4. 真实界面证明：主工作台、Agent 管理、定时任务、技能市场。
5. 接入能力：模型、微信/飞书、MCP、CLI、本地文件、浏览器。
6. 安装入口：桌面版优先，npm / Docker 作为安装方式页承接。
```

## 推荐页面结构

### 1. 首屏：一句话讲任务结果

首屏要回答：用户打开 NextClaw 后，可以把什么事情交给它？

推荐方向：

```text
NextClaw
把要做的事交给 AI，在你的工作台里一路完成。
```

候选说明文案：

```text
从资料整理、数据分析到写作、自动化和本机操作，NextClaw 会把模型、工具、聊天入口和本地文件放在同一个任务里。
```

注意：

- 可以出现“工作台”，但不要把“个人操作层”作为第一解释。
- 不要在首屏解释内部产品方法。
- 不要用截图做杂乱背景；如果首屏放图，必须是一张任务过程清楚的产品截图。

### 2. 场景区：可以直接交给 NextClaw 的事

这一块应放在能力区之前。它回答：我能拿它做什么？

建议首页保留 8 个高感知场景：

| 场景 | 标题方向 | 用户材料 | 交付结果 |
| --- | --- | --- | --- |
| 文件整理 | 整理一堆文件，生成清单 | 本地文件夹、PDF、图片、文档 | 新目录、整理日志、摘要 |
| 数据分析 | 抓取数据，做成图表和结论 | 网页、CSV、Excel、API | HTML 报告、图表、结论 |
| 写作产出 | 把材料写成文章、周报或方案 | 资料、想法、引用、旧文档 | 草稿、改写版、发布稿 |
| 研究整合 | 调研一个领域，输出对比表 | 网页、论文、竞品页面、评论 | 报告、对比表、来源 |
| 小工具开发 | 做一个自己用的小工具 | 重复任务、数据格式、页面需求 | 本地应用、脚本、工作流 |
| 图片生成 | 生成文章配图、产品草稿或视觉灵感图 | 文案、风格要求、参考描述 | 本地图片文件、会话记录 |
| 跨渠道协作 | 群里有人发来任务，AI 先处理 | 微信/飞书/钉钉/QQ 消息 | 回复、报告、后续任务 |
| 周期任务 | 每天定时生成简报或巡检 | 项目状态、日志、链接、数据源 | 定时执行记录、推送内容 |
| 多 Agent 分工 | 让不同 Agent 分头处理复杂任务 | 研究材料、数据、写作要求 | 分工结果、最终汇总 |

每张卡片建议字段：

```text
标题：包含对象、动作或结果。
一句话：输入什么，得到什么。
可选标签：文件 / 数据 / 写作 / 微信 / 飞书 / Agent / 定时。
```

错误示例：

- “统一模型入口”
- “需要新能力时安装技能”
- “给团队一个协作者管理界面”

正确示例：

- “把 20 份客户反馈整理成问题清单”
- “从网页抓数据，生成一份带图表的报告”
- “每天早上把项目状态发到飞书群”

### 3. 过程区：一个任务怎样被做完

这一块回答：它怎么从一句话变成结果？

建议用 4 步，不写复杂架构：

```text
1. 说出目标，选择相关资料或工作目录。
2. NextClaw 在会话里读取资料、调用技能或浏览器。
3. 需要分工、定时或渠道推送时，在同一个任务里继续打开。
4. 结果留在工作台里，下次还能接着改。
```

这块可以配一张主工作台截图。截图内容最好是真实任务，例如：

- 正在分析一份销售数据。
- 正在整理一组文档。
- 正在根据网页资料生成调研报告。

### 4. 截图区：用界面证明关键能力

当前 4 张截图可以保留，但表达要从“功能页名称”改为“任务证明”。

| 当前截图 | 建议表达重点 | 图片内容要求 |
| --- | --- | --- |
| 主工作台 | 一个任务从目标到结果持续推进 | 必须有真实会话、资料、输出结果 |
| Agent 管理 | 复杂任务可以交给不同角色处理 | Agent 名称最好体现研究、数据、写作、审校 |
| 定时任务 | 固定任务可以按时间自动执行 | 展示日报、巡检、提醒、同步等真实任务 |
| 技能市场 | 做任务时需要新工具可以补上 | 展示与任务相关的技能，不只是列表 |
| 图片生成结果 | 文生图任务可以产生本地图片文件 | 直接看到生成图片和输出信息，避免只展示技能入口 |
| 右侧工作区 | 任务结果可以边看边改 | 展示源码、Markdown、HTML 预览、文档浏览器或面板应用 |

如果截图颜色或主题混乱，优先处理截图一致性：

- 使用同一主题和缩放比例。
- 截图完整展示主内容区，不截角落。
- 卡片容器保持克制，截图本身做轻微降饱和即可。
- 不为了“丰富”把多个颜色强烈的截图硬放一起。

### 5. 能力区：解释为什么它能做到

这一块回答：背后接了什么能力？

建议分三组：

1. 模型和运行环境
   - 常见模型提供商。
   - OpenAI 兼容接口。
   - 自定义模型。
   - 本地/自托管运行。

2. 任务入口和沟通渠道
   - 微信、飞书、QQ、钉钉、企业微信。
   - Telegram、Discord、Slack、Email。
   - 重点表达“任务可以从这些地方进入”，不要只是 logo。

3. 工具、技能和自动化
   - Skill / MCP / CLI 工具。
   - 浏览器操作。
   - 本地文件。
   - 定时任务。
   - Agent 分工。

能力区文案必须遵循：

```text
能力名 + 它在任务里负责什么
```

例如：

- “微信和飞书：让任务从常用群聊里进来，也能把结果发回去。”
- “定时任务：让日报、巡检和提醒不用每次手动发起。”
- “技能和 MCP：把浏览器、数据处理、本地工具接进同一条任务。”

### 6. 任务模板页：从首页延伸出去

首页不宜承载过多细节。建议新增一个独立路由或文档页，用于展示“可以直接试的任务”。

页面可以叫：

```text
/use-cases
/examples
/tasks
```

建议内容结构：

| 字段 | 内容 |
| --- | --- |
| 任务名 | 从网页抓数据并生成图表报告 |
| 适合材料 | 网页、CSV、Excel、API |
| 示例指令 | 帮我抓取这个页面里的价格表，整理成 CSV，并做一份趋势图 |
| 预期结果 | CSV、图表、结论、会话记录 |
| 用到能力 | 浏览器、文件、本地脚本、模型 |

首批任务模板建议：

- 整理下载目录并生成文件清单。
- 抓取网页表格，生成 CSV 和图表。
- 把会议纪要转成行动项并发到飞书。
- 汇总客户反馈，按问题类型和优先级分类。
- 调研一个竞品，输出对比表和结论。
- 根据资料写一篇发布说明或周报。
- 生成一个本地 CSV 合并工具。
- 生成一张文章配图或产品草稿图，并保存到本地文件。
- 每天定时检查链接变化并推送提醒。

## 文案规则

### 用户可见内容要写结果，不写过程

对外内容可以写：

```text
从网页抓数据，生成一份带图表的报告。
```

不要写：

```text
我们决定把场景区前置，以避免能力入口冒充用例。
```

### 用例标题必须像真实待办

用例标题必须能接在“我想让 NextClaw...”后面。

可用：

- “整理这个文件夹”
- “生成一份数据分析报告”
- “每天早上发项目简报”
- “把客户反馈整理成优先级”

不可用：

- “统一入口”
- “技能市场”
- “模型接入”
- “Agent 管理”

### 中文文案优先短句

优先写：

```text
把资料放进来，先生成一版能改的草稿。
```

避免写：

```text
通过能力编排实现资料上下文与文档产出流程的持续连接。
```

## 推荐落地顺序

### 第一批：不新增路由，先改首页

1. 参考 WorkBuddy 的强结果承诺重写首屏，降低抽象定位，强调“任务 -> 执行 -> 结果”。
2. 在首屏后补一个“协作者/能力一起参与任务”的过渡区，用 NextClaw 自己的 Agent、技能、工具和渠道事实承接 WorkBuddy 的“专家团”心智。
3. 把用例区前置到截图展示之前或紧跟首屏之后。
4. 把 6 个用例扩成 8 个，并按“任务对象 + 结果”重写。
5. 调整截图卡片标题，让每张图服务一个任务证明。
6. 重写能力区，让模型、渠道、技能都绑定任务作用。
7. 保持安装方式入口，不把安装细节塞回首页。

### 第二批：新增任务模板页

1. 新增 `/use-cases` 或 `/examples` 路由。
2. 首批加入 8 个可复制任务模板。
3. 每个模板写清材料、指令、结果、用到能力。
4. 首页用例卡片链接到对应模板。

### 第三批：准备真实演示素材

1. 准备一组可公开的示例数据和文件。
2. 用真实本地实例跑出截图。
3. 截图统一主题、比例、缩放和信息密度。
4. 将截图生成流程纳入现有 `scripts/docs/refresh-product-screenshots.mjs`。
5. 右侧工作区截图必须覆盖至少一种文件/Markdown/代码/HTML 预览；图片生成能力必须有可见的图片结果，不只展示技能列表。

## Owner 与目录组织

推荐 owner：

- 首页结构与文案：`apps/landing/src/main.ts`
- 首页样式：`apps/landing/src/style.css`
- 静态截图资源：`apps/landing/public/`
- 产品截图生成：`scripts/docs/refresh-product-screenshots.mjs`
- 截图源文件：`images/screenshots/`
- 内部调研：`docs/internal/`
- 执行方案：`docs/plans/`
- 若新增任务模板页：优先继续放在 `apps/landing/src/main.ts` 的现有路由体系内，除非复杂度逼近拆分阈值，再考虑拆模块。

## 非目标

本方案不做这些事：

- 不重塑 NextClaw 品牌定位。
- 不照搬“专家团”命名和视觉主题；可以吸收多角色协作心智。
- 不承诺 QoderWork 那种垂直设计/幻灯片/写作工作台。
- 不写未经验证的效率数字。
- 不把竞品原文大段复制到官网。
- 不在首页塞入完整安装教程。
- 不为了视觉丰富使用抽象 AI 图、机器人图或无关氛围图。

## 验收标准

### 内容验收

- 首页所有用例标题都能回答“我能让它做什么”。
- 用例区不出现“接入模型 / 安装技能 / 统一入口 / 管理 Agent”这类能力入口冒充场景。
- 能力区每组都写清它在任务中的作用。
- 中文文案没有明显翻译腔，读起来像直接写给中文用户。
- 对外页面不出现“我们决定 / 方案 / 取舍 / 原则 / 方法论 / 第一现场”等内部话。

### 视觉验收

- 桌面端和移动端截图完整，不只露出角落。
- 四张截图风格统一，颜色不互相打架。
- 图片生成展示能看到生成结果本身。
- 右侧工作区展示能看到会话旁边的文件、代码、Markdown 或 HTML 预览。
- 移动端无水平溢出。
- 卡片文字不溢出、不遮挡图片。
- 首屏不使用无语义的截图背景。
