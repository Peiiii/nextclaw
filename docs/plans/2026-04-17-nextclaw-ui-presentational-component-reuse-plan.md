# NextClaw UI 展示组件复用治理计划

## 背景

`packages/nextclaw-ui` 已经具备一批共享 UI 原语，例如 `Button`、`Input`、`Card`、`Select`、`Switch`、`Tooltip`，也已经有 design token 体系。但在实际页面实现里，仍然存在不少“绕过共享原语、直接在业务文件里手写按钮/输入框/提示块/信息卡片”的情况，导致同类技术组件的半径、配色、hover、focus、disabled、信息提示样式逐步分叉。

这类问题如果只靠一次性重构，很容易回弹；因此应该先把“优先复用纯展示组件”的思想固化成规则，再按问题密度高、收益大的链路分批整改。

## 目标

1. 让 `packages/nextclaw-ui` 内部形成稳定共识：
   - 纯展示性、技术性 UI，默认先找现有组件。
   - 现有组件不够时，优先补共享组件，而不是继续局部手写。
2. 让技术页面的视觉与交互基线回到同一套共享入口。
3. 在不做大爆炸式重写的前提下，逐步消除高频重复样式块。

## 整改范围

本轮只针对 `packages/nextclaw-ui`，不扩散到其它 app。

重点治理两类问题：

1. 基础原语被绕过：
   - 原生 `<button>` / `<input>` / `<textarea>` / `<select>` 被局部手写样式替代。
2. 纯展示组合块未沉淀：
   - 提示块
   - 信息卡片
   - 设置行
   - 工具栏按钮
   - 列表项外壳
   - 空态 / 状态块

## 推荐整改顺序

### 第一阶段：收口最基础原语

目标：先减少最容易持续扩散的重复样式入口。

优先处理：

1. `marketplace`
2. `marketplace/mcp`
3. `agents`
4. `doc-browser`

动作：

1. 把局部手写的按钮优先收口到 `Button` / `buttonVariants` / `IconButton` 风格入口。
2. 把局部手写的输入框优先收口到 `Input`，必要时补 `Input` 的 `size`、前后缀、搜索态等能力。
3. 对必须保留原生元素的场景，先抽一个共享 wrapper，再由页面复用。

### 第二阶段：补纯展示组合组件

目标：减少“每个页面都重新拼一遍相同提示卡 / 设置行 / 信息块”的情况。

优先补的共享组件建议：

1. `NoticeCard`
   - 承接 info / success / warning / danger / dashed 等常见提示块。
2. `SettingRow`
   - 承接“标题/说明 + 控件”的设置行布局。
3. `SectionHeader`
   - 承接“标题 + 描述 + actions”的技术页面区块头。
4. `ActionChip` 或 `Tag`
   - 承接轻量状态标签、筛选标签、模型标签。

这些组件都应保持“纯展示、无业务逻辑、以 props 驱动”。

### 第三阶段：清理局部硬编码视觉体系

目标：避免同一个 UI 包内部再长出第二套 token 之外的颜色和壳子。

动作：

1. 清理技术页面里无必要的硬编码颜色。
2. 能回到 design token 的，统一回 token。
3. 只在确有产品意图、且不是通用技术组件时保留特殊视觉。

## 执行准则

每次触达 `packages/nextclaw-ui` 中的前端文件时，默认做一次快速判断：

1. 这块 UI 是不是纯展示性、技术性表面？
2. 仓库里是否已有可复用组件？
3. 如果没有，是否已经到了值得抽共享组件的频率？
4. 如果这次不抽，下一步最明确的收口位点在哪里？

默认原则：

1. 直接复用现有组件优先。
2. 抽共享展示组件次之。
3. 继续局部手写必须是最后选项，而且要能说明原因。

## 第一批建议落点

从当前审查结果看，第一批最值得做的不是“全量统一”，而是下面三条线：

1. `components/marketplace`
   - 按钮、卡片动作区、标签、搜索输入的复用化。
2. `components/agents`
   - 对话框提交按钮、提示块、空态壳子的复用化。
3. `components/config` 与 `components/remote`
   - 设置行、状态提示块、信息卡片的复用化。

## 验收标准

满足以下条件即可认为治理开始生效：

1. 新增技术页面默认不再直接写一套新的基础按钮/输入框样式。
2. 同类提示块、设置行、状态块开始共享同一批展示组件。
3. 技术组件的 hover、focus、disabled、圆角、边框、色调不再在不同页面自由漂移。
4. 后续 code review 可以用“是否复用了共享展示组件”作为明确检查项。
