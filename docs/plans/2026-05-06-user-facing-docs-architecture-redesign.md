# NextClaw 用户文档架构重整方案

**Goal:** 为 NextClaw 建立一套面向用户、可扩展、可持续治理的文档信息架构，让用户能以最少学习成本完成安装、接入、使用、常驻、远程访问与排障，同时把 AI 自管理与高阶命令参考从主用户叙事中剥离出去。

**Architecture:** 采用“用户任务流主轴 + 完整命令参考旁路 + AI 自管理独立真相源”的三层结构。公开文档站只承接用户真正需要理解和操作的内容；高频命令只保留最小必要集合；完整命令索引继续存在，但降级为查询型参考；AI/系统自管理命令继续留在包内 `USAGE.md` 体系，而不再主导公开文档站的信息架构。

## 1. 背景与核心判断

当前文档站不是没有内容，而是存在三个结构性问题：

1. 页面存在，但用户发现路径不稳定。
2. 用户文档、运维文档、AI 自管理文档的受众边界不够清晰。
3. 若干页面按“命令表”组织，而不是按“用户要完成什么任务”组织。

这会带来几个直接后果：

- 用户不知道该去哪看“安装后下一步”“后台常驻”“登录自启动”“远程访问”等关键能力。
- 一些实际上主要供 AI 或高级操作者使用的命令，被过早暴露在面向普通用户的主路径里。
- 文档随着能力增长容易继续变成“功能和命令的堆积”，而不是稳定的产品说明系统。

这与 NextClaw 的产品愿景不一致。NextClaw 追求的是统一入口、意图优先、开箱即用和统一体验；文档站也应该服务这个目标，而不是要求用户先理解大量命令表面。

## 2. 设计原则

### 2.1 面向用户优先

公开文档站的第一职责是帮助真实用户成功，而不是完整公开系统内部的所有命令表面。

判断标准不是“这个命令存在吗”，而是“普通用户为了完成目标，是否真的需要主动知道它”。

### 2.2 按任务流组织，不按功能堆叠组织

文档站一级结构应围绕用户生命周期展开：

- 了解产品
- 安装与首次跑通
- 接入能力
- 开始真实使用
- 持续运行与托管
- 排错与恢复

而不是把“命令 / 配置 / Cron / 渠道 / Secrets / Skills”并列成一组平面名词列表，让用户自己推断顺序。

### 2.3 命令最小暴露

在用户主叙事里，只保留最小必要命令集合，例如：

- 安装
- `start`
- `stop`
- `restart`
- `status`
- `doctor`
- `update`

其他命令按语义进入对应场景页，而不是继续放进“用户必读”的主路径。

### 2.4 参考信息旁路化

确实需要被公开保留的高级命令、宿主运行命令、部署命令，不应该消失，但应降级为旁路参考层，而不是主入口层。

它们应被组织成“参考 / 运维 / 宿主运行 / 命令索引”这一类低显著性结构，而不是作为上手主路径核心页面。

### 2.5 AI 自管理独立真相源

AI、自管理、内建助手、系统控制能力所需的完整命令知识，继续由：

- `docs/USAGE.md`
- `packages/nextclaw/resources/USAGE.md`

承接。

这套文档的目标受众不是普通用户，而是 AI、自管理链路和高阶系统操作场景。它不应反向决定公开文档站的主结构。

### 2.6 渐进暴露

文档应遵循“先完成任务，再理解机制”的节奏：

- 先告诉用户该做什么
- 再告诉用户为什么这样做
- 最后才在需要时暴露更底层的命令、平台差异与排障细节

## 3. 文档受众分层

后续所有文档内容都应先判断归属哪一层受众。

### A. 普通用户

关心：

- 如何安装
- 如何第一次跑通
- 如何接入模型/渠道
- 如何开始真实使用
- 如何让它稳定可用
- 出问题怎么办

这是公开文档站的主服务对象。

### B. 高阶用户 / 部署者 / 宿主操作者

关心：

- Docker、systemd、LaunchAgent、Windows Task
- 远程访问
- 常驻、后台运行、自启动
- 运行恢复与平台差异

这类内容可以出现在公开文档站，但应进入“运行与托管”或“高级参考”，而不是进入新手主路径。

### C. AI 自管理 / 系统级控制面

关心：

- 完整 CLI 表面
- Agent/Plugin/Marketplace/Automation 等命令
- 系统自管理合同

这层主要由包内 `USAGE.md` 承接，不应成为公开文档站的主叙事。

### D. 仓库贡献者 / 内部协作者

关心：

- 设计文档
- 迭代记录
- 发布记录
- 架构与治理规则

这部分不属于公开用户文档站主结构。

## 4. 文档面边界重定义

### 4.1 公开文档站 `apps/docs`

定位：用户产品文档站。

职责：

- 解释产品能力
- 帮助用户完成真实任务
- 提供必要但克制的运行与排障指导
- 承担对外稳定入口

不应承担：

- AI 自管理完整合同
- 仓库内部设计与迭代知识沉淀

补充说明：

- 公开文档站可以保留完整命令索引页
- 但该索引页的定位是查询型参考，不是用户入门主流程
- 文档站主导航与用户路径，不应围绕命令穷举展开

### 4.2 `docs/USAGE.md` 与 `packages/nextclaw/resources/USAGE.md`

定位：AI 自管理 / CLI 自说明真相源。

职责：

- 承接完整命令合同
- 支撑 AI 读取、系统自管理和高级操作者排查
- 为运行时内建 guide 提供稳定来源

不应直接映射为公开文档站的信息架构。

### 4.3 `docs/plans` / `docs/designs` / `docs/logs`

定位：设计、方案、治理与变更留痕。

职责：

- 服务团队设计与交付
- 不直接对普通用户暴露为主导航内容

## 5. 目标信息架构

建议把公开文档站重构为以下一级主轴。

### 5.1 认识 NextClaw

目标：帮助用户理解产品是什么、适合谁、核心价值是什么。

建议包含：

- 介绍
- 愿景
- 典型使用方式
- 不是什么

### 5.2 快速开始

目标：帮助用户在最短时间内完成安装与首次跑通。

建议包含：

- 安装
- 快速开始
- 首次启动检查
- 安装后下一步

### 5.3 接入与配置

目标：完成模型、渠道、Secrets、工具等能力接入。

建议包含：

- Provider / 模型接入
- 渠道接入
- Secrets
- 工具
- MCP / Skills 教程入口

### 5.4 使用与自动化

目标：让用户开始进入真实使用，而不是停留在安装完成。

建议包含：

- 对话能力
- 会话管理
- Cron / 自动化
- 常见使用路径

### 5.5 运行与托管

目标：承接“让 NextClaw 持续存在”的全部语义。

这是当前最缺失、也最需要新增的结构层。

建议包含：

- 前台运行 vs 后台运行
- 重启与恢复
- 登录自启动 / 宿主自启动
- Docker 部署
- Linux `systemd`
- macOS LaunchAgent
- Windows Task
- 远程访问

这里应把现在分散在 `commands.md`、教程页和设计理解里的内容收敛为统一产品语义。

### 5.6 排错与恢复

目标：让用户在问题发生时有一条稳定恢复路径。

建议包含：

- 故障排查
- 健康检查
- 常见安装失败
- 常见运行异常
- 反向代理 / 远程访问常见错误

### 5.7 参考

目标：保留必要参考，但明确降级为旁路层。

建议包含：

- 命令索引
- 核心命令速查
- 高级参考
- 宿主运行参考
- 术语说明

其中：

- `命令索引` 负责尽可能完整覆盖所有公开命令，承担查询、检索、跳转、兜底参考职责
- `核心命令速查` 只保留普通用户最常需要的少量命令

注意：完整命令索引应继续存在，但它不应再被当作用户上手主路径；它更接近 CLI reference / command reference，而不是“新用户必须先读完的页面”。

## 6. 命令暴露策略

这是本次重整的关键约束。

### 6.1 用户主路径命令

允许在快速开始、安装后下一步、首页 CTA 等主用户链路中反复出现：

- `npm i -g nextclaw`
- `nextclaw start`
- `nextclaw stop`
- `nextclaw restart`
- `nextclaw status`
- `nextclaw doctor`
- `nextclaw update`

### 6.2 场景命令

允许出现在特定能力页，但不应进入新手首页或命令主入口：

- 渠道登录相关命令
- Secrets 配置命令
- Cron 管理命令
- 远程访问相关命令

这些命令应作为“完成该能力的手段”，附着在对应教程或功能页中。

### 6.3 运维 / 宿主命令

例如：

- `nextclaw service install-systemd --user`
- `sudo nextclaw service install-systemd --system`
- `nextclaw service install-launch-agent`
- `nextclaw service install-task`

这些命令不应出现在面向普通用户的主命令页首屏，而应进入：

- `运行与托管`
- `宿主自启动`
- `部署/常驻`
- `高级参考`

### 6.4 完整命令索引收录范围

以下内容应尽可能完整进入 `命令索引`：

- 核心命令
- 场景命令
- 运维 / 宿主命令
- Agent / Plugin / Marketplace / Automation 等高级命令

也就是说，`命令索引` 的目标不是克制，而是完整；真正需要克制的是“这些命令在用户主路径里出现的频率与显著性”。

### 6.5 AI / 自管理命令

例如：

- Agent CRUD
- Plugin 管理
- Marketplace 管理
- 大量系统自管理命令

这类命令不应再进入公开文档站主导航。它们应回到包内 `USAGE.md` 体系。

## 7. 页面迁移建议

### 7.1 保留但重写定位的页面

- `getting-started.md`
  - 保留
  - 只承接最短跑通路径
  - 不继续堆更多能力说明

- `after-setup.md`
  - 保留
  - 改成“第一次让 NextClaw 真正跑起来”
  - 增加“是否需要后台常驻 / 登录自启动”的明确分流

- `commands.md`
  - 继续保留
  - 明确定位为“完整命令索引 / 命令参考”
  - 尽可能完整覆盖公开可用命令
  - 页面写法应支持查询、跳转、按主题查找，而不是承担用户入门叙事

- `core-commands.md` 或等价页面
  - 建议新增
  - 只保留普通用户高频核心命令
  - 在快速开始、安装后下一步、排障入口中反复引用

- `advanced.md`
  - 保留
  - 收纳高级用户和部署者需要理解的机制，而不是简单继续扩容

### 7.2 建议新增的页面

- `运行与托管` 总览页
- `后台运行与自启动`
- `部署方式选择`
- `宿主运行模型说明`
- `高级参考` 或 `宿主运行参考`

### 7.3 建议从公开主路径移出的内容

以下内容不再建议在公开文档站主路径中突出呈现：

- AI 自管理完整命令面
- 仅供内建助手使用的命令解释
- 与普通用户无关的系统控制细枝末节

## 8. 自启动内容的正确归位

以当前讨论的自启动为例，正确结构不应是：

- 用户先进入 `命令`
- 在大表格里自己发现 `install-systemd` / `install-launch-agent`

而应是：

1. 用户在“安装后下一步”看到分流：
   - 只想先跑起来
   - 想让 NextClaw 持续在后台可用
2. 若选择第二条，进入“运行与托管”
3. 在“后台运行与自启动”页理解：
   - 什么是前台运行
   - 什么是后台常驻
   - 什么是登录自启动
   - 不同平台该怎么做
4. 需要时再看到对应命令

也就是说，命令是落地手段，不是信息架构主轴。

## 9. 一次性重整目标

本次改造不采用“第一批 / 第二批 / 第三批”的产品叙事，而是一次性把结构原则定清、把文档边界拉直、把公开文档站重新收口到统一用户口径。

一次性重整后，应同时满足以下结果：

- 新的一级导航与页面归属稳定下来
- `commands.md` 明确成为完整命令索引，而不是新手主入口
- 用户主路径里只保留最小必要命令
- `运行与托管` 成为后台常驻、自启动、远程访问、宿主运行的统一归属
- `after-setup` 与 `getting-started` 回到面向用户的短路径职责
- `apps/docs`、`docs/USAGE.md`、`packages/nextclaw/resources/USAGE.md` 的边界一次性对齐

同时，至少验证以下问题是否能在 3 步内被用户找到答案：

- 怎么安装
- 怎么启动
- 怎么重启
- 怎么确认服务健康
- 怎么接入模型
- 怎么接入渠道
- 怎么让它后台常驻
- 怎么开启登录自启动
- 出问题先看哪里

## 10. 验收标准

若本轮重整成功，应该满足：

1. 普通用户不需要先读命令大全，就能完成安装到首次成功使用。
2. “自启动 / 常驻 / 宿主运行”有稳定归属，不再散落。
3. `commands.md` 仍然存在，并能作为尽可能完整的命令索引使用。
4. 文档站主路径里的命令数量明显下降。
5. AI 自管理命令与普通用户文档边界清晰。
6. 新增功能后，团队能明确知道应写到哪一层，而不是继续把内容堆进现有热点页面。

## 11. 非目标

本方案当前不直接处理：

- 文档站视觉改版
- 搜索系统改造
- 多语言翻译流程治理
- 所有历史页面的一次性全文重写

本轮优先级是先把结构和边界建立起来，再进入内容迁移。

## 12. 结论

NextClaw 的公开文档站，应该是用户理解和进入这套“个人操作层”的产品入口，而不是完整 CLI 和 AI 自管理合同的投影。

因此，后续文档改造的总原则应明确为：

**主文档站面向用户，按任务流组织，最小暴露命令；完整命令索引继续保留为查询型参考；AI 自管理知识独立承载，不再反向主导公开文档结构。**
