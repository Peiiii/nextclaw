# NextClaw GitHub Organization Setup Plan

日期：2026-04-10

## 1. 背景与核心判断

当前项目仍主要挂在创始人个人 GitHub 账号名下，这在项目早期并不罕见，但它会天然把项目叙事锁定为“个人 side project”。这和 NextClaw 的长期目标并不一致。

结合 [NextClaw 产品愿景](../VISION.md)，我们要做的不是一个短期开源仓库，而是一个有机会成长为长期产品、社区与公司的统一入口型项目。既然目标是成为 AI 时代的个人操作层，GitHub 上的项目承载方式也应该从一开始就朝“独立品牌资产”收敛，而不是继续把品牌、代码、协作关系和发布主体绑在个人身份上。

本方案的核心判断是：

1. 现在就应该建立 `GitHub Organization`。
2. 第一阶段只做“品牌独立 + 资产归拢 + 基础治理”，不做复杂组织设计。
3. 组织首先服务三个目标：`品牌独立`、`协作可扩展`、`未来公司化承接`。
4. 开源和创业并不冲突，但必须从第一天就把“开源资产”和“公司资产”视为同一品牌体系下的不同层次，而不是两套彼此割裂的主体。

## 2. 目标

### 2.1 产品与品牌目标

- 让外部用户先认识 `NextClaw`，而不是先认识创始人个人账号。
- 让项目在 GitHub 上呈现为一个独立品牌，而不是个人附属项目。
- 为未来官网、npm scope、文档站、社区入口、招聘与合作提供统一主体。

### 2.2 组织治理目标

- 让仓库权限、团队角色和维护者协作方式具备长期扩展空间。
- 降低“单个个人账号”成为唯一资产入口与权限中心的风险。
- 为未来引入核心维护者、员工、外部贡献者预留自然路径。

### 2.3 非目标

- 本轮不做复杂多组织矩阵。
- 本轮不做过早的法人、股权或法律结构设计。
- 本轮不把仓库拆成大量子仓库，只为了看起来“更专业”。
- 本轮不引入重流程化开源治理，例如复杂委员会、冗长 RFC 审批链。

## 3. 推荐方案

### 3.1 推荐采用单一品牌组织

建议直接建立一个与产品品牌一致的 GitHub Organization，例如：

- `nextclaw`
- 若主名已被占用，则优先选择清晰、长期可接受的品牌变体，而不是临时凑合名

推荐原则：

- 优先品牌一致性，而不是加入过多技术后缀
- 尽量与官网域名、npm scope、文档站命名一致
- 避免带 `labs`、`demo`、`project` 这类容易弱化主品牌地位的词

### 3.2 组织定位

第一阶段把这个组织定义为：

`NextClaw 的官方产品与开源协作主体`

这句话足够了。它同时覆盖：

- 开源项目承载
- 官方发布主体
- 社区协作入口
- 未来公司化之前的品牌基础设施

不需要在这一阶段刻意区分“公司组织”和“开源组织”。对创业早期来说，主体越统一越好。

## 4. 第一阶段最小结构

### 4.1 组织内只保留三类角色

建议第一阶段只保留：

- `Owners`
  - 创始人本人
  - 仅限极少数最高权限持有者
- `Core`
  - 长期核心建设者
  - 可维护核心仓库，但不默认拥有组织级最高权限
- `Maintainers`
  - 针对具体仓库协作
  - 可按仓库授予维护权限

早期不建议再细分更多 team。角色过多只会制造治理幻觉。

### 4.2 默认仓库策略

建议默认采用：

- 核心主仓库放在组织下
- 未来如确有必要，再增加官网仓库、文档仓库、示例仓库
- 非必要不拆仓

判断标准只有一个：

`拆分是否真的降低理解成本和维护成本`

如果只是为了“像大公司”，那就不拆。

## 5. 组织首页与基础资产

第一阶段建议补齐以下最小资产：

### 5.1 组织 Profile

- 一句话描述 NextClaw 是什么
- 指向官网或主文档入口
- 明确这是官方 GitHub 组织

### 5.2 `.github` 社区元仓库

建议建立 `.github` 仓库，用来统一：

- `profile/README.md`
- issue template
- pull request template
- `CODE_OF_CONDUCT.md`
- `CONTRIBUTING.md`
- `SECURITY.md`

这会让组织级协作体验更统一，也比在每个仓库里重复维护更省事。

### 5.3 包与发布主体

长期建议逐步把这些资产也对齐到组织身份：

- npm scope
- GitHub Packages
- GitHub Actions 发布流程
- 未来的网站部署与官方文档链接

本轮不要求一次性全部迁完，但方向必须统一。

## 6. 仓库迁移原则

### 6.1 核心原则

迁移的目标不是“搬家完成”，而是让资产边界更清楚。

优先迁移：

1. 主产品仓库
2. 官方网站或文档站仓库
3. 未来会作为官方发布或核心依赖的仓库

可以暂缓迁移：

- 纯实验仓库
- 一次性脚本仓库
- 还未准备公开承诺维护的试验性子项目

### 6.2 迁移时要注意

- 提前检查 GitHub Actions secrets、token、package publish 权限
- 提前检查仓库迁移后 webhook、badge、URL 引用是否失效
- 提前检查 npm 发布主体与仓库权限映射
- 提前检查 branch protection 和 CODEOWNERS 是否需要重配

迁移本身通常不难，真正容易漏的是外围配置。

## 7. 开源与创业边界建议

这部分不需要一次定死，但建议从现在就有一个朴素边界：

### 7.1 建议公开承诺开源的部分

- 核心产品代码
- 核心运行时与协议相关能力
- 基础集成能力
- 对开发者和生态有价值的通用能力

### 7.2 未来可保留商业化空间的部分

- 托管服务
- 云端能力
- 团队协作能力
- 商业版控制台或高级管理能力
- 官方托管、稳定性、运维与 SLA

建议心智是：

`开源是产品增长与信任机制，商业化是服务与交付机制`

不要从一开始就把商业化理解成“把核心藏起来”，也不要把开源理解成“未来无法做公司”。

## 8. 不要过度设计的几个边界

这次明确不建议做的事：

- 不要一开始就建很多 team 和权限层级
- 不要一开始就拆成很多仓库
- 不要为了“组织化”引入复杂流程审批
- 不要为了区分身份，再单独建一个“基金会式组织”
- 不要等一切都成熟再迁，早期反而最适合做干净切换

如果只能记住一句话，那就是：

`先把品牌主体立起来，再逐步把资产归过去。`

## 9. 推荐执行顺序

建议按下面的最小闭环推进：

1. 注册并锁定 `NextClaw` 对应 GitHub Organization 名称
2. 完成组织 profile 和基础描述
3. 建立 `.github` 元仓库
4. 把主产品仓库迁入组织
5. 校验 Actions、Secrets、badge、文档链接、npm 发布链路
6. 补齐最小社区治理文件
7. 之后再视需要补官网仓库、文档仓库与 team 细化

这个顺序的好处是：

- 先完成品牌独立
- 再完成资产归拢
- 最后才补协作治理

不会在“治理装修”上花太多时间。

## 10. 当前建议结论

对于 NextClaw 这种目标是成为伟大产品并有机会成长为公司的项目，GitHub 组织不是可有可无的装饰，而是非常早期的一层品牌与治理基础设施。

因此本方案的直接建议是：

- `要建`
- `现在建`
- `先做最小组织化`
- `不做过度设计`

## 11. 长期目标对齐 / 可维护性推进

这份方案与 NextClaw 的长期愿景一致，因为它强化的是“统一入口”的品牌与资产承载方式，而不是继续让项目以个人附属形态存在。对外主体越统一，未来产品、社区、生态和商业化路径越容易收敛到同一叙事里。

从可维护性角度，这次选择的是最小治理模型，而不是复杂组织设计。重点是减少未来再迁移、再改名、再重做权限模型的返工成本，同时避免现在就堆出一套并不需要的组织结构。这里优先遵循的是“删减优先、简单优先、边界清晰优先”。

## 12. 下一步建议

如果要继续推进，下一份文档建议只做一个更具体的执行清单，范围控制在：

- 组织命名确认
- 迁移前检查项
- 迁移当天操作顺序
- 迁移后验证清单

不要直接进入更大的公司治理设计。
