# Platform Admin Console Layout Design

## 背景

`platform-admin` 目前已经承接了平台治理能力，但信息架构仍停留在“一个 dashboard 把所有卡片平铺到底”的阶段。随着 Marketplace 审核、用户额度、充值审核、上游供应商管理等模块逐步增加，这种结构会很快出现三个问题：

1. 新管理员第一次进入后台时，不知道不同治理动作分别在哪里完成。
2. 页面一旦继续增长，筛选、审核、额度调整、充值确认会在同一纵向滚动流里互相干扰。
3. 后台虽然已经是独立站点，但产品感知仍像“临时运维页”，不够像一个稳定的统一控制面。

这和 NextClaw 的长期方向不一致。既然 NextClaw 想成为 AI 时代的个人操作层，那么它自己的治理面也需要朝“统一入口、统一体验、统一控制面”收敛，而不是继续把控制能力堆成一个长页面。

## 长期目标对齐

这次改造直接服务三条主线：

- 统一入口：管理员通过一个清晰的控制台完成审核、额度、充值等治理动作，而不是在长页面里找区块。
- 自感知与自治：系统管理能力要能被理解、被定位、被操作，不能继续依赖“只有熟悉内部结构的人才知道在哪”。
- 统一体验：平台前端要呈现出稳定的后台信息架构，为后续继续接入 `Remote 管控`、`平台设置`、`审计` 等模块留出自然扩展位。

## 方案选择

### 方案 A：继续单页平铺，只做视觉分组

优点：

- 改动最小
- 不需要重新组织页面壳层

缺点：

- 只能缓解，不解决信息架构问题
- 模块一多还是回到“长页面找卡片”
- 不利于深链路、导航和后续扩展

### 方案 B：轻量经典管控台壳层

即本次推荐方案：

- 左侧一级导航
- 顶部全局栏
- 中间内容区按模块切页
- 先只拆最常用的 4 个页面
- 尽量复用现有 section 和 API，不重写业务逻辑

优点：

- 成本可控
- 一次性把后台从“平铺页”升级为“正式控制台”
- 后续继续扩展新模块时不需要再推翻结构

缺点：

- 需要做一次页面拆分和导航壳层改造

### 方案 C：直接做完整多级后台体系

优点：

- 结构最完整

缺点：

- 明显过重
- 会把本次需求从“壳层升级”扩大成“后台体系重做”
- 不符合当前“先轻量收敛”的节奏

## 推荐方案

采用方案 B。

核心判断：

- 这次要解决的是“后台已经有能力，但没有像后台”的问题。
- 所以优先改信息架构和进入路径，而不是继续加更多功能。
- 壳层一旦稳定，后续无论是 Marketplace 治理、Remote 管控还是审计页，都能自然接进去。

## 信息架构

### 一级导航

第一批只放 4 个页面：

- `总览`
- `Marketplace 审核`
- `用户与额度`
- `充值审核`

暂不新开 `Remote 管控` 和 `平台设置` 页面，避免这次做重；现有“营收 / 上游供应商 / 模型目录”先归入 `总览`，后续再独立拆页。

### 页面边界

#### 总览

承接平台级总览信息与当前还没单独拆页的治理区块：

- 平台总览
- Remote 额度总览
- 利润总览
- 上游供应商账号
- 模型目录

#### Marketplace 审核

独立承接现有 `AdminMarketplaceReviewSection`，保持双栏审核体验不变，但从长页面中抽离成一等页面。

#### 用户与额度

独立承接：

- 全局免费池上限设置
- 用户额度管理

把“谁能用多少”“平台免费池给多少”放在同一个治理语义下。

#### 充值审核

独立承接充值审核列表与审核动作，避免和额度编辑、Marketplace 审核混在一起。

## 路由与壳层设计

为了保持轻量，本次不引入新路由库，直接使用 URL hash 做页面切换：

- `#/overview`
- `#/marketplace`
- `#/users`
- `#/recharge`

这样可以满足：

- 页面可深链
- 前后退可用
- 部署仍保持静态前端简单模型

壳层结构：

1. 左侧 `sidebar`
   - 产品标识
   - 一级导航
   - 当前管理员信息
2. 顶部 `topbar`
   - 当前页面标题
   - 当前页面说明
   - 退出按钮
3. 内容区 `content`
   - 渲染当前模块页面

移动端不做复杂抽屉，先采用“侧栏上置 + 内容下置”的响应式折叠方式，保证先可用。

## 交互设计

### 侧边导航

- 当前页高亮
- 每个页面显示简短说明
- 保持可点击面积足够大，便于长时间治理操作

### 顶部栏

- 展示当前页面标题和一句话描述
- 右侧保留管理员邮箱和退出入口
- 顶部栏不承载复杂筛选，避免再次变成拥挤工具栏

### 总览页

- 上半区先展示关键总览卡片
- 下半区继续承接营收 / 上游 / 模型管理
- 用分组标题控制节奏，避免再次回到“毫无层级的平铺”

## 技术实现

本次实现尽量复用现有代码：

- 新增一个控制台壳层页面，负责导航与内容区渲染
- 把当前 `AdminDashboardPage` 里的模块按页面边界拆成独立 page component
- 复用现有：
  - `AdminMarketplaceReviewSection`
  - `GatewayBusinessLoopSection`
  - 现有 API client
  - 现有 UI 组件

同时顺手做两点轻量收敛：

1. 把登录态页面的“当前用户”读取改成直接依赖 `fetchMe` 结果，减少 query -> store 的镜像同步。
2. 把后台首页入口从“长 dashboard”改成“console shell + page registry”，后续加新页只需要补导航定义和页面组件。

## 验证与发布

本次需要完成：

1. 本地验证
   - `pnpm -C apps/platform-admin tsc`
   - `pnpm -C apps/platform-admin lint`
   - `pnpm -C apps/platform-admin build`
2. 治理守卫
   - `pnpm lint:maintainability:guard`
3. 前端冒烟
   - 新增或补一条后台 UI 冒烟，至少确认：
     - 登录页正常
     - 进入后台后存在侧栏导航
     - 四个页面均可切换
     - Marketplace 审核页仍能展示核心文案
4. 发布
   - `pnpm deploy:platform:admin`
5. 线上探测
   - 访问 `https://platform-admin.nextclaw.io`
   - 确认登录页可打开
   - 如能拿到有效管理员 token，再做一次真实登录后的页面结构冒烟

## 非目标

本次明确不做：

- 重写后台 API
- 新增复杂权限体系
- 设计系统重做
- 新增二级导航、面包屑、快捷命令面板
- 拆出完整 `Remote 管控` / `平台设置` / `审计日志` 独立页

这些都应该建立在本次壳层收敛完成之后，再按优先级继续推进。
