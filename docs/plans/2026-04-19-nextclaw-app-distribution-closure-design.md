## 当前已经具备的能力

基于当前仓库与已发布结果，下面这些能力已经成立，不再属于本轮的核心缺口：

- 已有独立 runtime 包：`@nextclaw/app-runtime`
- 已有独立 CLI：`napp`
- 已有开发入口：`napp create / inspect / run / dev`
- 已有应用目录 contract：`manifest.json + main/ + ui/ + assets/`
- 已有最小权限模型：`documentAccess / allowedDomains / storage / capabilities`
- 已有本地宿主与桥接：`/__napp/*`
- 已有示例应用：`apps/examples/hello-notes`
- 已完成 npm 对外发布
- 已完成官方 marketplace skill 发布
- 已形成最小文档、测试和 smoke 闭环

一句话说：**runtime 本身已经有了，当前不缺“怎么跑”，当前缺“怎么分发、怎么安装、怎么作为产品被别人使用”。**

## 当前真正缺的东西

如果只看缺口，当前核心只剩下面五类：

### 1. 缺正式的应用分发物

现在应用的真实形态还是一个目录。

这意味着：

- 开发者可以做出 app
- 但分享时仍然是在传目录
- 用户收到后仍然要自己处理文件组织

这不是真正的产品化分发形态。

### 2. 缺安装体系

当前用户可以运行一个 app 目录，但还不能把 app 当成“已安装应用”来管理。

典型缺失包括：

- `napp install`
- `napp uninstall`
- `napp list`
- `napp info`

### 3. 缺本地应用注册表

当前 runtime 只会临时跑一个目录，还没有一个统一的本地 app registry 来记录：

- 已安装了哪些 app
- 每个 app 的版本是多少
- 安装目录在哪
- 当前启用的是哪个版本
- 元信息和权限摘要是什么

### 4. 缺 app marketplace 闭环

当前已经有的是 runtime 的 marketplace skill，不是 app marketplace。

也就是说：

- 我们已经能发布“如何使用 runtime”的 skill
- 但还不能发布“某个具体微应用”
- 也不能让另一个用户一条命令完成 app 的拉取与安装

### 5. 缺最小可用的信任与校验边界

我们不把产品定义成“安全产品”，但分发闭环里仍然需要最小边界：

- 安装前展示 manifest 与权限摘要
- 校验包结构是否有效
- 能识别作者/来源
- 后续版本升级时，能看见版本变化和权限变化

## 已冻结结论 1：应用的三种形态必须分开

这是当前最重要的收敛结论。

一个 NextClaw 微应用应当有三种不同形态：

1. **源形态**：目录
2. **分发形态**：归档包
3. **运行形态**：安装后解包的目录

具体解释：

- 开发者开发时，面对的是真实目录
- 分享给别人时，传输的是一个 bundle 包
- runtime 运行时，不直接运行包本体，而是运行安装后落盘的目录

这三者必须故意分开，不能混成一个概念。

## 已冻结结论 2：推荐方案是“目录为真，包为运输，安装后解包运行”

这是当前推荐的主方案，也是目前最平衡的方案。

### 推荐主路线

- `napp pack <app-dir>`：把 app 目录打成 bundle
- `napp install <bundle-or-source>`：校验后安装到本地统一目录
- `napp run <installed-app>` 或 `napp open <installed-app>`：运行安装后的 app

### 为什么推荐这条路线

因为它同时满足三个目标：

- 最大化复用现有 runtime 能力
- 最小化新复杂度
- 最容易形成“别人可以分享、别人可以安装”的产品体验

它不会把现有的 `create / inspect / run / dev` 推倒重来，而是在现有 runtime 外面补一层分发与安装。

## 已冻结结论 3：不推荐直接运行包本体

理论上可以设计“直接从包里运行”，但当前不推荐。

原因很直接：

- 静态资源服务会更复杂
- Wasm 加载会更复杂
- 调试与诊断会更复杂
- 权限映射与升级管理也会更复杂

当前阶段更合理的路线是：

- 包只负责运输和交付
- 安装时解包
- 运行时继续复用现在已经可用的目录式宿主

## 已冻结结论 4：不需要发明新的底层封装格式

我们需要一个 app bundle 概念，但不需要发明新的二进制容器格式。

当前最佳建议是：

- 产品概念上，把它叫做 `NextClaw App Bundle`
- 文件扩展名可以叫 `.napp`
- 底层物理格式直接复用 `zip`

这样做的好处：

- 用户侧有明确的“app 包”概念
- 技术侧不用自造轮子
- 打包、校验、解包、跨平台支持都会简单很多

## 已冻结结论 5：`.napp` 的职责是分发，不是执行

`.napp` 的职责应当是：

- 可分享的交付物
- 可安装的 bundle
- 可被校验和展示摘要的分发单元

它不应当是：

- 直接执行形态
- 新的运行时 contract
- 另一套不同于目录的 app 结构

一句话说：**`.napp` 只是 app 目录的运输载体。**

## 已冻结结论 6：第一版 bundle 内容要非常克制

第一版 `.napp` 建议只包含下面几类内容：

- `manifest.json`
- `main/`
- `ui/`
- `assets/`
- `bundle.json` 或等价元数据文件
- `checksums.json` 或等价摘要文件
- 可选的 `signature.json`

第一版不建议引入：

- 复杂嵌套 bundle
- 多运行时二进制矩阵
- 安装脚本
- 任意 post-install hook
- 自定义执行器

## 已冻结结论 7：运行时仍然应该以目录为核心 contract

这条结论非常关键，因为它直接影响后面所有命令设计。

当前已有 runtime contract 是目录式的，这本身是对的，不应该因为要做分发闭环而推翻。

也就是说，后续应该是：

- `inspect` 仍然面向目录
- `dev` 仍然面向目录
- `run` 可以继续优先面向目录，也可以后续扩展到安装态 app
- `install` 负责把 bundle 变成目录

这个收敛非常重要，因为它让开发工作流和使用工作流保持清晰分层，而不是混成一坨。

## 方案对比结论

### 方案 A：只传目录，不做 bundle

优点：

- 最简单

缺点：

- 不像真正产品
- 分享体验差
- 安装与版本管理困难

结论：不推荐作为正式闭环。

### 方案 B：目录为源，`.napp(zip)` 为分发，安装后解包运行

优点：

- 和现有 runtime 最兼容
- 工程复杂度最低
- 最适合作为第一版产品闭环

缺点：

- 需要补安装层和 registry

结论：**当前最佳推荐方案。**

### 方案 C：直接从 `.napp` 包里运行

优点：

- 概念上更“完整”

缺点：

- 明显更复杂
- 会推高宿主、调试和升级复杂度

结论：不推荐作为第一版主路线。

### 方案 D：直接用 npm 包作为 app 分发格式

优点：

- 对开发者熟悉

缺点：

- 对普通用户不友好
- 会把 app 分发过度绑定到 npm / Node 生态

结论：不建议作为主分发模型。

### 方案 E：直接上 OCI / 容器包

优点：

- 标准化成熟

缺点：

- 太重
- 明显偏离当前目标

结论：不推荐。

## 当前最合理的总判断

如果只看当前缺口与现有资产，最合理的下一阶段主张是：

**不是重做 runtime，而是在现有 runtime 外面补齐一层“分发、安装、注册、拉取”的产品闭环。**

换句话说，当前最应该新增的，不是另一套执行体系，而是：

- app bundle
- install / uninstall / list / info
- 本地 registry
- app marketplace 最小闭环

## 下一轮继续讨论的未决问题

虽然上面的方向已经基本收敛，但还有几件事需要继续讨论并冻结：

### 1. `.napp` 内部目录与元数据到底怎么定

例如：

- 要不要单独 `bundle.json`
- 摘要文件叫什么
- 签名文件第一版要不要上

### 2. 本地安装目录怎么设计

例如：

- `~/.nextclaw/apps/<app-id>/<version>/`
- 是否需要 `current` 软链或等价指针
- 是否要区分用户数据目录与 app 安装目录

### 3. 本地 registry 怎么设计

例如：

- registry 放 JSON 还是别的结构
- 记录哪些字段
- 如何表达已安装版本与当前启用版本

### 4. CLI 命令面怎么冻结

至少需要继续讨论：

- `napp pack`
- `napp install`
- `napp uninstall`
- `napp list`
- `napp info`
- 是否需要 `napp open`

### 5. app marketplace 的最小形态怎么定义

例如：

- 是不是先只支持元数据 + bundle URL
- 安装时是下载 bundle 再走本地 install
- 是否先不做复杂审核与推荐流

### 6. 最小信任模型怎么做

例如：

- 第一版只做结构校验和摘要校验，还是就上签名
- 安装前要展示哪些信息
- 升级时如何提示权限变化

## 当前推荐的下一步讨论顺序

为了避免讨论再次发散，下一轮最推荐按下面顺序继续：

1. `.napp` 包内容与结构
2. 本地安装目录
3. 本地 registry
4. `pack / install / list / uninstall / info` 命令面
5. app marketplace 最小闭环

这样讨论会最顺，因为前一项基本都是后一项的输入。
