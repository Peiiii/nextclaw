# Shape Workflow

Use this workflow before writing code for a new feature or major redesign.

The goal is not to produce UI code. The goal is to produce a design brief strong enough that later implementation choices stop feeling generic or accidental.

## Discovery Rules

- Ask only the questions needed to remove ambiguity.
- Prefer one or two targeted questions at a time instead of a giant questionnaire.
- Do not jump to layout ideas before the product goal is clear.
- Do not infer the audience or tone from code alone.

## Discovery Areas

Cover these areas before writing the brief:

1. Purpose and context
   - what the feature is for,
   - who uses it,
   - what success looks like,
   - what emotional state the user is in.
2. Content and data
   - what the interface shows or collects,
   - realistic minimum, typical, and extreme data sizes,
   - empty, error, loading, and first-time states.
3. Design goals
   - the primary user action,
   - the desired tone,
   - what must feel obvious or memorable.
4. Constraints
   - framework or implementation limits,
   - accessibility requirements,
   - responsive requirements,
   - existing design-system constraints.
5. Anti-goals
   - what would be the wrong direction,
   - what risks matter most if the design misses.

## Deliverable

Produce a compact design brief with these sections:

- feature summary,
- primary user action,
- design direction,
- layout strategy,
- key states,
- interaction model,
- content requirements,
- recommended reference files,
- open questions.

## Quality Bar

A good brief should make these things obvious:

- what the user is trying to accomplish,
- how the interface should feel,
- what deserves emphasis,
- what states must exist,
- and which Impeccable references are worth loading next.

If those are still fuzzy, keep shaping before building.
