# Proactive Work Continuation

## 目标

把“阶段性进展”与“任务完成”分开。除非用户明确要求暂停、只讨论、只汇报状态或等待确认，否则在目标未完成时必须继续推进下一步。

## 触发场景

- 用户说“主动干活”“继续”“别停”“你怎么停了”“还没做完”。
- 用户纠偏、补充事实、指出方案方向错了，但没有要求暂停。
- 当前任务包含代码、设计、排查、验证、规则沉淀、发布、提交等多个环节，容易在中途误收尾。
- 已经完成一个局部动作，但仍有未验证、未落地、未记录或未回答的新问题。

## 判断是否能停

只有满足以下任一条件，才可以停下并给最终回复：

- 原始目标和用户最新追加目标都已经完成，并且有可观察验证或明确的不适用说明。
- 遇到真实阻塞：缺少不可推断的信息、外部权限、不可逆高风险操作需要确认，或继续会破坏用户改动。
- 用户明确要求暂停、先讨论、先别改、等他确认、只要状态或只回答问题。
- 当前 turn 的任务本身就是一个纯解释问题，且用户没有要求继续落地。

下面这些都不能当作停止理由：

- 写完一个方案文档。
- 补了一条规则或 skill。
- 跑过一个局部测试。
- 发现一个可能原因但还没验证链路。
- 用户纠正了方向，但没有要求暂停。
- 代码已经能编译，但用户可见问题还没做同链路验收。

## 工作协议

1. 先用一句话重述当前未完成目标和最新用户约束。
2. 如果刚被纠偏，先承认失效点，再把它转成下一步动作，不展开空泛道歉。
3. 继续执行最高优先级的下一步；不要把“我会做”当成结果。
4. 每完成一个局部动作，检查是否仍有未完成项；有就继续。
5. 如需暂时停下，必须说明属于哪一种真实阻塞，以及解除阻塞后下一步是什么。

## 与其它 Skill 的关系

- 源码、脚本、测试或运行链路任务仍由 `nextclaw-delivery-workflow` 作为总流程 owner。
- 复杂 debug 仍使用 `long-chain-debugging`；本 skill 只负责“不在证据不足时误收尾”。
- 用户指出同类停止问题反复发生时，同时使用 `learning-from-failures`，并判断是否需要更新 `AGENTS.md` 或其它 skill。
- 修改 `AGENTS.md`、skill 或命令索引时，同时使用 `nextclaw-agent-instructions-governance`。

## 输出要求

使用本 skill 后，最终回复至少说明：

- 已完成哪些具体动作。
- 是否还有未完成项；如果有，为什么本轮可以或不可以继续。
- 若停止，停止原因必须是用户明确要求或真实阻塞，而不是阶段性产物。
