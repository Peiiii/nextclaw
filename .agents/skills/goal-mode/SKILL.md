# Goal Mode

## 概述

目标模式是一种执行情景模式，用来把长任务从“阶段性响应”切换成“目标完成闭环”。

目标模式支持任意尺度的目标。用户只需要定义最终目标、边界和验收口径；AI 必须自行拆分内部阶段、维护锚点、逐步验证并持续推进。

内部阶段只用于降低风险、定位回归和持续验证，不能把阶段拆分责任反推给用户，也不能用“第一阶段完成”冒充总目标完成。

它只做四件事：

- 明确目标
- 建立锚点
- 周期自检
- 完成前不停止

## 触发与退出

用户明确说下面任一意思时，进入目标模式：

- 进入目标模式
- 开启目标模式
- 目标模式：...
- 持续推进直到目标完成
- 用锚点追踪目标

用户明确说下面任一意思时，退出目标模式：

- 退出目标模式
- 关闭目标模式
- 暂停目标模式

## 回复前缀

目标模式开启后，每次对用户回复都必须在常驻前缀后追加：

```text
[目标模式][锚点 n/20]
```

如果同时处于深思模式，前缀组合为：

```text
[我严格遵守规则][深思模式][目标模式][锚点 n/20]
```

`n` 是有效工作回复计数。每次完成一次实质推进后 `+1`。

## 锚点文件

优先复用 `goal-progress-anchor` 的落点规则。

默认写入当前活跃迭代目录：

```text
docs/logs/<iteration>/work/goal-progress.md
```

如果当前不应创建迭代目录，不要为了目标模式单独新建 `docs/logs`。此时在当前任务可用的计划文档、工作笔记或回复中维护短锚点，并在真正需要落盘时再创建文件。

锚点必须短小，默认不超过 `50` 行，最多不超过 `100` 行。

推荐结构：

```text
# Goal Progress

## 当前目标

## 明确非目标

## 冻结边界 / 不变量

## 已完成进展

## 当前下一步

## 锚点计数器
```

## 进入目标模式时必须做

1. 用一句话重述用户目标。
2. 明确验收条件。
3. 明确非目标和禁止偏离边界。
4. 决定锚点落点。
5. 初始化锚点计数器为 `0/20`。
6. 把 `0/20` 当作一个周期起点：先回看一次目标、边界、验收条件与下一步，判断是否需要补充或调整计划。
7. 给出最小可执行下一步，然后开始执行。

如果目标本身不清楚，先问一个必要澄清问题；不要靠猜测启动长链路执行。

## 执行循环

每轮都按这个顺序推进：

1. 对照锚点判断当前目标是否仍然正确。
2. 自行选择并执行当前最小下一步；大目标必须内部拆成可验证批次推进。
3. 验证这一步是否推进目标。
4. 更新锚点中的已完成进展和当前下一步。
5. 回复前缀中的锚点计数器 `+1`。
6. 如果目标未完成，继续推进，不因阶段性产物、阶段计划或局部验证通过而停止。

## 强制回看锚点

出现任一情况时，必须先回看锚点并极短重述目标、边界、下一步：

- 计数器处于 `0/20`
- 计数器达到 `20/20`
- 用户指出偏航、目标不对、别造轮子、不要过度抽象
- 开始讨论新抽象、新核心概念或新架构层
- 连续多轮讨论方案但没有推进原目标
- 上下文压缩、恢复或明显存在记忆断层

其中：

- `0/20` 表示一个新周期的起点，必须先 review 计划，再开始这一轮周期执行。
- `20/20` 表示一个周期的结尾，必须先 review 计划，再决定是否补充、收缩或调整，再把计数器重置为 `0/20`。

## 停止条件

目标模式下，只有满足任一条件才停止：

1. 验收条件全部完成，并明确说明完成证据。
2. 遇到真实阻塞，并说明阻塞原因、已尝试路径、剩余最小下一步。
3. 用户明确暂停、转向或退出目标模式。

不能因为只完成了方案、只完成了部分实现、只跑了一个检查就停止。

如果目标很大，阶段完成只能作为锚点进展记录；除非总体验收条件已经完成，否则必须继续推进或说明真实阻塞。

## 与其他 skill 的关系

- 目标漂移风险、锚点文件和计数器细节优先复用 `goal-progress-anchor`。
- 复杂跨轮执行可同时使用 `iteration-work-notes`。
- 编码任务仍必须使用 `nextclaw-clean-implementation` 和验证相关 skill。
- 修改规则、命令或 skill 分层时仍必须使用 `nextclaw-agent-instructions-governance`。

目标模式不是替代其他 skill，而是把它们纳入同一个目标完成闭环。
