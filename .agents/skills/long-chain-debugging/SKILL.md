# Long Chain Debugging

## 概述

把“全链路盲跑”改成“证据驱动的链路切片”。

目标不是一次看懂全部系统，而是尽快回答两个问题：

- 第一个出错的 hop 在哪里
- 最小长期修复 owner 是谁

## 何时使用

当问题具备以下任一特征时使用这个 skill：

- 经过多层边界，例如 `UI -> API -> runtime -> provider -> state -> UI`
- 经过多进程、多事件源、流式传输或异步回写
- 复现成本高，整条链路跑一次很慢
- 日志很多但信息密度低，越看越乱
- 团队容易各猜一段，没有统一证据面
- 用户给的是“现象”，而不是已经指向单个函数的直接根因

以下情况通常不需要这个 skill：

- 纯函数逻辑错误，单测或局部断言已经能直接命中
- 单一模块内的简单空值、类型或语法问题

## 固定输出骨架

使用这个 skill 时，先把排查工作压成下面 7 个模块再动手：

1. `症状定义`
2. `黄金复现`
3. `链路地图`
4. `观察点计划`
5. `当前假设与缩圈实验`
6. `根因与修复位点`
7. `同链路验收`

如果这 7 个模块里前 5 个都还写不清，默认说明还不该进入改代码阶段。

## 工作流

### 1. 先冻结症状

用一句话写清：

- 用户实际看到什么
- 正确行为本应是什么
- 哪个具体信号能证明“现在真的坏了”

不要写“好像不对”“似乎慢了”“感觉没生效”这类无法验证的表述。

### 2. 建立黄金复现

优先追求“最小但真实”的失败样本，而不是“最完整”的全流程。

黄金复现至少要冻结这些变量：

- 入口形态
- 关键配置
- 会话 / 项目 / 运行时上下文
- provider / model / transport
- 是否是隔离环境

如果仓库里已有相关 smoke、诊断或回放脚本，优先复用它们做黄金复现，而不是重新手工点击。

### 3. 画链路地图

把问题链路写成 hop 序列，每个 hop 只回答一件事：

- 它收到什么
- 它承诺输出什么

示例格式：

`入口 -> 参数归一化 -> 编排层 -> 传输层 -> 上游系统 -> 响应解析 -> 状态写回 -> 用户表面`

目标不是把实现细节全写出来，而是找到“合同边界”。

### 4. 先切链，再观测

不要一上来在整条链路上铺日志。

先找一个最能把搜索空间减半的边界，问：

- 这个边界之前是对的还是错的
- 如果这里已经错了，后面全都不用看
- 如果这里还对，前面全都可以先排除

默认优先使用二分法，而不是从头到尾线性猜。

### 5. 只在边界打结构化观察点

观察点应落在 hop 边界，而不是到处 `console.log`。

每个观察点尽量只采这几类证据：

- `correlation id`
- `hop 名称`
- `归一化后的输入摘要`
- `归一化后的输出摘要`
- `状态迁移 / 事件发射`
- `耗时 / 重试 / 走到的分支`

优先结构化、可对比、可删掉的临时探针；避免大段叙事型日志污染链路。

### 6. 一次实验只验证一个假设

每轮实验都写成下面这个句式：

`如果 X 是根因，那么在 Y 边界应该观察到 Z。`

实验结束后只回答：

- 这个假设被支持了，还是被排除了
- 下一刀要切哪里

不要一轮实验同时试图证明三件事。

### 7. 找到第一个错误 hop

真正需要继续下钻的点，不是“最后哪里看起来坏了”，而是：

- 第一个违反合同的边界

谁产出了第一份错误输出，谁就是下一步的 owner。

### 8. 在正确 owner 上修，而不是在症状表面补

默认优先修：

- 合同错误
- 边界映射错误
- 状态所有权错误
- 事件顺序错误
- 真实 source of truth 错误

默认不要修：

- 渲染层掩盖
- 下游兜底
- 事故特判
- 解释型补丁

### 9. 用同链路验收收尾

修完后，验证优先级固定为：

`真实链路复现 > 贴近真实的定向 smoke > 最小可证明替代验证`

最终必须明确写出：

- 验证场景
- 观察指标
- 结果
- 剩余未验证缺口

## 快速缩圈手法

长链路问题默认优先考虑以下策略：

- `链路二分`：优先找到中点边界，而不是从入口一路猜到出口
- `成功 / 失败对比`：抓一份成功样本和一份失败样本，先看第一处分叉
- `上游回放`：把失败输入或关键事件固定下来，避免每次都重新走全链路
- `旁路验证`：跳过下游表面，用更早的真相源确认问题是否已存在
- `阴影断言`：在不改变主逻辑的前提下加临时 invariant 检查，尽快暴露违约边界

## NextClaw 场景提示

在本仓库里做长链路排查时，优先注意这些冻结项：

- 隔离 `NEXTCLAW_HOME=/tmp/...`
- 固定 `session id`
- 固定 `project root`
- 固定 `provider / model / runtime`
- 固定所选 skill 集合

常见高杠杆观察点：

- 流式问题：先看第一个错误事件类型、事件顺序和中途丢失的字段
- UI 状态问题：对比 `server truth / persisted truth / rendered truth`
- 配置问题：对比加载时快照与保存后快照
- prompt / skill 可见性问题：对比可用 skills 快照与实际请求里带上的 skill 选择

## 反模式

- 没有稳定复现就直接改代码
- 整条链路到处铺日志，最后还是不知道第一处错误在哪里
- 一次实验同时改探针、改逻辑、改配置，导致结论失真
- 明明是边界合同错误，却在渲染层或最末端补一个掩盖分支
- 修完只跑单测，不做同链路复验

## 完成标准

只有同时满足以下条件，才算真正完成：

1. 症状被压成了可验证的失败定义
2. 已找到第一个错误 hop，而不是最后一个报错点
3. 修复位点属于正确 owner，而不是下游表面
4. 已用同链路完成验收，或明确写出剩余验证缺口
