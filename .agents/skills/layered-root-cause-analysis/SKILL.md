# 分层根因分析

## 目标

把“找到一个直接原因”升级为“识别多层原因结构”。

本 skill 不替代 `long-chain-debugging`。  
`long-chain-debugging` 负责跨 hop 排查、观察点设计和第一个错误边界定位；本 skill 负责在已有现象或证据基础上，把根因拆成不同层级，避免停在最近的技术触发点。

## 触发场景

使用本 skill 时，通常用户真正要的不是一句“哪里坏了”，而是：

- 为什么这个问题会发生；
- 为什么直接修复不够；
- 哪一层才是长期 owner；
- 哪些设计、合同、韧性、治理或产品机制让问题能扩散；
- 这次教训应该沉淀到代码、测试、skill、规则还是产品机制。

## 分层模型

默认至少检查 5 层，不要求每次都全部展开，但不能只停在第一层：

1. **现象层**
   用户看见了什么，哪个可观察信号证明问题存在。

2. **直接触发层**
   哪个具体输入、状态、数据、请求、文件、事件或分支触发了当前失败。

3. **生成路径层**
   这个坏状态是怎样被写入、生成、传播或允许进入系统的。

4. **防线缺口层**
   哪个合同、校验、原子性、隔离、恢复、错误表达或测试缺失，让局部坏状态拖垮了更大表面。

5. **系统/产品层**
   这暴露了什么更长期的架构、数据资产、用户体验、自感知、自恢复或治理机制缺口。

## 分析纪律

- 直接原因不是完整根因，只是根因树的叶子或中间节点。
- 每一层都要说清“它解释了什么”和“它还没解释什么”。
- 不把“修坏数据”“加 try/catch”“提示用户重试”当成完整解决方案。
- 如果问题涉及用户长期资产、自动化任务、会话历史、配置或发布物，必须额外检查读取端韧性和恢复路径。
- 如果结论暴露可复用教训，联动 `learning-from-failures`，把它落到会自动触发的位置。
- 如果下一步需要跨进程、跨 runtime、跨事件源继续缩圈，再联动 `long-chain-debugging`，不要把长链路观察点塞进本 skill。

## 输出骨架

复杂问题默认按这个顺序输出：

1. `现象`
2. `直接触发`
3. `生成路径`
4. `防线缺口`
5. `系统性原因`
6. `最小修复`
7. `长期修复`
8. `需要沉淀的机制`

如果证据不足，要明确标注“已证实 / 高概率 / 待验证”，不要把推断写成事实。

## 完成标准

一次根因分析只有同时满足以下条件，才算完成：

- 直接触发点明确；
- 至少追问到生成路径或防线缺口其中之一；
- 区分了短期恢复和长期修复；
- 指出了真正 owner；
- 判断了是否需要机制沉淀。
