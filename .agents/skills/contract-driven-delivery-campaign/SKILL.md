# 契约驱动交付战役

## 概述

当任务核心不是“修一个点”或“补一个小改动”，而是“把一个复杂编码任务真正从目标对齐推进到最终交付”时，使用这个 skill。

这个 skill 面向 Nextbot / NextClaw 仓库中的长期编码交付任务。它的重点不是单纯提高执行速度，而是降低四类长期任务漂移：

- 目标漂移：最初想要的结果在执行中被忘掉或替换
- 范围漂移：该做的没做完，不该做的做多了
- 完成漂移：做到一半就误判为“已经差不多完成”
- 验收漂移：最终没有按最初约定的标准判断是否完成

它的核心原则是：

1. 先锁定任务契约
2. 再按阶段推进
3. 每一轮都检查是否偏航
4. 只有在契约中的完成条件全部满足时，才允许宣布完成

不要把聊天上下文当成长期任务的唯一真相源。

## 何时使用

以下场景适合使用：

- 用户希望在仓库内推进一个复杂编码任务，且任务会跨多轮
- 用户强调“最后结果必须真的符合最初要求”
- 任务除了编码实现，还可能包含验证、提交、发布等收尾环节
- 当前风险不在于“不会写”，而在于“容易做偏、做漏、做一半就停”
- 用户希望 agent 主动把任务闭环，而不是停在某个中间阶段

以下场景通常不适合使用：

- 一次性的小修小补
- 局部作用域很明确的简单 bugfix
- 纯文档、纯措辞或纯元信息调整
- 明确只做“可维护性治理战役”而不是一般编码交付

若任务已经明确收窄成“长期可维护性治理”，优先使用：

- `.agents/skills/autonomous-maintainability-campaign/SKILL.md`

## 必需输入

开始第一轮前，至少先对齐以下真相源：

- 用户原始任务要求
- 当前对任务的关键澄清
- `docs/VISION.md`
- `AGENTS.md` 的 Rulebook / Project Rulebook 与 `docs/logs` 机制
- `commands/commands.md` 中的 `/validate` 与 `/maintainability-review`
- `.agents/skills/iteration-work-notes/SKILL.md`

如果当前任务属于非功能改动，或在交付中会触发可维护性要求，还应补读：

- `.agents/skills/post-edit-maintainability-guard/SKILL.md`
- `.agents/skills/post-edit-maintainability-review/SKILL.md`

把仓库治理规则和任务契约都视为硬约束。

## 最小运行时状态

在当前活跃迭代目录的 `work/` 下，默认维护三个文件：

- `work/task-contract.md`
- `work/working-notes.md`
- `work/state.json`

默认职责分工：

- `task-contract.md`：定义用户真正想要什么，什么才算完成
- `working-notes.md`：沉淀当前事实、进展、阻塞与下一步
- `state.json`：提供机器可读的续跑状态

若当前迭代目录已存在而这些文件还不存在，使用本 skill 的模板生成。

除非确实失控，否则不要继续新增其它状态文件。

真实任务状态只允许落在对应迭代目录的 `work/` 下，禁止把运行时实例文件写回 skill 目录本身。

## 当前活跃迭代目录判定

进入长期任务模式前，必须先明确当前活跃迭代目录，不能模糊处理。

默认按以下顺序判定：

1. 先扫描 `docs/logs` 下所有合法的 `v<semver>-<slug>` 迭代目录
2. 若最近一次相关迭代仍服务于同一交付目标 / 同一问题域 / 同一收尾窗口，则直接复用该迭代目录
3. 若不存在相关迭代目录，但用户已经明确要求“现在开始持续推进 / 整夜跑 / 提前留痕 / 先建起来再跑”，则按 `AGENTS.md` 的版本规则立即新建一个合法迭代目录
4. 若不存在相关迭代目录，且用户也没有明确授权提前留痕，则先不要在 skill 目录或其它临时位置创建运行时状态；必须先按仓库规则确定合法迭代目录，再进入长期任务模式

判定完成后：

- `task-contract.md`
- `working-notes.md`
- `state.json`

都必须归属到该迭代目录的 `work/` 下。

不要在多个迭代目录里同时维护同一个长期任务的活跃状态。

## 最小维护原则

这个 skill 的目标是降低任务漂移，不是制造新的流程负担。

因此默认遵循以下最小维护原则：

- `task-contract.md` 只在目标、非目标、交付物、完成条件、验收清单发生变化时更新
- `working-notes.md` 只记录当前真相、关键决策、剩余缺口与恢复入口，不写流水账
- `state.json` 只保存机器续跑真正需要的最小状态，不承载长篇解释

不要为了“看起来更严谨”而把状态文件越写越重。

## 任务契约

`task-contract.md` 是第一等公民。没有任务契约，就不要进入长期战役模式。

一个合格的任务契约至少要写清楚：

- 用户真正想要的结果
- 明确非目标
- 最终交付物清单
- 硬约束与禁止项
- 完成条件
- 验收清单
- 提交 / 发布是否属于完成条件

若当前任务没有明确这些内容，先补齐契约，再继续执行。

## 启动前快速核对

在真正开始第一轮改动前，至少先回答：

1. 这次是不是长期编码交付，而不是小范围局部修改？
2. 用户真正要的最终结果，是否已经被写进 `task-contract.md`？
3. 当前任务的非目标，是否已经写清楚？
4. 当前活跃迭代目录是否已确定，`docs/logs` 留痕路径是否明确？
5. 当前是否已经知道最终交付物有哪些？
6. 当前是否已经知道完成条件和验收方式？
7. 当前是否包含提交、发布、冒烟或上线闭环要求？

若第 2、3、5、6 任一答不清，就先完善任务契约，不要急着写代码。

## 同步检查点

为了降低上下文压缩后的失忆风险，默认只在以下检查点强制写回状态：

1. 锁定任务契约后
2. 完成一个阶段后
3. 做完一次关键验证后
4. 准备长时间等待、切换任务或结束当前轮次前

不要每做一个微小动作就重写全部状态文件；那样会增加负担并稀释真正重要的信息。

只要用户中途更新了目标、边界、优先级或取舍，必须把这些变化视为新的外部事实，并在继续执行前同步更新 `task-contract.md` 或 `working-notes.md`。

## 阶段模型

默认把长期编码交付分为以下阶段：

1. 契约锁定
2. 方案收敛
3. 编码实现
4. 验证修正
5. 交付收尾
6. 提交 / 发布（若契约要求）

每次推进前，必须先知道当前处于哪个阶段。

不要在“编码实现”阶段误把任务宣布完成，也不要在“验证修正”还没结束时提前跳到“收尾总结”。

## 执行循环

每一轮长期任务都必须按以下顺序执行：

1. 重新读取 `task-contract.md`
2. 重新读取 `working-notes.md` 与 `state.json`
3. 确认当前所处阶段
4. 选择一个仍然服务于任务契约的最小推进批次
5. 执行该批次
6. 运行该批次需要的最小充分验证
7. 判断当前结果是否偏离契约
8. 把结果写回三个状态文件
9. 判断是继续下一批、等待唤醒，还是停止

不要让“附近顺手能做的事”盖过“契约里真正要求完成的事”。

## 偏航检查

每一轮结束前，至少检查以下问题：

1. 这一步是否仍然直接服务于用户最初要的结果？
2. 这一步是否只是局部最优，而没有推动整体闭环？
3. 有没有把某个中间成果误当成最终交付？
4. 当前是否出现“先做了很多，但契约里的关键验收项还没动”的情况？
5. 如果现在停下，离真正完成还差哪些明确项？

只要第 2、3、4 任一答案为“是”，就不要宣布完成。

## 压缩恢复协议

当对话被压缩、长时间中断或通过 heartbeat 恢复时，默认先执行恢复协议，而不是直接继续改代码。

恢复协议固定为：

1. 先读 `task-contract.md`
2. 再读 `working-notes.md`
3. 再读 `state.json`
4. 核对最近的用户消息是否已经改变目标、边界、优先级或取舍
5. 核对当前仓库状态是否已经让现有状态失效，例如新增改动、工作树变化、验证结果变化或其它外部修改
6. 先写一个简短恢复摘要，再决定是否继续

恢复摘要至少包含：

- 当前任务是什么
- 当前阶段是什么
- 已完成什么
- 剩余缺口是什么
- 下一步为什么是它

如果以下任一情况成立，就先修正状态，不要直接执行下一步：

- 恢复摘要无法和任务契约对齐
- 最新用户消息已经推翻或收窄了旧契约
- 当前 repo 状态已经让原来的阶段判断、下一步或验证结论失效

恢复后的下一步，必须同时满足三方一致：

- `task-contract.md`
- 最新用户要求
- 当前 repo 状态

## 批次选择规则

一个批次只有同时满足以下条件，才适合继续推进：

- 它服务于任务契约中的某个明确交付项
- 它不会把任务带离当前阶段
- 它有对应的验证方式
- 它完成后能缩短“离最终完成还差什么”的列表

默认优先顺序：

1. 直接推进最终交付项的批次
2. 解锁后续工作的阻塞批次
3. 清理当前阶段中妨碍闭环的局部问题

不要优先做“看起来爽”的局部优化，而忽略真正阻塞完成的关键项。

## 验证与完成规则

这个 skill 不替代仓库验证规则，它负责编排“什么时候必须验证、验证什么、验证通过后能不能算完成”。

默认要求：

- 代码改动后必须按触达范围做最小充分验证
- 若契约要求用户可见行为正确，必须有贴近真实场景的验证
- 若契约要求提交或发布，则仅代码完成还不算完成
- 若仍有未满足的验收项，就不能宣布完成

若当前任务包含非功能改动、重构、结构整理或命名治理，还应接入：

- `post-edit-maintainability-guard`
- `post-edit-maintainability-review`

## 完成判定

只有在以下条件同时成立时，任务才算完成：

- `task-contract.md` 中的交付物清单已完成
- `task-contract.md` 中的验收清单已满足
- 当前阶段已经推进到真正的交付收尾，而不是仍停留在实现或验证中间态
- 所有明确未完成项都已关闭，或被显式记录为未完成并获用户接受
- 若契约要求提交 / 发布，这些动作已完成或已明确为不适用

不要使用“差不多”“基本完成”“主体完成”来替代完成判定。

## 停机条件

当出现以下任一情况时，停止并写入阻塞：

- 任务契约仍不完整，无法继续安全推进
- 当前阶段所需的关键决策缺失
- 验证结果含混不清，无法判断是否满足契约
- 任务已经从编码交付漂移成开放式产品探索
- 剩余工作需要用户重新定义目标、边界或取舍

停止时必须写清楚：

- 卡在哪里
- 为什么卡
- 剩余缺口是什么
- 下次最合理的恢复入口是什么

如果停机的根因是“契约不完整”或“恢复后无法确认当前阶段”，必须优先补状态，而不是盲目继续推进。

## 与现有治理机制的配合

本 skill 默认直接接入以下现有机制：

- 用 `iteration-work-notes` 维护 `working-notes.md`
- 用 `commands/commands.md` 中的 `/validate` 作为验证执行协议
- 用 `commands/commands.md` 中的 `/maintainability-review` 作为主观二审入口
- 必要时接入 `post-edit-maintainability-guard`
- 必要时接入 `post-edit-maintainability-review`

这个 skill 负责的是“任务闭环、防漂移与完成判定”，不是替代已有验证与治理工具。

## 整夜续跑 / Heartbeat 模式

当用户要求长时间无人值守推进时，优先使用 thread heartbeat automation。

每次唤醒都按新的一轮处理：

1. 读任务契约
2. 读当前笔记与状态
3. 检查当前阶段与剩余缺口
4. 只推进一个仍然服务于契约的批次
5. 若不能安全继续，就干净停下

automation prompt 应该描述长期 standing task，而不是某个瞬时子步骤。

## 非目标

这个 skill 不是用来：

- 替代需求澄清本身
- 以“长期任务”为名无限扩 scope
- 把没有完成的工作包装成“已经基本完成”
- 把局部实现进展误写成最终交付
- 牺牲用户最初要求去换局部执行便利

## 完成收尾

任务完成时：

1. 确保 `task-contract.md` 反映最终结果
2. 确保 `working-notes.md` 反映最终状态
3. 确保 `state.json` 标记为 `已完成` 或 `已停止`
4. 若仓库规则要求留痕，更新对应迭代 `README.md`
5. 最终总结必须明确：
   - 完成了哪些交付物
   - 哪些验收项已满足
   - 是否存在未完成项
   - 若有提交 / 发布要求，是否已真正闭环
