# 代码评审

## 概述

使用这个 skill 时，默认采用“找问题优先”的评审视角。

优先关注正确性、回归风险、契约清晰度和可维护性压力，而不是停留在样式层面的评论。

## 工作流

1. 明确范围。
   识别 diff、触达文件、相邻契约以及受影响测试。
2. 重建行为变化。
   判断代码改动前做什么、改动后做什么，以及真实调用方或用户现在会观察到什么。
3. 先看高风险表面。
   优先检查正确性、边界条件、状态迁移、异步行为、数据流、API/UI 契约变化以及运行失败模式。
4. 用外部契约视角检查测试。
   只有在真实回归路径或用户可见契约缺少保护时，才把“缺测试”作为问题提出。
5. 先量化非测试代码增减。
   如果这次评审的是纯 bugfix、纯重构或其它不新增用户能力的改动，先判断排除测试后的净增是否大于 `0`；只要为正数，默认就是硬 finding，而不是“可以解释一下”的轻提醒。
6. 检查是否应改为配置驱动与组合式实现。
   如果改动里出现了多份相似组件、重复 JSX、分支堆叠、平台/入口差异散落等情况，先判断是否应把稳定骨架保留在共享组件中，再用类型化配置和薄壳层组合表达变化。
7. 做一轮反思优化。
   在准备给出结论前，再反问一次：这份实现是否还能通过删除、简化、职责收敛、配置驱动或更薄的组合层继续变好；如果答案是“能”，默认应继续报 finding，而不是直接放过。
8. 输出 findings-first 结果。
   按严重级别列出问题，并附上具体文件引用、风险说明和最小可信修复方向。
9. 以不确定性收尾，而不是堆废话。
   在 findings 之后补充开放问题或前提假设；只有确实有价值时才给简短总结。

## 评审问题

- 对真实用户、调用方或运维方来说，什么会坏？
- 哪个状态迁移、分支或输入现在违反了预期契约？
- 这次改动是否制造了重复逻辑或多份真相来源？
- 这块实现是否更适合“共享骨架 + 配置表达变化 + 薄壳层组合”，而不是继续堆分支或复制组件？
- 它是否加入了掩盖更深层缺陷的隐藏兜底行为？
- 测试是在保护稳定外部行为，还是只是在钉死实现细节？
- 如果这不是新增用户能力，这个 diff 的非测试代码净增是否已经大于 `0`？
- 相比再加一个分支、开关或适配层，直接删除或收敛路径是否更安全？
- 现在这个结论是否已经经过一轮反思优化，确认没有明显更简单、更清晰、更可组合的替代方案？

## 评审规则

- findings 优先，总结靠后。
- 明确区分已确认问题、假设判断和后续待确认问题。
- 正确性、回归风险、数据丢失、安全/隐私暴露和运行失败，优先级高于样式细节。
- 只要可能，每条问题都给出具体文件引用。
- 优先基于外部契约推理，而不是基于个人实现偏好下判断。
- 只有当外部契约或高风险回归缺少保护时，才把“缺测试”列为问题。
- `delete-simplify-before-add`：评审改动或拟议修复时，先判断能否通过删除代码、删除分支或移除兼容路径来解决；如果不能，再判断能否把多条路径简化为一条显式主路径；只有前两步都不足以解决问题时，才建议新增逻辑。把“再加一个开关/分支/兜底”视为最后手段。
- `non-feature-non-test-growth-fails-review`：如果评审对象是纯 bugfix、纯重构、结构整理、命名调整或其它不新增用户能力的改动，只要排除测试后的非测试代码净增大于 `0`，默认就必须报 finding，并明确指出这次改动尚未通过 review；不能用“这次写得更优雅”“解释得通”代替通过。
- `config-driven-composition-preferred`：如果同一结构存在多组仅在顺序、文案、图标、显隐、跳转或局部布局上变化的实现，默认要先判断能否改成“共享稳定骨架 + 类型化配置 + 薄壳层组合”；若可以却没有这样做，应报 finding，指出当前写法会降低可读性、可扩展性或可维护性。
- `reflective-optimization-required`：准备给出结论前，必须再做一轮“反思优化”检查，确认当前实现不是只做到“能工作”，而是真的已经逼近当前上下文下更简单、更清晰的方案；若还能明显删减、简化、配置化或收敛职责，默认应继续报 finding。
- `review-is-a-gate`：review 不是摘要，不是庆功会，而是结束闸门。只要仍存在未解决 findings，结论就必须是“未通过 review / 需继续修改”；只有 findings 清零后，才允许给出 `no findings` 或等价的通过结论。
- 如果没有发现问题，必须明确写出 `no findings`，同时注明剩余风险或验证缺口。

## 输出要求

使用这个 skill 时，回复顺序固定为：

1. 按严重级别排序的问题列表，每条都包含文件引用、风险和理由。
2. 开放问题或前提假设。
3. 简短变更总结，或者明确写出 `no findings` 并说明剩余风险。

## 通过标准

- 只要存在一个未解决 finding，本次 review 就不通过，任务不能按“已完成”收尾。
- 只有在 findings 清零后，才允许输出 `no findings` 或等价通过结论。
- 若受外部阻塞无法清零 findings，必须明确写出阻塞项、风险和下一步，结论仍应保持未通过，而不是模糊收尾。
